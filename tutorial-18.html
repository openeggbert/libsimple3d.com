What you will learn

  • How to create fill-based progress bars with CreateProgressBar() and drive them with SetValue(0–1)
  • How to color-code a progress bar at runtime using SetColor() — for example switching green to red as health drops
  • How to display vertical progress bars using SetVertical(true) — useful for XP columns or audio level meters
  • How to create interactive value sliders using CreateSlider(min, max) with a custom numeric range
  • How to read the current slider value with GetValue() and write it programmatically with SetValue()
  • How to react to user interaction in real time by supplying a lambda to SetOnChange()
  • How to group UI widgets into panels using panel->AddChild(element) so that positions become panel-relative and visibility is inherited
  • How to control rendering order when UI layers overlap using SetZOrder() — higher values always render on top
  • How to center a panel on screen precisely with SetAnchor(UI::Anchor::Center)
  • How to enable gamepad D-pad navigation between buttons using SetFocusGroup() and EnableFocusNavigation()
  • How to handle the gamepad cancel (B) button with SetOnFocusCancel()

Overview

Part 17 introduced the basic UI widgets: panels, buttons, labels, and images. This part builds on that foundation and covers the two remaining interactive widget types — progress bars and sliders — together with three techniques for structuring more complex UIs: parent-child hierarchies, Z-order layering, and gamepad focus navigation.

The example program builds a working in-game settings menu. A 3D box spins in the background. A permanent HUD layer shows a health bar (green, draining slowly) and an XP bar (vertical, blue). When the player presses Escape, a centered settings panel slides into view over the top of the HUD. The panel contains a volume slider, a brightness slider, a Resume button, and a Quit button. D-pad navigation lets gamepad users move between the two buttons without a mouse.

Complete source

Copy this into main.cpp, build, and run. The step-by-step explanation that follows breaks every new API call down in detail. 

// main.cpp — Part 18: UI Advanced
#include <Simple3D/Simple3D.h>
#include <string>

using namespace Simple3D;

class UIAdvancedDemo : public Game {
public:

    // ── Start() ──────────────────────────────────────────────────────────────
    void Start() override {
        SetWindowTitle("UI Advanced — simple-3d");
        SetClearColor(Color(0.05f, 0.07f, 0.12f, 1.0f));

        // ── Background scene ──────────────────────────────────────────────────
        // A simple spinning box gives us something to see behind the UI.
        Entity* sun = CreateEntity("Sun");
        sun->AddDirectionalLight(Color(1.0f, 0.95f, 0.85f), 1.0f);
        sun->SetRotation(Quaternion(45.0f, 30.0f, 0.0f));

        box_ = CreateEntity("Box");
        box_->AddModel("Models/Box.mdl");
        box_->SetPosition(0.0f, 0.0f, 0.0f);

        cam_ = CreateCamera();
        cam_->SetPosition(0.0f, 2.0f, -6.0f);
        cam_->LookAt(Vector3::ZERO);

        // ── HUD layer (ZOrder 1) ─────────────────────────────────────────────
        // These widgets live directly on the screen root (no parent panel).
        // ZOrder 1 keeps them below the settings menu (ZOrder 10).

        // Health progress bar — horizontal, color-coded
        hpBar_ = CreateProgressBar();
        hpBar_->SetSize(200, 18);
        hpBar_->SetPosition(12, 12);
        hpBar_->SetColor(Color(0.2f, 0.85f, 0.25f, 1.0f));      // green fill
        hpBar_->SetBackgroundColor(Color(0.1f, 0.1f, 0.1f, 0.85f));  // dark trough
        hpBar_->SetValue(0.78f);   // 78% health to start
        hpBar_->SetZOrder(1);

        Label* hpLabel = CreateLabel("HP");
        hpLabel->SetPosition(12, 34);
        hpLabel->SetFontSize(12);
        hpLabel->SetColor(Color(0.6f, 0.9f, 0.6f, 1.0f));

        // XP progress bar — vertical orientation
        xpBar_ = CreateProgressBar();
        xpBar_->SetSize(12, 80);
        xpBar_->SetPosition(8, 55);
        xpBar_->SetVertical(true);       // fill grows upward
        xpBar_->SetColor(Color(0.2f, 0.5f, 1.0f, 1.0f));        // blue fill
        xpBar_->SetBackgroundColor(Color(0.06f, 0.06f, 0.2f, 0.9f));
        xpBar_->SetValue(0.45f);   // 45% XP
        xpBar_->SetZOrder(1);

        Label* xpLabel = CreateLabel("XP");
        xpLabel->SetPosition(5, 138);
        xpLabel->SetFontSize(12);
        xpLabel->SetColor(Color(0.5f, 0.7f, 1.0f, 1.0f));

        Label* hint = CreateLabel("Press Esc to open Settings");
        hint->SetPosition(12, 162);
        hint->SetFontSize(13);
        hint->SetColor(Color(0.55f, 0.55f, 0.65f, 1.0f));

        // ── Settings menu panel (ZOrder 10) ──────────────────────────────────
        // Anchor::Center places the panel's own centre at the screen centre.
        // SetPosition(0, 0) means no additional offset from that centre point.
        // ZOrder 10 ensures it renders on top of ZOrder-1 HUD elements.
        menuPanel_ = CreatePanel();
        menuPanel_->SetSize(370, 310);
        menuPanel_->SetAnchor(UI::Anchor::Center);
        menuPanel_->SetPosition(0, 0);
        menuPanel_->SetColor(Color(0.04f, 0.06f, 0.14f, 0.94f));
        menuPanel_->SetZOrder(10);
        menuPanel_->SetVisible(false);   // hidden until Escape is pressed

        // Title — position is relative to panel top-left after AddChild
        Label* titleLabel = CreateLabel("SETTINGS");
        titleLabel->SetPosition(133, 20);
        titleLabel->SetFontSize(20);
        titleLabel->SetColor(Color(0.85f, 0.85f, 1.0f, 1.0f));
        menuPanel_->AddChild(titleLabel);   // now owned by the panel

        // ── Volume slider ────────────────────────────────────────────────────
        Label* volLabel = CreateLabel("Volume");
        volLabel->SetPosition(24, 78);
        volLabel->SetFontSize(14);
        volLabel->SetColor(Color(0.75f, 0.75f, 0.88f, 1.0f));
        menuPanel_->AddChild(volLabel);

        volSlider_ = CreateSlider(0.0f, 1.0f);
        volSlider_->SetSize(220, 20);
        volSlider_->SetPosition(110, 75);
        volSlider_->SetValue(0.8f);    // default: 80%
        volSlider_->SetColor(Color(0.3f, 0.55f, 1.0f, 1.0f));
        volSlider_->SetOnChange([this](float v) {
            SetMasterVolume(v);
            volValueLabel_->SetText(std::to_string(static_cast<int>(v * 100)) + "%");
        });
        menuPanel_->AddChild(volSlider_);

        volValueLabel_ = CreateLabel("80%");
        volValueLabel_->SetPosition(336, 78);
        volValueLabel_->SetFontSize(13);
        volValueLabel_->SetColor(Color(0.55f, 0.78f, 1.0f, 1.0f));
        menuPanel_->AddChild(volValueLabel_);

        // ── Brightness slider (range 0–2 so 1.0 = neutral) ───────────────────
        Label* briLabel = CreateLabel("Brightness");
        briLabel->SetPosition(24, 120);
        briLabel->SetFontSize(14);
        briLabel->SetColor(Color(0.75f, 0.75f, 0.88f, 1.0f));
        menuPanel_->AddChild(briLabel);

        briSlider_ = CreateSlider(0.0f, 2.0f);   // 0 = black, 1 = normal, 2 = max
        briSlider_->SetSize(220, 20);
        briSlider_->SetPosition(110, 117);
        briSlider_->SetValue(1.0f);    // default: 100%
        briSlider_->SetColor(Color(1.0f, 0.85f, 0.25f, 1.0f));   // warm yellow
        briSlider_->SetOnChange([this](float v) {
            // Express as percentage of neutral (1.0 = 100%)
            briValueLabel_->SetText(std::to_string(static_cast<int>(v * 100)) + "%");
        });
        menuPanel_->AddChild(briSlider_);

        briValueLabel_ = CreateLabel("100%");
        briValueLabel_->SetPosition(336, 120);
        briValueLabel_->SetFontSize(13);
        briValueLabel_->SetColor(Color(1.0f, 0.9f, 0.5f, 1.0f));
        menuPanel_->AddChild(briValueLabel_);

        // ── Divider label ────────────────────────────────────────────────────
        Label* divLabel = CreateLabel("──────────────────────────────────────────");
        divLabel->SetPosition(14, 158);
        divLabel->SetFontSize(10);
        divLabel->SetColor(Color(0.25f, 0.28f, 0.4f, 1.0f));
        menuPanel_->AddChild(divLabel);

        // ── Action buttons ────────────────────────────────────────────────────
        resumeBtn_ = CreateButton("Resume");
        resumeBtn_->SetSize(148, 42);
        resumeBtn_->SetPosition(24, 218);
        resumeBtn_->SetColor(Color(0.08f, 0.42f, 0.12f, 0.92f));
        resumeBtn_->SetOnClick([this] { CloseMenu(); });
        menuPanel_->AddChild(resumeBtn_);

        quitBtn_ = CreateButton("Quit");
        quitBtn_->SetSize(148, 42);
        quitBtn_->SetPosition(198, 218);
        quitBtn_->SetColor(Color(0.42f, 0.07f, 0.07f, 0.92f));
        quitBtn_->SetOnClick([this] { Quit(); });
        menuPanel_->AddChild(quitBtn_);

        // ── Gamepad D-pad focus navigation ────────────────────────────────────
        // SetFocusGroup defines the tab order: Resume → Quit → (wrap) → Resume.
        // D-Pad Right / Down moves forward; D-Pad Left / Up moves backward.
        // Pressing A fires SetOnClick on the focused button.
        // Pressing B fires the cancel callback (close the menu).
        SetFocusGroup({resumeBtn_, quitBtn_});
        EnableFocusNavigation(true);
        SetOnFocusCancel([this] { CloseMenu(); });
    }

    // ── Update() ─────────────────────────────────────────────────────────────
    void Update(float dt) override {
        // Spin the background box
        angle_ += 30.0f * dt;
        box_->SetRotation(Quaternion(angle_ * 0.55f, angle_, 0.0f));

        // Slowly drain health to demonstrate runtime color changes on a progress bar
        float hp = hpBar_->GetValue() - 0.008f * dt;
        if (hp < 0.0f) hp = 1.0f;   // wrap back to full for the demo
        hpBar_->SetValue(hp);

        // Color-code: green (healthy) → yellow (hurt) → red (critical)
        if (hp > 0.6f)
            hpBar_->SetColor(Color(0.2f, 0.85f, 0.25f, 1.0f));   // green
        else if (hp > 0.3f)
            hpBar_->SetColor(Color(0.9f, 0.75f, 0.1f, 1.0f));    // yellow
        else
            hpBar_->SetColor(Color(0.85f, 0.15f, 0.1f, 1.0f));   // red

        // Toggle settings menu on Escape
        if (IsKeyPressed(Key::Escape)) {
            if (menuPanel_->IsVisible())
                CloseMenu();
            else
                OpenMenu();
        }
    }

private:
    void OpenMenu() {
        menuPanel_->SetVisible(true);
        EnableFocusNavigation(true);
    }

    void CloseMenu() {
        menuPanel_->SetVisible(false);
        EnableFocusNavigation(false);
    }

    Entity*              box_            = nullptr;
    Camera*              cam_            = nullptr;

    UI::ProgressBar*     hpBar_          = nullptr;
    UI::ProgressBar*     xpBar_          = nullptr;

    UI::Panel*           menuPanel_      = nullptr;
    UI::Slider*          volSlider_      = nullptr;
    UI::Slider*          briSlider_      = nullptr;
    Label*               volValueLabel_  = nullptr;
    Label*               briValueLabel_  = nullptr;
    UI::Button*          resumeBtn_      = nullptr;
    UI::Button*          quitBtn_        = nullptr;

    float angle_ = 0.0f;
};

int main(int, char**) {
    auto game = Simple3D::CreateGame<UIAdvancedDemo>();
    game->SetResourcePrefixPaths("/path/to/u3d/bin");
    return game->Run();
}

Step-by-step explanation

1. CreateProgressBar()

hpBar_ = CreateProgressBar();
hpBar_->SetSize(200, 18);
hpBar_->SetPosition(12, 12);
hpBar_->SetColor(Color(0.2f, 0.85f, 0.25f, 1.0f));
hpBar_->SetBackgroundColor(Color(0.1f, 0.1f, 0.1f, 0.85f));
hpBar_->SetValue(0.78f);

CreateProgressBar() returns a UI::ProgressBar* that renders as two rectangles stacked in the same bounding box: a trough (background) that always fills the full widget size, and a fill that expands from left to right proportional to the current value.

SetColor() sets the fill colour — that is, the part that represents the actual quantity (health remaining, XP earned, download progress, etc.). In the example the fill starts green. SetBackgroundColor() sets the trough colour shown behind the fill. Using a dark semi-transparent background makes the bar readable on any scene.

SetValue(float v) accepts a normalised float in the range [0.0, 1.0]. Values outside this range are clamped automatically. A value of 0.0 shows an empty bar (only the trough is visible); 1.0 shows a completely filled bar; 0.78 shows 78 percent filled. You can call SetValue() at any time, including inside Update(), to animate the bar.

2. Runtime color coding with SetValue() and SetColor()

float hp = hpBar_->GetValue() - 0.008f * dt;
if (hp < 0.0f) hp = 1.0f;
hpBar_->SetValue(hp);

if (hp > 0.6f)
    hpBar_->SetColor(Color(0.2f, 0.85f, 0.25f, 1.0f));   // green
else if (hp > 0.3f)
    hpBar_->SetColor(Color(0.9f, 0.75f, 0.1f, 1.0f));    // yellow
else
    hpBar_->SetColor(Color(0.85f, 0.15f, 0.1f, 1.0f));   // red

GetValue() reads back the current fill fraction. In Update() we subtract a small delta-time-scaled amount each frame to create a slow drain animation, and call SetValue() to apply the new fraction.

After updating the value we immediately update the fill colour by calling SetColor() again with a different Color. This is a common pattern for health bars in games: the bar itself stays the same size; only the fill fraction and fill colour change. Calling SetColor() on a progress bar is cheap — it simply updates the vertex colour of the fill rectangle and has no impact on layout.

3. Vertical progress bars with SetVertical(true)

xpBar_ = CreateProgressBar();
xpBar_->SetSize(12, 80);
xpBar_->SetVertical(true);   // fill grows from bottom upward
xpBar_->SetValue(0.45f);

By default progress bars fill horizontally from left to right. Calling SetVertical(true) switches the fill direction so it grows from the bottom of the widget upward. This is useful for audio level meters, XP columns, tower charge indicators, or any quantity that is naturally represented as a vertical fill. The value semantics (0 = empty, 1 = full) are identical; only the rendering direction changes. The trough still occupies the full widget bounds.

4. CreateSlider(min, max)

volSlider_ = CreateSlider(0.0f, 1.0f);
volSlider_->SetSize(220, 20);
volSlider_->SetPosition(110, 75);
volSlider_->SetValue(0.8f);
volSlider_->SetColor(Color(0.3f, 0.55f, 1.0f, 1.0f));

CreateSlider(min, max) creates an interactive horizontal drag widget. The user clicks and drags the thumb knob along the slider track. min and max define the numeric range the slider represents. They are plain floats and can be any range you need: the volume slider uses (0.0, 1.0) for a 0–100% gain; the brightness slider uses (0.0, 2.0) so that the midpoint of travel corresponds to normal (100%) brightness.

SetValue(float v) positions the thumb at the given value, clamped to [min, max]. Call it in Start() to establish the initial position of the thumb, or at any time to set it programmatically (for example when loading a saved settings file).

GetValue() returns the current thumb position as a float in [min, max]. Use it when you need to read the value on demand (such as when a "Apply Settings" button is pressed) rather than reacting to each drag movement.

5. SetOnChange(lambda)

volSlider_->SetOnChange([this](float v) {
    SetMasterVolume(v);
    volValueLabel_->SetText(std::to_string(static_cast<int>(v * 100)) + "%");
});

SetOnChange() accepts a std::function<void(float)> that the engine calls every time the slider value changes. The single argument v is the new value in the slider's own [min, max] range — no need to convert it yourself.

The callback fires on every mouse-drag pixel, not just on mouse button release, so the audio volume in the example changes in real time as the user drags the knob. The companion label (volValueLabel_) is also updated inside the same callback so the numeric percentage always stays in sync with the knob position.

Lambda capture: [this] captures the game object by pointer so the callback can call methods like SetMasterVolume() and access member variables like volValueLabel_. This is the standard C++ idiom for UI callbacks in simple-3d.

6. panel->AddChild(element) — UI hierarchy

// Create the panel
menuPanel_ = CreatePanel();
menuPanel_->SetSize(370, 310);

// Create a child element — position is relative to the panel top-left
Label* titleLabel = CreateLabel("SETTINGS");
titleLabel->SetPosition(133, 20);   // 133px right, 20px down from panel origin
menuPanel_->AddChild(titleLabel);   // reparents; position is now panel-relative

By default every UI widget you create is a child of the UI root (the screen). Calling parent->AddChild(child) moves the child out of the root and makes it a direct child of parent. From that point on:

  • The child's SetPosition() values are interpreted as pixel offsets from the parent's top-left corner, not from the top-left of the screen.
  • When the parent moves, all children move with it automatically.
  • When the parent is hidden with SetVisible(false), the entire subtree is hidden without needing to iterate children manually.
  • When the parent is destroyed, all children are destroyed with it.

The method is overloaded for both UI::Element* (panels, buttons, sliders, progress bars, images) and Label* (labels live in a slightly different class hierarchy but are accepted by the same AddChild call).

In the example every widget inside the settings menu is added to menuPanel_. When the user presses Escape, calling menuPanel_->SetVisible(false) hides the title label, both sliders, their value labels, the divider, and both buttons in a single call. No loop over children is required.

7. SetZOrder() — layer ordering

// HUD widgets
hpBar_->SetZOrder(1);
xpBar_->SetZOrder(1);

// Settings menu panel — on top of the HUD
menuPanel_->SetZOrder(10);

SetZOrder(int z) controls the rendering order of UI elements when they overlap on screen. Elements with a higher Z-order value are drawn on top. The default Z-order for newly created elements is 0.

In the example the HUD bars sit at Z-order 1, slightly above the default (0) so they render on top of any plain background panels you might add later. The settings menu panel sits at Z-order 10, guaranteeing it is always visible on top of the HUD when shown — no matter how many HUD elements you add in the future, as long as they stay below 10.

Z-order applies to the element itself and is not inherited by children. If you need all children of a panel to appear above an external element, give the panel a high Z-order and then use small relative Z-orders among the children.

Tip — reserved Z-order values

The engine uses Z-order 0 for scene-attached labels and Z-order 100 and above for the engine's own debug overlays. Keep your game UI widgets in the range 1–90 to avoid conflicts. Good conventions: HUD at 1–5, in-game menus at 10–20, dialog boxes at 30, fade-to-black overlays at 50.

8. Anchor::Center — screen-centered panels

menuPanel_->SetAnchor(UI::Anchor::Center);
menuPanel_->SetPosition(0, 0);    // (0,0) = exactly centered, no offset

Setting an anchor on a UI element changes what the SetPosition() call means. Without an anchor, position is measured from the screen's top-left corner to the element's own top-left corner. With Anchor::Center, position is measured from the screen's geometric center to the element's own geometric center. Passing (0, 0) therefore places the panel's centre exactly at the screen's centre.

The panel stays centered even if you resize the window at runtime, because the anchor is re-evaluated every frame relative to the current window dimensions. To nudge the panel away from the exact centre, use a non-zero SetPosition offset: for example SetPosition(0, -30) would shift the panel 30 pixels above the exact centre, which is a common trick for placing menu dialogs slightly above the mathematical midpoint for a more pleasing visual result.

Other available anchor values:

ValueScreen anchor point
Anchor::TopLeftTop-left corner (default)
Anchor::TopCenterTop edge, horizontally centered
Anchor::TopRightTop-right corner
Anchor::CenterLeftLeft edge, vertically centered
Anchor::CenterExact screen centre
Anchor::CenterRightRight edge, vertically centered
Anchor::BottomLeftBottom-left corner
Anchor::BottomCenterBottom edge, horizontally centered
Anchor::BottomRightBottom-right corner

9. Gamepad D-pad focus navigation

SetFocusGroup({resumeBtn_, quitBtn_});
EnableFocusNavigation(true);
SetOnFocusCancel([this] { CloseMenu(); });

These three calls wire up gamepad controller navigation for the buttons inside the settings menu. The engine's focus system handles D-pad input automatically once you configure it.

SetFocusGroup(vector<UI::Element*>) registers an ordered list of focusable elements and immediately highlights the first element in the list (the Resume button in this case). The order of the list defines the navigation order: D-Pad Down or D-Pad Right moves to the next element; D-Pad Up or D-Pad Left moves to the previous element. The list wraps around at both ends.

EnableFocusNavigation(bool) activates or deactivates the D-pad input handler. In the example it is enabled when the menu opens and disabled when the menu closes. Leaving it enabled while the menu is hidden would still consume D-pad input, which could interfere with gameplay that uses D-pad for other actions.

While a button is focused, pressing the A button fires that button's SetOnClick callback exactly as if the user had clicked it with the mouse. No additional code is required to wire up the A button.

SetOnFocusCancel(callback) registers a callback for the B button. In the example the cancel action closes the menu, matching the conventional game-UI pattern where B dismisses the current screen. The cancel callback is fired even if no element is currently focused, so it provides a reliable "escape hatch" for any controller user.

Focus navigation and sliders

The current focus system is designed for buttons. Sliders inside the panel are interacted with via mouse click-and-drag. If you need full controller support for sliders (adjusting the value with the D-pad or analog stick), read the slider value with GetValue() in Update() and call SetValue() manually when the relevant stick or D-pad axis is moved while the slider has logical focus.

10. The open/close menu pattern

void OpenMenu() {
    menuPanel_->SetVisible(true);
    EnableFocusNavigation(true);
}

void CloseMenu() {
    menuPanel_->SetVisible(false);
    EnableFocusNavigation(false);
}

Factoring the open and close operations into private helper methods keeps Update() clean and ensures the two operations are always symmetrical. Every code path that opens the menu (keyboard Escape, a "settings" button, etc.) calls OpenMenu(); every path that closes it (Resume button click, B button cancel, Escape again) calls CloseMenu(). There is no risk of accidentally leaving focus navigation enabled while the menu is hidden.

Note that SetFocusGroup() only needs to be called once in Start() because the element list never changes. EnableFocusNavigation(true/false) is called repeatedly to gate the D-pad input on and off.

Lua equivalent

Lua note

Lua callbacks for SetOnChange and SetOnClick use plain anonymous functions. The Anchor enum is accessed as Anchor.Center (dot notation, not ::). The SetFocusGroup call accepts a Lua table used as an array.

Common mistakes

Common mistakes — Part 18

  • Calling AddChild() and then SetPosition() in the wrong order. The position must be set to the panel-relative coordinates you intend before calling AddChild(). If you call SetPosition() after AddChild(), the position is still applied, but if you previously set it to screen-space coordinates, those will now be interpreted as panel-relative offsets and the element will appear in the wrong location.
  • Forgetting to set SetZOrder on overlapping UI layers. If the settings menu appears behind the HUD health bar, it is because both are at the default Z-order of 0. The drawing order is unspecified when Z-orders are equal. Always assign explicit Z-orders to any elements that you know may overlap.
  • Leaving EnableFocusNavigation(true) when the menu is hidden. If focus navigation is enabled while the menu panel is hidden, pressing the D-pad still moves focus between the (invisible) buttons and pressing A fires their click callbacks silently. Always pair menuPanel->SetVisible(false) with EnableFocusNavigation(false).
  • Passing the slider's min/max range incorrectly to display logic. SetOnChange(float v) gives you v in the slider's own [min, max] range, not in [0, 1]. If the brightness slider range is (0.0, 2.0), v = 1.0 corresponds to 100% and you display it as static_cast<int>(v * 100) which gives 100. A common bug is forgetting this scaling and displaying static_cast<int>(v * 100) for a (0, 2) slider, which would print 200 at max instead of 100. Decide upfront whether to scale inside the callback or in the display string.
  • Confusing Anchor::Center with zero-position. SetAnchor(UI::Anchor::Center) does not set the position to the screen centre by itself. It changes what zero means. Without the anchor call, SetPosition(0, 0) places the element at the top-left corner of the screen. With the anchor, SetPosition(0, 0) places the element's center at the screen center. You need both the anchor call and SetPosition(0, 0) to achieve exact centering.
  • Using SetValue() before SetOnChange() and expecting the callback to fire. Calling SetValue(0.8f) in Start() to set the initial thumb position does not fire the SetOnChange callback if the callback has not been registered yet. Always register the callback before calling SetValue() if you need the initial value to trigger side effects (like updating a companion label). Alternatively, explicitly update the label after setting the initial value.
  • Not assigning a value label before the SetOnChange lambda captures it. If volValueLabel_ is nullptr when the SetOnChange lambda runs, you will get a null pointer dereference. Create and assign all widgets the lambda references before registering the callback. In the example volValueLabel_ is assigned just before the SetOnChange call for exactly this reason.

Try it yourself

  1. Add a third slider to the settings panel for "Music Volume" with a range of (0.0f, 1.0f). Wire its SetOnChange to print the value to the window title with SetWindowTitle("Music: " + std::to_string(int(v * 100)) + "%") so you can confirm it is firing.
  2. Change the XP bar to be horizontal and make it refill slowly over time (the reverse of the health drain). Observe how GetValue() reads back the current fraction each frame.
  3. Create a second panel with a "Load" and "Delete" button at Z-order 20, and verify that it renders on top of the settings panel (Z-order 10) when both are visible simultaneously. Then try swapping the Z-orders and observe which one is drawn on top.
  4. Add a third button "Options" to the focus group so the D-pad cycles through Resume → Options → Quit → Resume. Notice that SetFocusGroup must be called again with the new three-element list.
  5. Change the settings panel anchor to UI::Anchor::BottomCenter and set SetPosition(0, -20). Observe the panel appearing at the bottom-center of the screen with a 20-pixel upward offset, useful for dialog-bar style menus.

What you learned

This part introduced the remaining interactive UI widgets and three structural techniques that make complex UIs manageable:

  • CreateProgressBar() creates a normalised fill widget driven by SetValue(0–1). SetColor() controls the fill colour and can be called at any time for dynamic color coding. SetBackgroundColor() sets the trough. SetVertical(true) switches to a bottom-up fill direction.
  • CreateSlider(min, max) creates a draggable thumb widget with a user-defined numeric range. SetValue() positions the thumb; GetValue() reads it. SetOnChange(lambda) fires a callback on every drag event with the current value as its argument.
  • panel->AddChild(element) reparents a widget so its position is relative to the panel's top-left corner. Hiding or destroying the parent automatically affects the entire subtree. This is the primary mechanism for building reusable, toggleable UI groups like menus and dialog boxes.
  • SetZOrder(int) controls the rendering order of overlapping UI elements. Higher values render on top. Assign explicit Z-orders to any layers that can overlap to avoid undefined drawing order.
  • SetAnchor(UI::Anchor::Center) redefines what SetPosition(0, 0) means: the center of the element is placed at the named anchor point on the screen. Combined with SetPosition(0, 0), this is the simplest way to center a panel on any resolution.
  • SetFocusGroup(), EnableFocusNavigation(), and SetOnFocusCancel() together implement gamepad D-pad navigation between buttons. The focus group defines tab order; D-Pad Down/Right and D-Pad Up/Left cycle through it; A fires the focused button's click callback; B fires the cancel callback. Enable and disable navigation in sync with panel visibility.
