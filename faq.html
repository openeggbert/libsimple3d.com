Frequently Asked Questions

Answers to common questions about simple-3d. If your question is not covered here, open an issue on the GitHub repository.

General

What is simple-3d?

simple-3d is a high-level C++23 game framework built as a clean, engine-agnostic wrapper around an Urho3D fork. It exposes a minimal, game-first API so you can write game logic without touching any engine internals.

You subclass Game, override Start() and Update(float dt), call methods like entity->AddRigidBody(1.0f) or camera->Follow(player, 6.0f), and ship. The underlying Urho3D types (Urho3D::Node, Urho3D::Scene, etc.) are never visible in public headers — they live behind PIMPL classes.

Why use simple-3d instead of raw Urho3D?

Urho3D is a powerful engine but its API exposes a lot of engine-level concepts: subsystems, component registration, context objects, string hashes, and so on. simple-3d hides all of that behind a game-oriented surface.

Practically: creating a physics object in raw Urho3D means calling GetOrCreateComponent<RigidBody>, GetOrCreateComponent<CollisionShape>, setting shape type, setting mass, registering with the world, and configuring collision masks. In simple-3d that is two lines: 

entity->AddRigidBody(1.0f);
entity->AddCapsuleCollider(0.4f, 1.75f);

Additionally, because the API is stable and engine types never appear in public headers, you can switch from the U3D backend to NOVA3D with a single CMake flag and zero changes to your game code.

What is the difference between the U3D and NOVA3D backends?

Both backends implement the same public simple-3d API. The difference is the underlying engine build they link against:

  • U3D (default) — links against the u3d-community/U3D Urho3D community fork. Targets Linux, Windows, and macOS desktop. Stable Bullet physics, LuaJIT scripting, Recast/Detour navigation, ENet networking.
  • NOVA3D — links against Robert Vokac's Urho3D fork, which adds an XNA C++ runtime underneath for rendering and audio. The primary target for Android and experimental hardware. Identical public API surface; identical game source compiles against either backend without changes.
Can I use simple-3d without Lua?

Yes. Lua scripting is entirely optional. You can write all game logic in C++ and never call entity->AddScript(). LuaJIT is still linked as part of the Urho3D backend, but no Lua state is created unless you attach at least one script to an entity.

Is simple-3d production-ready?

The core modules (Entity, Camera, Physics 3D/2D, Animation, Navigation, Networking, Audio, 2D sprites, Tile maps, Labels) are complete and covered by 78 unit tests. The Lua binding layer is in progress and some planned modules (full UI system, scene management, spatial audio, Android port) are not yet implemented.

For small-to-medium games and prototypes using the existing modules, simple-3d is stable. For large productions that depend on the planned features, consider waiting for those milestones or contributing them.

What license does simple-3d use?

simple-3d is open source. See the LICENSE file in the repository for the full text. The underlying Urho3D fork (U3D) is distributed under the MIT license.

Building & Setup

How do I switch from the U3D backend to NOVA3D?

Pass -DSIMPLE3D_ENGINE=NOVA3D and -DNOVA3D_HOME to CMake instead of the U3D equivalents. No changes to game source are required. 

# U3D (default)
cmake -S . -B build \
  -DSIMPLE3D_ENGINE=U3D \
  -DU3D_HOME=/path/to/u3d/build

# NOVA3D
cmake -S . -B build \
  -DSIMPLE3D_ENGINE=NOVA3D \
  -DNOVA3D_HOME=/path/to/nova3d/build
How do I run the unit tests, and how many should pass?

Enable tests at configure time with -DSIMPLE3D_BUILD_TESTS=ON, then build and run with CTest: 

cmake -S . -B build -DSIMPLE3D_BUILD_TESTS=ON -DU3D_HOME=/path/to/u3d/build
cmake --build build -j$(nproc)
cd build && ctest --output-on-failure

All 78 tests should pass. If any fail, check that the U3D (or NOVA3D) build was compiled with the same compiler and C++ standard as simple-3d itself.

I get "Could not find U3DConfig.cmake" — how do I fix it?

CMake cannot locate the Urho3D/U3D installation. Set -DU3D_HOME to the root of your U3D build output directory (the directory that contains lib/cmake/U3D/U3DConfig.cmake): 

cmake -S . -B build -DU3D_HOME=/absolute/path/to/u3d/build

Make sure you built U3D first and that the path is absolute, not relative. If you installed U3D to a system prefix, you can also set -DCMAKE_PREFIX_PATH=/usr/local instead.

I get "std::expected not declared" — what compiler do I need?

simple-3d uses C++23 features including std::expected. You need a recent compiler:

  • GCC 13 or newer
  • Clang 16 or newer
  • MSVC 19.36 (Visual Studio 2022 17.6) or newer

Verify your version with g++ --version or clang++ --version. On Ubuntu 22.04 you may need to install a newer toolchain from the ubuntu-toolchain-r PPA or use the LLVM apt repository.

LuaJIT is not found during the build — how do I install it?

LuaJIT is required by the Urho3D backend. Install the development package for your distribution: 

# Debian / Ubuntu
sudo apt install libluajit-5.1-dev

# Fedora / RHEL
sudo dnf install luajit-devel

# Arch Linux
sudo pacman -S luajit

# macOS (Homebrew)
brew install luajit

After installing, re-run the CMake configure step. If CMake still cannot find it, set -DLUAJIT_INCLUDE_DIR and -DLUAJIT_LIBRARY explicitly.

API Usage

Can I mix 2D and 3D in the same scene?

Yes. simple-3d runs Bullet (3D) and Box2D (2D) physics worlds simultaneously in the same scene. You can have 3D rigid bodies and 2D rigid bodies on different entities in the same scene without any conflict. The two worlds operate independently and do not interact with each other; it is your responsibility to use the correct API calls for each entity.

Why can't I call engine API methods from my Game constructor?

The engine is not fully initialized until the framework calls Start(). The constructor runs before the window is created, the scene graph is set up, or any subsystem is ready. Calling methods like CreateEntity() or PlayMusic() from the constructor results in undefined behavior or a crash.

Always perform all initialization — entity creation, camera setup, audio start, script loading — inside Start() or later.

How do I safely destroy an entity from inside a physics callback?

Destroying an entity while the physics simulation is iterating over its contact list is unsafe. simple-3d provides a deferred mechanism: call DestroyEntity(entity) from within any collision callback (e.g. OnCollisionEnter, OnTriggerEnter). The destruction is queued and executed at the start of the next frame, after the physics step completes. You do not need to guard the call or defer it manually.

What are the built-in collision layers and what do they mean?

simple-3d defines three default collision layer constants used when adding rigid bodies and colliders:

NameBit valueUsed for
CollisionLayer::Actor1Player characters, enemies — dynamic bodies that move
CollisionLayer::StaticGeometry2Ground, walls, platforms — zero-mass static bodies
CollisionLayer::Trigger4Invisible trigger volumes — no physical response, events only

You can combine layers with bitwise OR when calling AddRigidBody or SetCollisionLayer. Custom layers can use higher bit values.

How do I persist data between scene loads?

Two mechanisms are available:

  • SetPersistentEntity(entity) — marks an entity so it is not destroyed when LoadScene() is called. Useful for a player entity or a game manager that must survive across levels.
  • SaveData / LoadData — a key-value store for primitive values (scores, flags, settings) that persists independently of the scene graph. Data is not tied to any entity and survives both scene loads and game restarts when written to disk.
How do I support both gamepad and keyboard with the same input action?

Use the input action binding API to map a logical action to multiple physical inputs. BindAction lets you assign a named action to a keyboard key, mouse button, or gamepad button — whichever fires first triggers the action. For analogue movement use BindAxis2D to map both keyboard WASD and a gamepad stick to the same axis: 

// In Start()
BindAction("Jump", Key::Space);
BindAction("Jump", GamepadButton::South);

BindAxis2D("Move", Key::W, Key::S, Key::A, Key::D);
BindAxis2D("Move", GamepadAxis::LeftStick);

// In Update(dt)
if (IsActionPressed("Jump")) { /* ... */ }
Vector2 move = GetAxis2D("Move");

Lua

How do I access the entity that owns the current Lua script?

Inside a Lua script attached with entity->AddScript(), the owning entity is available as self.node. By convention, simple-3d scripts alias it to a local called entity at the top of Start() for readability: 

function Start()
    entity = self.node   -- simple-3d convention
    -- now use entity:SetPosition(), entity:ApplyImpulse(), etc.
end

Both self.node and entity refer to the same object; using the alias is purely a style choice.

Can I call C++ API methods from Lua?

Yes. All major simple-3d systems are exposed to Lua. From a script you can call entity methods (entity:MoveRelative, entity:ApplyImpulse, entity:IsOnGround, etc.), access input via the global Input table, play audio with PlaySound and PlayMusic, and query the animation controller via entity:GetAnimationController(). See the Lua API reference for the full list of bindings.

How do I check whether a Lua script has syntax errors before running?

Use the luac compiler in check-only mode. It parses the script without executing it and prints any syntax errors with line numbers: 

luac -p Scripts/player.lua

No output means the script is syntactically valid. This is useful in a CI pipeline or as a pre-run check. Note that luac -p catches syntax errors only — runtime errors (calling a nil value, wrong argument types) are still reported at execution time.

What happens if Lua code calls a method on an entity that has already been destroyed?

simple-3d wraps all entity references exposed to Lua with a liveness guard. If a Lua script calls a method (e.g. entity:SetPosition(0,0,0)) on an entity that has been destroyed, the call is a safe no-op — it does nothing and does not throw a Lua error. This prevents crashes that would otherwise occur if, for example, a script holds a reference to an entity that was destroyed by a different script or by a collision callback.

Assets

What 3D model formats does simple-3d support?

simple-3d (via the Urho3D backend) supports loading OBJ, FBX, GLTF, and MDL model files. MDL is the preferred format for runtime use: it is Urho3D's native binary format, loads faster than text-based formats, and stores skeleton and animation data in a compact layout. OBJ, FBX, and GLTF are typically used as interchange formats and converted to MDL before shipping.

How do I convert an FBX file to MDL?

Use the AssetImporter command-line tool that ships with Urho3D/U3D: 

AssetImporter model character.fbx Models/Character.mdl
AssetImporter anim  character.fbx Animations/Run.ani

Run AssetImporter --help for the full list of flags (scale factor, axis orientation, LOD generation, etc.). The tool is built alongside the engine and is located in <U3D_BUILD>/bin/tool/AssetImporter.

Where should I place asset files so the game can find them?

Place all assets in a directory named Data/ located next to the game executable. simple-3d registers this directory as a resource path on startup. The path you pass to API calls is relative to Data/; for example: 

entity->AddModel("Models/Character.mdl");   // loads Data/Models/Character.mdl
PlayMusic("Music/theme.ogg");              // loads Data/Music/theme.ogg

You can register additional resource paths with AddResourcePath(path) from inside Start() if you want to load assets from other locations.

What audio formats are supported, and when should I use each?

simple-3d supports two audio formats:

  • OGG Vorbis — use for background music and long ambient tracks. OGG files are decoded and streamed from disk at runtime, so they use very little memory regardless of duration. Pass the path to PlayMusic().
  • WAV — use for short sound effects (footsteps, gunshots, UI clicks). WAV files are decoded and loaded entirely into memory, which gives low-latency playback without buffering delays. Pass the path to PlaySound().
What tile map format is supported?

simple-3d supports the Tiled .tmx XML format. Both orthographic and isometric map orientations are supported. Load a tile map with: 

LoadTileMap("Maps/level1.tmx");

The tile set images referenced by the TMX file must also be present in the Data/ directory at the relative paths defined inside the TMX file.

What animated sprite format is supported?

simple-3d supports Spriter .scml files for 2D skeletal / bone-based sprite animation. Load an animated sprite on an entity with: 

entity->AddAnimatedSprite2D("Sprites/hero.scml");

The .scml file and all referenced image assets must be present under Data/. Spriter Pro and Spriter Free both export compatible .scml files. Frame-by-frame sprite sheet animation (atlas-based) is also supported separately via AddSprite2D with manual frame control.
