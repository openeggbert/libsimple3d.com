Part 12 of 302D Mode

New in this part

What you will learn

  • How simple-3d's 2D mode works: orthographic projection on top of the same scene graph used for 3D.
  • How to switch the camera to orthographic projection with SetOrthographic(true) and control the zoom level with SetOrthoSize().
  • Why the 2D coordinate system lives in the XY plane at Z = 0, and why the camera sits at Z = −10.
  • How to render a single static image on an entity using AddSprite2D().
  • How to mirror a sprite horizontally with SetFlipX2D() so a character faces left or right without needing separate art.
  • How to load a Spriter .scml animation with AddAnimatedSprite2D() and switch clips with PlayAnimation2D().
  • How to load a Tiled tile map (.tmx file) with a single LoadTileMap() call.
  • How to follow a 2D entity with the camera while keeping the Z axis fixed.
  • The equivalent Lua API for all 2D rendering calls.

Overview: how 2D mode works

simple-3d does not have a separate engine mode for 2D games. Instead, 2D games are built using the same scene graph, entity system, and rendering pipeline as 3D games, but with two important differences:

  1. The camera is switched to orthographic projection. In perspective projection (the 3D default) objects shrink as they move away from the camera. In orthographic projection all objects are drawn at the same size regardless of distance. This is the projection used by every classic 2D game.
  2. Entities live on the XY plane at Z = 0. X is the horizontal axis (positive = right), Y is the vertical axis (positive = up). Z is used only for depth ordering — entities with a higher Z value are rendered behind entities with a lower Z value (because the camera looks toward positive Z from its position at Z = −10).

Everything else — entity creation, input, physics (via Box2D), audio, UI labels — works identically to the 3D tutorials you have already completed.

The 2D coordinate system

Understanding the coordinate system is critical before writing any 2D code. Here is how the three world axes are used in 2D mode:

Axis Direction 2D meaning
X Positive = right Horizontal position in the world. A coin at X = 5 is 5 world units to the right of the origin.
Y Positive = up Vertical position. A platform at Y = −3 is 3 world units below the centre.
Z Positive = away from camera Depth / draw order only. All game entities sit at Z = 0. The camera sits at Z = −10.

Camera position in 2D

Always place the 2D camera at Z = −10 (or any sufficiently negative value). It looks toward positive Z, so entities at Z = 0 are in front of it. Setting Z to 0 or a positive value puts the camera inside or behind the scene and you will see nothing. The near clip plane is typically set to 0.1 and the far clip to 100 so that Z = 0 entities are well within the visible range.

The same rule applies to all 2D API methods: AddRigidBody2D, AddBoxCollider2D, GetLinearVelocity2D, and IsOnGround2D all operate in this XY plane and ignore Z entirely (2D physics is covered in depth in Part 13). When you set an entity's position in 2D code, always leave Z at 0.0f.

Orthographic camera setup

A regular simple-3d camera uses perspective projection. To switch to 2D you need to enable orthographic projection and set the visible height in world units: 

Camera* cam = CreateCamera();

// Enable orthographic projection (disables perspective foreshortening).
cam->SetOrthographic(true);

// Set how many world units are visible from the bottom to the top of the screen.
// At 12.0f the visible window is 12 units tall.
// The width is derived automatically from the window aspect ratio.
cam->SetOrthoSize(12.0f);

// Clip range: 0.1 near, 100 far. Z=0 entities are well within range.
cam->SetNearClip(0.1f);
cam->SetFarClip(100.0f);

// Place the camera at Z=-10 looking toward positive Z.
cam->SetPosition(0.0f, 0.0f, -10.0f);

For pixel-art games there is a convenience method that combines all of the above into a single call: 

// 100 pixels = 1 world unit.
// Internally calls SetOrthographic(true), SetOrthoSize(...), and positions the camera.
cam->Setup2D(100.0f);

Setup2D(ppu) takes a pixels-per-unit value. With a window height of 720 px and 100 ppu, the visible height is 720 / 100 = 7.2 world units. This lets you think in pixels when laying out sprites while still using floating-point world coordinates in code.

How SetOrthoSize controls zoom

SetOrthoSize(height) sets the number of world units visible from the bottom edge to the top edge of the screen. The width is computed automatically from the window aspect ratio. For example:

OrthoSize Visible height Visible width (16:9) Effect
6.0f 6 world units 10.67 world units Zoomed in — each world unit is large on screen
12.0f 12 world units 21.33 world units Normal — good starting point for most side-scrollers
24.0f 24 world units 42.67 world units Zoomed out — shows more of the level at once

You can change the zoom level at runtime by calling SetOrthoSize() again in Update(). This is the basis of zoom-in/zoom-out effects.

Static sprites: AddSprite2D

AddSprite2D(texturePath) attaches a StaticSprite2D component to the entity. The sprite displays the given texture centred on the entity's position. The texture path is relative to a Data/ directory, exactly like model and sound paths. 

// Create a coin at position (4, 2, 0) in the world.
Entity* coin = CreateEntity("Coin");
coin->SetPosition(4.0f, 2.0f, 0.0f);   // Z = 0 for all 2D entities
coin->AddSprite2D("Sprites/coin.png");

The sprite is rendered in the XY plane facing the camera. Because the camera is orthographic and positioned at Z = −10, the sprite appears as a flat 2D image with no perspective distortion.

Sprite size in world units is determined by the texture's pixel dimensions divided by the pixels-per-unit configured in Setup2D(), or by the entity's scale. For example, a 64×64 texture with 100 ppu is 0.64×0.64 world units. Call entity->SetScale(2.0f) to double the displayed size without changing the texture.

Mirroring sprites: SetFlipX2D

When a character walks left and right, you normally need to mirror the sprite rather than supplying two separate sets of artwork. SetFlipX2D(true) flips the sprite horizontally so it faces left; SetFlipX2D(false) restores the default facing. 

// Player walks right — show sprite facing right (default).
player->SetFlipX2D(false);

// Player walks left — mirror the sprite horizontally.
player->SetFlipX2D(true);

Call this once per movement direction change, not every frame. The typical pattern is to track the last movement direction and only update the flip flag when it changes: 

float mx = 0.0f;
if (IsKeyDown(Key::A)) mx = -1.0f;
if (IsKeyDown(Key::D)) mx =  1.0f;

// Only flip when direction is non-zero so idle does not reset facing.
if (mx < 0.0f) player_->SetFlipX2D(true);
if (mx > 0.0f) player_->SetFlipX2D(false);

Animated sprites: AddAnimatedSprite2D and PlayAnimation2D

simple-3d supports animated 2D sprites using the Spriter format (.scml files). Spriter is a free bone-and-sprite animation tool that exports a self-contained .scml file referencing individual sprite sheet images. The .scml file and all referenced texture files must be placed in your Data/ directory.

Adding an animated sprite is a two-step process:

  1. Call AddAnimatedSprite2D(scmlPath) to load the Spriter animation and attach an AnimatedSprite2D component to the entity.
  2. Call PlayAnimation2D(name) to start a named animation clip. The clip name must match a name defined inside the .scml file.
// Step 1: Load the Spriter file and attach it to the entity.
player->AddAnimatedSprite2D("Sprites/hero.scml");

// Step 2: Start the "idle" clip (looped by default).
player->PlayAnimation2D("idle");

PlayAnimation2D accepts two optional parameters: a loop flag (default true) and a speed multiplier (default 1.0f): 

// Looped run animation at normal speed (default).
player->PlayAnimation2D("run");

// One-shot jump animation at 1.5x speed — does not loop.
player->PlayAnimation2D("jump", false, 1.5f);

// Looped walk animation at half speed.
player->PlayAnimation2D("walk", true, 0.5f);

Switch clips at any time by calling PlayAnimation2D again. The new clip starts from the beginning immediately. To switch clips smoothly (for example, from "run" to "idle" when the player stops), call it in your Update() method whenever the game state changes: 

bool moving = (mx != 0.0f);
if (moving != wasMoving_) {
    player_->PlayAnimation2D(moving ? "run" : "idle");
    wasMoving_ = moving;
}

Tile maps: LoadTileMap

Tile maps let you design levels visually in Tiled, a free open-source level editor, and load them directly into simple-3d. Tiled exports levels as .tmx files (XML-based). simple-3d uses Urho3D's built-in TileMap2D component to render and parse them. 

// Load a Tiled .tmx file. The path is relative to Data/.
// Returns an Entity* that owns the TileMap2D component.
Entity* map = LoadTileMap("Maps/level1.tmx");

LoadTileMap() does several things automatically:

  • Creates an entity with a TileMap2D component and renders all tile layers.
  • Reads any Tiled object layers that have collision shapes defined and creates static RigidBody2D and CollisionShape2D components for them. This means your level's physics boundaries are set up for free just by marking objects in Tiled.
  • Sets the map entity's position to the world origin. To shift the level, call map->SetPosition() after the load.

Tiled workflow

In Tiled, create a tile layer for your visual tiles and a separate object layer for collision. Draw rectangles or polygons on the object layer where you want solid ground and walls. simple-3d reads these as Box2D static bodies automatically when you call LoadTileMap(). No additional code is needed to set up tile collision.

The PhysicsWorld2D component is added to the scene automatically when Game::Run() starts, so tile map collision works out of the box. You do not need to create or configure the physics world manually for 2D scenes.

Complete example

The program below builds a complete 2D side-scroller prototype. It demonstrates:

  • Orthographic camera with a custom OrthoSize
  • A static background sprite
  • A Tiled tile map for the level layout
  • A player entity with an animated Spriter sprite
  • Left/right movement with sprite flipping
  • Camera following the player (X axis only, Y pinned)
  • Animation switching between "idle" and "run"

For the example to run you need three asset files placed in your Data/ directory: Sprites/background.png, Sprites/hero.scml (with its referenced textures), and Maps/level1.tmx. 

// Game2DDemo.cpp — Part 12 of the simple-3d tutorial series
#include <Simple3D/Simple3D.h>

using namespace Simple3D;

class Game2DDemo : public Game {
public:

    // ── Scene objects ─────────────────────────────────────────────────
    Camera* cam_     = nullptr;
    Entity* player_  = nullptr;
    Entity* bg_      = nullptr;
    Label*  hud_     = nullptr;

    // ── State ─────────────────────────────────────────────────────────
    bool  wasMoving_ = false;
    float playerX_   = 0.0f;

    static constexpr float MOVE_SPEED  = 5.0f;   // world units per second
    static constexpr float ORTHO_SIZE  = 12.0f;  // visible world units (height)
    static constexpr float CAM_Z       = -10.0f; // camera must be behind the XY plane

    void Start() override {
        SetWindowTitle("Part 12 — 2D Mode");
        SetClearColor(Color(0.53f, 0.81f, 0.98f));   // sky blue

        // ── 1. Orthographic camera ───────────────────────────────────────
        // SetOrthographic(true) switches off perspective foreshortening.
        // SetOrthoSize(12) means the screen shows 12 world units vertically.
        // Camera Z = -10 so it looks toward the XY plane at Z = 0.
        cam_ = CreateCamera();
        cam_->SetOrthographic(true);
        cam_->SetOrthoSize(ORTHO_SIZE);
        cam_->SetNearClip(0.1f);
        cam_->SetFarClip(100.0f);
        cam_->SetPosition(0.0f, 0.0f, CAM_Z);

        // ── 2. Background sprite ─────────────────────────────────────────
        // A large static sprite placed behind everything else.
        // We push it slightly further away (Z = 1) so it renders behind
        // the player and tile map, which sit at Z = 0.
        bg_ = CreateEntity("Background");
        bg_->SetPosition(0.0f, 0.0f, 1.0f);   // Z > 0 = drawn behind Z=0 objects
        bg_->SetScale(24.0f);                    // stretch to fill the view
        bg_->AddSprite2D("Sprites/background.png");

        // ── 3. Tile map ───────────────────────────────────────────────────
        // LoadTileMap() reads a Tiled .tmx file, renders all tile layers,
        // and automatically creates Box2D static colliders from object layers.
        // The entity is created and owned by the engine.
        Entity* map = LoadTileMap("Maps/level1.tmx");
        // Shift the map down so the ground centre is at Y = -4.
        map->SetPosition(0.0f, -4.0f, 0.0f);

        // ── 4. Player entity ──────────────────────────────────────────────
        // Place the player near the centre of the visible area.
        // All 2D entities must be at Z = 0 (or close to it).
        player_ = CreateEntity("Player");
        player_->SetPosition(0.0f, 0.0f, 0.0f);

        // AddAnimatedSprite2D loads a Spriter .scml file.
        // The .scml and all referenced texture files must be in Data/.
        player_->AddAnimatedSprite2D("Sprites/hero.scml");

        // Start with the idle animation (looped, normal speed).
        player_->PlayAnimation2D("idle");

        // ── 5. A coin using a static sprite ───────────────────────────────
        Entity* coin = CreateEntity("Coin");
        coin->SetPosition(4.0f, 1.5f, 0.0f);
        coin->AddSprite2D("Sprites/coin.png");

        // ── 6. HUD ────────────────────────────────────────────────────────
        hud_ = CreateLabel("A/D or Arrows = move  |  ESC = quit");
        hud_->SetPosition(12, 12);
        hud_->SetFontSize(16);
        hud_->SetColor(Color::WHITE);
    }

    void Update(float dt) override {
        if (IsKeyPressed(Key::Escape)) Quit();

        // ── Read horizontal input ──────────────────────────────────────────
        float mx = 0.0f;
        if (IsKeyDown(Key::A) || IsKeyDown(Key::Left))  mx = -1.0f;
        if (IsKeyDown(Key::D) || IsKeyDown(Key::Right)) mx =  1.0f;

        // ── Move player (direct position update — physics in Part 13) ──────
        // In 2D the player moves along X only.
        // Y is handled by Box2D gravity once we add a RigidBody2D.
        playerX_ += mx * MOVE_SPEED * dt;
        player_->SetPosition(playerX_, 0.0f, 0.0f);

        // ── Flip sprite to face movement direction ─────────────────────────
        // SetFlipX2D(true) mirrors the sprite horizontally (faces left).
        // Only update when direction is non-zero so idle keeps the last facing.
        if (mx < 0.0f) player_->SetFlipX2D(true);
        if (mx > 0.0f) player_->SetFlipX2D(false);

        // ── Switch animation clip on state change ──────────────────────────
        // PlayAnimation2D restarts the named clip from frame 0.
        // We only call it when the moving state changes to avoid restarting
        // the animation every frame.
        bool moving = (mx != 0.0f);
        if (moving != wasMoving_) {
            player_->PlayAnimation2D(moving ? "run" : "idle");
            wasMoving_ = moving;
        }

        // ── Camera follow (X axis only) ────────────────────────────────────
        // Keep the camera centred on the player horizontally.
        // Fix Y at 0 and Z at CAM_Z — do not let the camera drift vertically
        // or move toward the XY plane.
        cam_->SetPosition(playerX_, 0.0f, CAM_Z);

        // ── Update HUD ────────────────────────────────────────────────────
        std::string dir = moving ? (mx < 0 ? "<-- left" : "right -->") : "idle";
        hud_->SetText("Player X: " + std::to_string((int)playerX_)
                      + "  [" + dir + "]  |  ESC = quit");
    }
};

int main() {
    return Simple3D::CreateGame<Game2DDemo>()->Run();
}

Step-by-step explanation

cam->SetOrthographic(true)

Passing true switches the camera from perspective projection to orthographic projection. In perspective mode the engine applies a depth-based scale factor to everything: objects far away appear smaller. In orthographic mode that scale factor is removed. Two objects at the same size in world units are drawn at exactly the same pixel size regardless of their distance from the camera.

This is the foundational step for any 2D game. Without it the 3D perspective effect makes the XY-plane sprite world look like 3D cards floating in space rather than a flat 2D scene.

To revert to 3D at any time simply call cam->SetOrthographic(false). You can toggle this at runtime to create a 2D menu that precedes a 3D game, for example.

cam->SetOrthographic(bool enable);

cam->SetOrthoSize(height)

SetOrthoSize controls the zoom level of the orthographic camera. The argument is the number of world units visible from the bottom edge to the top edge of the screen. The horizontal extent is derived automatically from the window's aspect ratio.

Think of it like a window frame of a fixed height that you hold up to your world. A small value zooms in; a large value shows more of the world. The relationship between pixels and world units is: 

// At 720p (1280×720) with OrthoSize = 12.0:
// pixels_per_unit_vertical = 720 / 12 = 60 px/unit
// pixels_per_unit_horizontal = 1280 / (12 * 16/9) = 60 px/unit  (same, aspect preserved)

For pixel-perfect rendering, choose an OrthoSize that divides your window resolution evenly. Alternatively, use cam->Setup2D(ppu) which computes the correct OrthoSize automatically from the pixels-per-unit value you provide.

cam->SetOrthoSize(float worldUnitsHeight);

entity->AddSprite2D(texturePath)

AddSprite2D creates a StaticSprite2D component and attaches it to the entity. Internally Urho3D's 2D rendering system handles z-sorting, blending, and UV mapping. The sprite is rendered as a textured quad in the XY plane at the entity's position and scale.

The texture file must be a standard image format (PNG, JPG, TGA) placed under your Data/ directory. PNG with transparency is recommended for sprites so that the background behind the character shows through.

Multiple entities can share the same texture without loading it twice — the engine caches loaded textures by path. Always use the same relative path string for the same texture to benefit from the cache.

entity->AddSprite2D(std::string texturePath);

entity->SetFlipX2D(bool)

SetFlipX2D(true) sets a flag on the underlying StaticSprite2D or AnimatedSprite2D component that mirrors the rendered quad horizontally. This works for both static sprites added with AddSprite2D and animated sprites added with AddAnimatedSprite2D.

The flip is purely visual — it does not affect the entity's world-space position, rotation, or physics collision shape. A flipped character's hitbox remains in the same place.

There is no need to call SetFlipX2D every frame. Set it once when the movement direction changes and the sprite stays flipped until you change it again.

entity->SetFlipX2D(bool flip);

entity->AddAnimatedSprite2D(scmlPath)

This call loads a Spriter Pro .scml file and attaches an AnimatedSprite2D component to the entity. A .scml file is an XML file that describes animation timelines using bone and sprite sheet references. Spriter exports the .scml and the referenced sprite sheet images together; all of those files must reside in the same directory under Data/.

A .scml file can contain multiple named animations (clips). The entity displays nothing until you start a clip with PlayAnimation2D().

Spriter Pro is commercial but has a free edition. The Urho3D 2D animation documentation also describes the .scml format in detail if you need to author compatible files manually.

entity->AddAnimatedSprite2D(std::string scmlPath);

entity->PlayAnimation2D(name, loop, speed)

Starts playing a named animation clip on the AnimatedSprite2D component that was added by AddAnimatedSprite2D. The clip name must match a name defined inside the .scml file exactly, including capitalisation.

Calling PlayAnimation2D while an animation is already playing switches to the new clip immediately, restarting from frame 0. If you want to switch without a hard cut, you will need to implement your own timing logic (or wait for the current clip to finish before switching).

The optional loop flag (default true) controls whether the clip loops. Set it to false for one-shot animations such as jumping, dying, or attacking. The optional speed multiplier (default 1.0f) scales the playback rate. A value of 2.0f plays at twice the authored speed; 0.5f plays in slow motion.

entity->PlayAnimation2D(std::string name, bool loop = true, float speed = 1.0f);

LoadTileMap(tmxPath)

LoadTileMap() is a Game-level method (call it as a member function in Start() or Update() just like CreateEntity()). It reads a Tiled .tmx file, creates an entity, attaches a TileMap2D component, and returns the entity pointer.

The .tmx file references a tileset image (typically a .tsx file or an embedded tileset). Both the .tmx and the tileset image must be placed under your Data/ directory, in paths that match what Tiled records in the .tmx.

Box2D static bodies are created automatically for all rectangles, polygons, and ellipses drawn on Tiled object layers — no additional code is required. If a Tiled object layer has the name "Collision" that layer is processed as collision data and may not be rendered as tiles, depending on your Tiled setup.

The returned Entity* can be repositioned or rescaled like any other entity. Scaling the tile map entity uniformly scales both the visual tiles and the generated collision shapes.

Entity* map = LoadTileMap(std::string tmxPath);

Key differences between 2D and 3D in simple-3d

If you are coming from the 3D tutorials (Parts 01–11), the table below summarises what changes and what stays the same in 2D mode:

Topic 3D mode 2D mode
Camera projection Perspective (default) Orthographic (SetOrthographic(true))
Camera position Anywhere in 3D space Z = −10, X/Y track the player
Movement axes X/Y/Z (Y = up) X/Y only (Z = depth order, not movement)
Visual component AddModel(path) AddSprite2D(path) or AddAnimatedSprite2D(path)
Physics engine Bullet (3D rigid body) Box2D (2D rigid body — see Part 13)
Physics calls AddRigidBody(mass) AddRigidBody2D(mass)
Velocity Vector3 (3 axes) Vector2 (X and Y only)
Level geometry 3D models with mesh colliders Tiled .tmx maps via LoadTileMap()
Ground check IsOnGround() — 3D ray against layer 2 IsOnGround2D() — 2D downward ray (Box2D)
Lighting Full PBR / directional / point lights Not normally used; sprites are rendered unlit or with ambient only
Entity creation CreateEntity(name) CreateEntity(name) (identical)
Input, audio, UI Identical API Identical API

Mixing 2D and 3D in the same scene

Because simple-3d uses the same scene graph for both, you can technically place 3D models and 2D sprites in the same scene. However, the camera can only be in one projection mode at a time. A common approach for games that need both (for example a 3D game with a 2D HUD element drawn in world space) is to use 3D for the world and the normal CreateLabel/CreateImage UI layer for all flat elements.

Camera following in 2D

Because the camera must stay fixed at Z = −10, you cannot use cam->Follow(player) directly (that method is designed for 3D third-person use). Instead, update the camera position manually in Update(): 

// In Update(dt):
// Follow the player on X; fix Y at 0 and Z at -10.
Vector3 pPos = player_->GetPosition();
cam_->SetPosition(pPos.x_, 0.0f, -10.0f);

For a platformer that scrolls both horizontally and vertically, follow both axes: 

// Full 2D camera follow — tracks both X and Y.
Vector3 pPos = player_->GetPosition();
cam_->SetPosition(pPos.x_, pPos.y_, -10.0f);

To add smoothing (so the camera glides toward the player instead of snapping), use linear interpolation: 

Vector3 camPos = cam_->GetPosition();
Vector3 pPos   = player_->GetPosition();
float lerpSpeed = 5.0f;   // higher = snappier; lower = more floaty
float newX = camPos.x_ + (pPos.x_ - camPos.x_) * lerpSpeed * dt;
float newY = camPos.y_ + (pPos.y_ - camPos.y_) * lerpSpeed * dt;
cam_->SetPosition(newX, newY, -10.0f);

Lua equivalent

All 2D rendering API calls are available from Lua with identical names. The example below reproduces the same scene as the C++ program above: an orthographic camera, a background sprite, a tile map, an animated player character, and camera following. 

-- Game2DDemo.lua
-- Run this as a game-level script or attach it via entity:AddScript().

local cam, player_, hud_
local wasMoving = false
local playerX   = 0

local MOVE_SPEED = 5.0
local ORTHO_SIZE = 12.0
local CAM_Z      = -10.0

function Start()
    SetWindowTitle("Part 12 — 2D Mode (Lua)")
    SetClearColor(Color(0.53, 0.81, 0.98))

    -- Orthographic camera
    cam = CreateCamera()
    cam:SetOrthographic(true)
    cam:SetOrthoSize(ORTHO_SIZE)
    cam:SetNearClip(0.1)
    cam:SetFarClip(100.0)
    cam:SetPosition(0, 0, CAM_Z)

    -- Background static sprite
    local bg = CreateEntity("Background")
    bg:SetPosition(0, 0, 1)   -- Z=1 puts it behind Z=0 entities
    bg:SetScale(24.0)
    bg:AddSprite2D("Sprites/background.png")

    -- Tile map
    local map = LoadTileMap("Maps/level1.tmx")
    map:SetPosition(0, -4, 0)

    -- Player with animated sprite
    player_ = CreateEntity("Player")
    player_:SetPosition(0, 0, 0)
    player_:AddAnimatedSprite2D("Sprites/hero.scml")
    player_:PlayAnimation2D("idle")

    -- A coin with a static sprite
    local coin = CreateEntity("Coin")
    coin:SetPosition(4, 1.5, 0)
    coin:AddSprite2D("Sprites/coin.png")

    -- HUD
    hud_ = CreateLabel("A/D or Arrows = move  |  ESC = quit")
    hud_:SetPosition(12, 12)
    hud_:SetFontSize(16)
    hud_:SetColor(Color.WHITE)
end

function Update(dt)
    if IsKeyPressed(Key.Escape) then Quit() end

    -- Horizontal input
    local mx = 0
    if IsKeyDown(Key.A) or IsKeyDown(Key.Left)  then mx = -1 end
    if IsKeyDown(Key.D) or IsKeyDown(Key.Right) then mx =  1 end

    -- Move player
    playerX = playerX + mx * MOVE_SPEED * dt
    player_:SetPosition(playerX, 0, 0)

    -- Flip sprite
    if mx < 0 then player_:SetFlipX2D(true)  end
    if mx > 0 then player_:SetFlipX2D(false) end

    -- Switch animation
    local moving = (mx ~= 0)
    if moving ~= wasMoving then
        if moving then
            player_:PlayAnimation2D("run")
        else
            player_:PlayAnimation2D("idle")
        end
        wasMoving = moving
    end

    -- Camera follow
    cam:SetPosition(playerX, 0, CAM_Z)

    -- HUD update
    local dir = moving and (mx < 0 and "<-- left" or "right -->") or "idle"
    hud_:SetText("Player X: " .. tostring(math.floor(playerX))
                  .. "  [" .. dir .. "]  |  ESC = quit")
end

Lua note

All 2D API methods (AddSprite2D, AddAnimatedSprite2D, PlayAnimation2D, SetFlipX2D, SetOrthographic, SetOrthoSize, LoadTileMap) are available in Lua with exactly the same names. Boolean values are the Lua literals true and false. Key constants use dot notation: Key.A, Key.Left, and so on.

In Lua ~= is the not-equal operator (C++ uses !=).

2D rendering API quick reference

Method Where called Description
cam->SetOrthographic(bool) Start() Enable/disable orthographic projection. Must be true for 2D.
cam->SetOrthoSize(float) Start() or Update() Visible height in world units. Smaller = zoomed in; larger = zoomed out.
cam->Setup2D(float ppu) Start() Shorthand: sets orthographic, OrthoSize from ppu, and camera Z position.
entity->AddSprite2D(texPath) Start() Attach a static StaticSprite2D textured quad to the entity.
entity->SetFlipX2D(bool) Update() Mirror the sprite horizontally. Use for left-facing movement.
entity->AddAnimatedSprite2D(scmlPath) Start() Load a Spriter .scml animation and attach an AnimatedSprite2D component.
entity->PlayAnimation2D(name, loop, speed) Start() or Update() Start a named clip. Restarts from frame 0. loop defaults to true.
LoadTileMap(tmxPath) Start() Load a Tiled .tmx map, render tiles, auto-create Box2D static colliders. Returns Entity*.

Common mistakes

Common mistakes in 2D mode

  • Forgetting to call SetOrthographic(true). Without it the camera uses perspective projection. Sprites still appear but objects further from the camera look smaller, breaking the flat 2D look. Always call this before the first frame.
  • Setting the camera Z to 0 or a positive value. The camera must be behind the XY plane (negative Z) to see entities at Z = 0. Z = 0 places the camera on the same plane as the sprites; you will see nothing. Use Z = −10 as the standard value.
  • Setting entity Z to a non-zero value by accident. In 2D all game entities must sit at Z = 0 (or very close to it). Setting an entity to Z = 100 moves it far behind the scene. Check your SetPosition calls and make sure the third component is always 0.0f for gameplay entities.
  • Calling PlayAnimation2D every frame instead of only on state change. PlayAnimation2D restarts the clip from frame 0 each time it is called. Calling it in Update() without a state-change guard causes the animation to restart at frame 0 every frame, resulting in a frozen first frame. Guard the call with a boolean flag that tracks whether the state changed, as shown in the examples.
  • Mixing 3D physics (AddRigidBody) with 2D physics (AddRigidBody2D) on the same entity. These attach components from two different physics worlds (Bullet 3D and Box2D 2D). Having both on the same entity produces undefined behaviour. For 2D games only use the AddRigidBody2D, AddBoxCollider2D, and AddCircleCollider2D family of calls. 2D physics is the subject of Part 13.
  • Placing the .scml file without its referenced textures. A Spriter .scml file contains relative paths to the sprite sheet images it needs. If those images are missing or in a different directory, the animated sprite loads but renders as a white or invisible quad. Always export the .scml and all referenced PNG files together and keep them at the same relative paths under Data/.
  • Using the wrong clip name in PlayAnimation2D. The clip name must match the name defined in the .scml file exactly. Open the .scml in a text editor and look for <animation name="..."> tags to verify the correct names. A mismatched name produces no visual change (the previous clip continues playing) and no error message by default.
  • Updating the camera X and Y but hard-coding the wrong Z. If you copy the player's full Vector3 position to the camera without overriding Z, the camera slides toward Z = 0 and clips through all the sprites. Always set the camera Z to −10 explicitly when following a 2D entity.

What you learned

This part introduced the full 2D rendering pipeline in simple-3d. You now know how to build a complete 2D scene from scratch:

  • simple-3d's 2D mode is not a separate engine — it is the same scene graph used in 3D, but with orthographic projection enabled and entities living in the XY plane at Z = 0.
  • cam->SetOrthographic(true) disables perspective foreshortening. Without it the scene looks 3D even though sprites are flat.
  • cam->SetOrthoSize(height) controls zoom by setting the number of world units visible vertically. Use cam->Setup2D(ppu) for pixel-art games where you want to think in pixels.
  • The 2D camera always sits at Z = −10 and must be updated manually in Update() to follow the player. Copy the player's X and Y position but always override Z to −10.
  • entity->AddSprite2D(path) attaches a static textured quad to an entity. The texture is centred on the entity's world position.
  • entity->SetFlipX2D(bool) mirrors the sprite horizontally. Call it once when the movement direction changes, not every frame.
  • entity->AddAnimatedSprite2D(scmlPath) loads a Spriter animation file. Follow with PlayAnimation2D(name) to start a clip.
  • PlayAnimation2D restarts the named clip from frame 0. Only call it when the animation state changes to avoid flickering.
  • LoadTileMap(tmxPath) reads a Tiled .tmx file, renders tile layers, and automatically creates Box2D static colliders from Tiled object layers. No additional physics setup is needed.
  • 2D and 3D physics are separate: 2D uses Box2D (Part 13) and AddRigidBody2D; 3D uses Bullet and AddRigidBody. Never mix them on the same entity.
