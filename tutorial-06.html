Why Lua alongside C++?

simple-3d embeds LuaJIT (via U3D/Urho3D) so that you can drive per-entity runtime behaviour without recompiling C++. The intended division of labour is:

  • C++ (Start / Update / Stop on the Game subclass) — scene creation, entity construction, component wiring, action binding, and any logic that must be fast or type-safe.
  • Lua scripts (Start / Update / Stop per entity) — per-entity runtime behaviour: movement, animation, AI decisions, sound triggers. Scripts can be edited and hot-reloaded without rebuilding the binary.

Each script file is a plain Lua 5.1 / LuaJIT-compatible text file. The engine gives it its own execution environment and calls the global functions Start, Update, and Stop automatically, exactly like the C++ lifecycle callbacks you have already seen on the Game class.

LuaJIT vs standard Lua

simple-3d uses LuaJIT which is binary-compatible with Lua 5.1. All standard Lua 5.1 libraries (math, string, table, io, etc.) are available inside scripts. LuaJIT's JIT compiler makes numeric-heavy Update loops run at near-native speed with no extra effort.

Where scripts live: Data/Scripts/

All Lua scripts must reside somewhere inside the engine's resource search path. The canonical location, consistent with the project structure introduced in the Getting Started guide, is: 

my-game/
└── Data/
    └── Scripts/          <-- all .lua files go here
        ├── crystal.lua
        └── enemy.lua

When you call entity->AddScript("Scripts/crystal.lua") in C++, the engine resolves this path relative to the Data/ root, just like model or texture paths. The full path on disk becomes Data/Scripts/crystal.lua next to the executable.

You do not include the Data/ prefix in the path string — the engine searches inside all registered resource directories automatically.

Complete working example

The demo below creates a spinning, keyboard-controlled crystal object. All movement and rotation logic lives entirely in a Lua script; C++ only constructs the scene and wires things together. This is the recommended pattern.

C++ — main.cpp

#include <Simple3D/Simple3D.h>
using namespace Simple3D;

class LuaDemo : public Game {
    Entity* crystal_ = nullptr;

public:
    void Start() override {
        SetWindowTitle("Part 06 — Lua Scripting Basics");
        SetClearColor(Color(0.05f, 0.05f, 0.12f));

        // ── Ground plane (static physics body) ──────────────────────
        Entity* ground = CreateEntity("Ground");
        ground->AddModel("Models/Plane.mdl");
        ground->SetScale(12.0f);
        ground->AddRigidBody(0.0f);                        // 0 = static
        ground->AddBoxCollider(Vector3(12.0f, 0.2f, 12.0f));
        ground->SetCollisionLayer(CollisionLayer::StaticGeometry);

        // ── Crystal entity: model + Lua script ──────────────────────
        crystal_ = CreateEntity("Crystal");
        crystal_->AddModel("Models/Box.mdl");
        crystal_->SetPosition(0.0f, 1.5f, 0.0f);

        // NEW: attach a Lua script to this entity.
        // Path is relative to Data/ inside the resource search path.
        crystal_->AddScript("Scripts/crystal.lua");

        // ── Lighting ─────────────────────────────────────────────────
        Entity* sun = CreateEntity("Sun");
        sun->AddDirectionalLight(Color(1.0f, 0.9f, 0.8f), 1.2f);
        sun->SetRotation(Quaternion(45.0f, -30.0f, 0.0f));

        Entity* fill = CreateEntity("Fill");
        fill->AddDirectionalLight(Color(0.2f, 0.3f, 0.6f), 0.5f);
        fill->SetRotation(Quaternion(-30.0f, 60.0f, 0.0f));

        // ── Camera following the crystal ──────────────────────────────
        Camera* cam = CreateCamera();
        cam->Follow(crystal_, 8.0f, 3.0f);
        cam->LookAt(crystal_);
        cam->SetSmoothFollow(true, 4.0f);
    }

    void Update(float dt) override {
        // C++ Update only handles global quit; per-entity logic is in Lua.
        if (IsKeyDown(Key::Escape)) Quit();
    }
};

int main() {
    return CreateGame<LuaDemo>()->Run();
}

Lua — Data/Scripts/crystal.lua

-- Data/Scripts/crystal.lua
-- Demonstrates: lifecycle callbacks, self.node alias,
--               Vector3, Quaternion, SetRotation, MoveRelative.

local entity           -- module-level reference to self.node (set in Start)
local spinAngle = 0    -- accumulated Y-axis rotation in degrees
local spinSpeed = 90   -- degrees per second
local moveSpeed = 4    -- world units per second (WASD movement)

-- ── Start(): called once when the script is loaded ───────────────────
function Start()
    -- self.node is the Urho3D Node that owns this script.
    -- Copying it to a module-level variable avoids repeating "self.node"
    -- in every callback and keeps the code readable.
    entity = self.node

    -- GetName() is a standard Urho3D Node method exposed to Lua.
    print("[crystal.lua] Start() on entity: " .. entity:GetName())
end

-- ── Update(dt): called every frame; dt = seconds since last frame ────
function Update(dt)
    -- 1. SPIN: accumulate the angle, then set rotation from Euler angles.
    --    Quaternion(yaw, pitch, roll) — first argument is yaw (Y-axis).
    spinAngle = spinAngle + spinSpeed * dt
    entity:SetRotation(Quaternion(spinAngle, 0, 0))

    -- 2. MOVE: build a direction Vector3 from WASD keys, then translate.
    local dir = Vector3(0, 0, 0)

    if Input:IsKeyDown(KEY_W) then dir.z = dir.z + 1 end
    if Input:IsKeyDown(KEY_S) then dir.z = dir.z - 1 end
    if Input:IsKeyDown(KEY_A) then dir.x = dir.x - 1 end
    if Input:IsKeyDown(KEY_D) then dir.x = dir.x + 1 end

    -- Normalise so diagonal movement is not faster than cardinal movement.
    if dir:Length() > 0 then
        -- MoveRelative adds the delta to the entity's world-space position.
        entity:MoveRelative(dir:Normalized() * moveSpeed * dt)
    end

    -- 3. SPEED BOOST: hold Shift to spin twice as fast.
    if Input:IsKeyDown(KEY_LSHIFT) then
        spinAngle = spinAngle + spinSpeed * dt   -- extra spin this frame
    end
end

-- ── Stop(): called when the entity is removed or the scene closes ────
function Stop()
    print("[crystal.lua] Stop() — entity is being removed.")
end

Step-by-step explanation

1. entity->AddScript(path)

This single C++ call is the only bridge between your C++ scene setup and the Lua world: 

crystal_->AddScript("Scripts/crystal.lua");

The engine loads the file from Data/Scripts/crystal.lua, compiles it with LuaJIT, and registers the script's lifecycle callbacks. From that point on, every game frame the engine calls Update(dt) in the script — with no further involvement from C++ code.

You can call AddScript on any entity at any time — but the most common pattern is to do it in Game::Start() immediately after constructing the entity, before the first frame runs. A single entity may only hold one script at a time.

2. The Data/Scripts/ directory convention

The path string you pass to AddScript is resolved inside the engine's resource search path, the same way model and texture paths work. The canonical home for scripts is: 

Data/
└── Scripts/
    └── crystal.lua     <-- engine resolves "Scripts/crystal.lua" to here

Never put scripts outside Data/ (or another registered resource directory). The engine will log Failed to load Script and silently skip the file if it cannot be found.

3. Lua lifecycle callbacks

The engine checks for three global functions in every loaded script and calls them automatically. You do not need to register them anywhere — the names are the contract:

Callback Signature Called when
Start() function Start() … end Once, immediately after the script is loaded. Use this to obtain component proxies and save the self.node reference.
Update(dt) function Update(dt) … end Every frame. dt is the scaled delta time in seconds (same value the C++ Update(float dt) receives).
Stop() function Stop() … end Once, when the entity is removed from the scene or the scene is unloaded. Good for cleanup and final logging.

All three are optional. A script that only defines Update is perfectly valid, as is a script that only defines Start. The engine checks for each function's existence before calling it.

4. self.node — accessing the owning entity

Every Lua script runs in an environment where self is a table representing the ScriptInstance component. The underlying Urho3D Node (the entity) is accessible as self.node. Through this reference you can call any Urho3D Node method, including all the simple-3d extensions (SetRotation, MoveRelative, GetCharacterController, and so on). 

function Start()
    -- Access the owning node through self.node
    local name = self.node:GetName()
    print("My entity name: " .. name)
end

5. local entity = self.node — the alias convention

Typing self.node in every method call inside Update is verbose and slightly slower than a local variable lookup. The simple-3d convention is to copy the reference into a module-level (file-scope) local in Start(): 

local entity   -- declared at the top of the file (module scope)

function Start()
    entity = self.node   -- assign once; available in Update and Stop
end

function Update(dt)
    entity:SetRotation(Quaternion(0, 0, 0))  -- clean, no "self.node"
end

You must declare the variable at file scope (local entity at the top) so that both Start and Update can see it. Declaring it inside Start with local would limit it to that function's scope and make it invisible to Update.

Module-level vs local scope in Lua

In Lua, a local variable declared at file scope (outside any function) is accessible to all functions defined in the same file. This is sometimes called "upvalue" or "module-level" scope. It is the correct place for state that persists across frames, such as entity, accumulated angles, timers, and component proxies.

6. Vector3(x, y, z) in Lua

Vector3 is exposed to Lua as a global constructor. You create instances the same way you would in C++: 

local zero  = Vector3(0, 0, 0)
local up    = Vector3(0, 1, 0)
local speed = Vector3(0, 0, 5) * dt   -- scalar multiply supported

You can read and write individual components with .x, .y, and .z, and call methods directly on the value: 

local dir = Vector3(1, 0, 1)
local len  = dir:Length()           -- float ~= 1.414
local norm = dir:Normalized()       -- unit vector

dir.x = dir.x + 0.5               -- component assignment

Arithmetic operators (+, -, *) work element-wise and with scalars, just like the C++ type. Subtraction of two entity positions gives a direction vector: 

-- Direction from this entity toward another entity
local target = FindEntity("Enemy")
if target then
    local diff = target:GetWorldPosition() - entity:GetWorldPosition()
    local dir  = diff:Normalized()
    entity:MoveRelative(dir * 3.0 * dt)
end

7. Quaternion(yaw, pitch, roll) in Lua

Quaternion is also exposed as a global constructor. The three arguments are Euler angles in degrees:

ArgumentAxisExample use
yawY (up)Character facing direction, compass heading
pitchX (right)Looking up/down, nose-up angle
rollZ (forward)Banking, tilting sideways
-- Spin around Y axis only (common for upright objects)
entity:SetRotation(Quaternion(spinAngle, 0, 0))

-- Tilt 30 degrees forward (pitch only)
entity:SetRotation(Quaternion(0, 30, 0))

-- Combined: face south-east at 45 degrees while tilted slightly
entity:SetRotation(Quaternion(135, 15, 0))

Quaternions do not suffer from gimbal lock when composed, but constructing them from Euler angles (as above) is the most readable approach for simple rotations. For more complex orientations (slerping between two rotations, for example), use the full Quaternion API exposed by Urho3D.

8. entity:SetRotation(Quaternion)

Sets the entity's world-space rotation to the given quaternion. Call it every frame with an updated angle to create continuous rotation: 

local angle = 0

function Update(dt)
    angle = angle + 90 * dt              -- 90 degrees per second
    entity:SetRotation(Quaternion(angle, 0, 0))
end

Notice that we accumulate angle outside the function (module scope) so that its value persists between frames. A variable declared inside Update would reset to 0 on every call and produce no visible rotation.

SetRotation is an absolute set, not an increment. To add a small rotation delta on top of the current rotation, read entity:GetRotation() first and compose the quaternions, or simply track the accumulated angle yourself as shown above.

9. entity:MoveRelative(Vector3)

Adds the given offset to the entity's current world-space position. It is equivalent to entity:SetPosition(entity:GetWorldPosition() + delta), but more concise. 

-- Move 5 units per second in the +Z direction
entity:MoveRelative(Vector3(0, 0, 1) * 5 * dt)

-- Move in a player-controlled direction
if dir:Length() > 0 then
    entity:MoveRelative(dir:Normalized() * moveSpeed * dt)
end

Always multiply by dt when moving an entity so that the speed stays consistent regardless of the frame rate. Forgetting dt means the object will move 60× faster on a 60 fps machine than on a 1 fps machine.

The name "MoveRelative" reflects that the movement is added relative to the entity's current position in world space. It does not move along the entity's local forward axis — use TranslateLocal for that.

10. Low-level keyboard input in Lua: Input:IsKeyDown(KEY_*)

The global Input object exposes Urho3D's input system to Lua. The method Input:IsKeyDown(KEY_W) returns true as long as the W key is held: 

if Input:IsKeyDown(KEY_W) then
    entity:MoveRelative(Vector3(0, 0, 1) * speed * dt)
end

Key constants follow the pattern KEY_ + the key name in capitals: KEY_W, KEY_SPACE, KEY_LSHIFT, KEY_ESCAPE, etc. These are integer constants defined by Urho3D and available as globals in every script.

For first-frame-only detection (key was just pressed this frame), use Input:IsKeyPress(KEY_SPACE) instead of IsKeyDown: 

if Input:IsKeyPress(KEY_SPACE) then
    -- fires only on the first frame Space is pressed
    PlaySound("Sounds/jump.wav")
end

Input Actions are preferred for game controls

Raw KEY_* checks are useful for quick prototyping, but for a real game you should bind named Input Actions in C++ (BindAction, BindAxis2D) and then query them in Lua with IsActionDown("Jump") and GetAxis2D("Move"). Named actions work with keyboard and gamepad without any extra code, and support player remapping. Part 16 covers Input Actions in detail.

Extended example: multiple scripts, shared world state

A more realistic scene has several entities, each with its own script. The example below adds a second entity — a slowly orbiting satellite — to the C++ setup, with its own dedicated Lua script. This shows how multiple scripts run in parallel, each with private module-level state.

C++ addition in Start()

// Add after the crystal_ setup in Start()
Entity* satellite = CreateEntity("Satellite");
satellite->AddModel("Models/Sphere.mdl");
satellite->SetScale(0.4f);
satellite->SetPosition(3.0f, 1.5f, 0.0f);
satellite->AddScript("Scripts/satellite.lua");

Lua — Data/Scripts/satellite.lua

-- Data/Scripts/satellite.lua
-- Orbits around world origin at a fixed radius and height.

local entity
local orbitAngle = 0     -- current angle around Y axis (degrees)
local orbitRadius = 3.0  -- orbit radius in world units
local orbitSpeed  = 45   -- degrees per second
local orbitHeight = 2.5  -- Y position (constant)

function Start()
    entity = self.node
    print("[satellite.lua] Start() on: " .. entity:GetName())
end

function Update(dt)
    orbitAngle = orbitAngle + orbitSpeed * dt

    -- Convert polar angle to Cartesian XZ coordinates
    local rad = math.rad(orbitAngle)   -- degrees to radians for math.cos/sin
    local x   = math.cos(rad) * orbitRadius
    local z   = math.sin(rad) * orbitRadius

    -- SetPosition sets the absolute world position each frame.
    entity:SetPosition(Vector3(x, orbitHeight, z))

    -- Also spin the satellite on its own axis.
    entity:SetRotation(Quaternion(orbitAngle * 2, 0, 0))
end

function Stop()
    print("[satellite.lua] Stop().")
end

Each script has its own orbitAngle, completely separate from crystal.lua's spinAngle. Lua module-level locals are per-script-instance, so there is no risk of one script's state leaking into another.

SetRotation vs MoveRelative: a comparison

Method What it does Typical use
entity:SetRotation(Quaternion) Replaces the entity's rotation with the given quaternion. Position is unchanged. Spinning objects, turret facing direction, character yaw.
entity:MoveRelative(Vector3) Adds the delta vector to the entity's current world position. Rotation is unchanged. Keyboard-driven movement, frame-by-frame translation.
entity:SetPosition(Vector3) Sets the absolute world position. Overwrites any previous translation. Scripted paths, orbit calculations, teleportation.

Use MoveRelative when you want to accumulate movement over multiple frames (player walking). Use SetPosition when you know the exact desired world coordinate on every frame (orbital path, scripted cutscene).

Checking syntax with luac -p

Lua syntax errors do not produce compiler errors — they appear as runtime log messages when the engine loads the file. The fastest way to catch them before running the game is the luac command-line tool: 

luac -p Data/Scripts/crystal.lua

The -p flag ("parse only") compiles the script without writing a bytecode file. If the file is syntactically correct, the command exits silently with code 0. If there is an error, it prints the line number and description: 

# Example error output (missing 'end' on a function):
luac: Data/Scripts/crystal.lua:22: 'end' expected (to close 'function' at line 14) near <eof>

Run luac -p on every script before launching the game during development. It is also easy to add to a Makefile or CMake custom command so that syntax is validated at build time: 

# Validate all scripts before building
find Data/Scripts -name '*.lua' -exec luac -p {} \;

Install luac

On Debian/Ubuntu: sudo apt install lua5.1 (installs luac for Lua 5.1, compatible with LuaJIT). On macOS with Homebrew: brew install lua@5.1. Alternatively, luajit -bl can be used to check LuaJIT-specific bytecode.

Common mistakes

Declaring entity with local inside Start().
If you write local entity = self.node inside Start(), the variable is local to that function and invisible in Update(). Always declare it at the top of the file: local entity, then assign entity = self.node (without local) inside Start().

Forgetting dt in movement.
entity:MoveRelative(Vector3(0, 0, 1) * 5) moves 5 units per frame, not per second. On a 60 fps display that is 300 units/second. Always multiply by dt: Vector3(0, 0, 1) * 5 * dt.

Wrong script path.
If AddScript("Scripts/foo.lua") silently does nothing, the file was not found. Check that: (a) the file is at Data/Scripts/foo.lua next to the executable, (b) the path in the call does not start with Data/, (c) the filename capitalisation matches exactly (Linux filesystems are case-sensitive). The log will contain Failed to load Script if the path is wrong.

Syntax errors silent until runtime.
A missing end or a typo in a keyword will not fail at compile time. The error only appears in the log when the game loads the script. Run luac -p yourscript.lua before each test session to catch these early.

Calling entity methods before Start() assigns entity.
Never call entity:SetRotation() at file scope (outside a function). The module runs before Start() is called, so entity is still nil at that point and the call will crash with a nil-dereference error.

Using KEY_w (lowercase).
Key constants are all-caps: KEY_W, KEY_SPACE, KEY_LSHIFT. Lowercase KEY_w is nil in Lua and IsKeyDown(nil) always returns false, so the key will appear to never be pressed.

Multiplying a nil vector.
dir:Normalized() * speed * dt crashes if dir:Length() is exactly 0 (the zero vector has no defined normalisation). Always guard with if dir:Length() > 0 then before normalising.

What you learned

  • entity->AddScript("Scripts/foo.lua") in C++ attaches a Lua script to any entity. The path is relative to Data/ in the resource search path.
  • The engine automatically calls Start() once when the script loads, Update(dt) every frame, and Stop() when the entity is removed. All three are optional.
  • Inside every script, self.node is the Urho3D Node that owns the script. The convention is to copy it into a module-level entity variable inside Start() for cleaner syntax everywhere else.
  • Vector3(x, y, z) and Quaternion(yaw, pitch, roll) are global constructors available in every script. Arithmetic operators and methods like :Length() and :Normalized() work as expected.
  • entity:SetRotation(Quaternion(angle, 0, 0)) sets the entity's world-space rotation from an accumulated Euler angle. Call it every frame to create continuous rotation.
  • entity:MoveRelative(Vector3(dx, 0, dz) * speed * dt) adds a position delta every frame. Always multiply by dt to make movement frame-rate independent.
  • Input:IsKeyDown(KEY_W) (held) and Input:IsKeyPress(KEY_SPACE) (first frame) give raw keyboard access inside Lua. Named Input Actions (Part 16) are preferred for production games.
  • luac -p Data/Scripts/foo.lua validates syntax without executing the script. Running it before each test session catches missing end keywords and other common errors immediately.
  • Multiple entities can each carry their own script. Each script file has private module-level state — variables do not leak between script instances.
