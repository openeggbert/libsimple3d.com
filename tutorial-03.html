Part 03 of 30Keyboard Input

New in this part

  • IsKeyPressed(Key) — returns true only on the first frame a key is pressed; contrasts with IsKeyDown which fires every frame while held
  • Full Key enum coverage: Key::W, Key::A, Key::S, Key::D, Key::Space, Key::Escape, Key::Up, Key::Down, Key::Left, Key::Right
  • Frame-rate independent movement via entity->TranslateWorld(direction * speed * dt)
  • Diagonal movement normalization with Vector3::Normalized() to prevent the speed-on-diagonals bug
  • Live HUD position display — calling label->SetText() every frame with a formatted Vector3

What you will learn

  • The difference between IsKeyDown (fires every frame the key is held) and IsKeyPressed (fires exactly once per physical keypress)
  • Which method to choose for movement, jumping, toggling, and quitting
  • How to poll WASD and arrow-key alternatives from the same Update() block
  • Why you must multiply movement by dt to get frame-rate independence
  • How to normalize a movement vector to prevent diagonal movement from being faster than axis-aligned movement
  • How to read an entity's world-space position and display it live in a HUD Label
  • Using std::snprintf inside Update() to format floating-point values into a readable string

Complete example

The program below creates a movable box on a ground plane. WASD and the arrow keys move the box. Space resets it to the origin. Escape quits. A HUD label in the top-left corner shows the box's live world-space position, updated every frame. 

#include <Simple3D/Simple3D.h>
#include <cstdio>              // std::snprintf
using namespace Simple3D;

class KeyInputDemo : public Game {
    Entity* player_   = nullptr;
    Label*  posLabel_ = nullptr;

    // Movement speed in world units per second
    static constexpr float speed_ = 5.0f;

public:
    void Start() override {
        SetWindowTitle("Part 03 — Keyboard Input");
        SetClearColor(Color(0.08f, 0.1f, 0.14f));

        // ── Ground plane ──────────────────────────────────────────
        auto* ground = CreateEntity("Ground");
        ground->AddModel("Models/Plane.mdl");
        ground->SetPosition(0.0f, -0.5f, 0.0f);
        ground->SetScale(20.0f);

        // ── Movable player box ────────────────────────────────────
        player_ = CreateEntity("Player");
        player_->AddModel("Models/Box.mdl");
        player_->SetPosition(0.0f, 0.0f, 0.0f);

        // ── Lighting ──────────────────────────────────────────────
        auto* sun = CreateEntity("Sun");
        sun->AddDirectionalLight(Color(1.0f, 0.95f, 0.85f), 1.2f);
        sun->SetRotation(Quaternion(50.0f, -30.0f, 0.0f));

        SetAmbientColor(Color(0.25f, 0.28f, 0.35f));

        // ── Camera: follow player from above and behind ────────────
        auto* cam = CreateCamera();
        cam->Follow(player_, 8.0f, 4.0f);
        cam->LookAt(player_);

        // ── HUD: live position display (top-left) ─────────────────
        posLabel_ = CreateLabel("Position: (0.00, 0.00, 0.00)");
        posLabel_->SetPosition(20, 20);
        posLabel_->SetFontSize(16);
        posLabel_->SetColor(Color(0.85f, 0.95f, 0.85f));

        // ── HUD: controls hint ────────────────────────────────────
        Label* hint = CreateLabel("WASD / Arrows: Move  |  Space: Reset  |  ESC: Quit");
        hint->SetPosition(20, 48);
        hint->SetFontSize(13);
        hint->SetColor(Color(0.45f, 0.6f, 0.45f));
    }

    void Update(float dt) override {
        // ── Continuous movement (IsKeyDown: true every frame the key is held) ──
        Vector3 move(0.0f, 0.0f, 0.0f);

        if (IsKeyDown(Key::W) || IsKeyDown(Key::Up))    move.z_ += 1.0f;
        if (IsKeyDown(Key::S) || IsKeyDown(Key::Down))  move.z_ -= 1.0f;
        if (IsKeyDown(Key::A) || IsKeyDown(Key::Left))  move.x_ -= 1.0f;
        if (IsKeyDown(Key::D) || IsKeyDown(Key::Right)) move.x_ += 1.0f;

        // Normalise first (prevents diagonal 41% speed boost), then scale by speed and dt
        if (move.Length() > 0.0f)
            player_->TranslateWorld(move.Normalized() * speed_ * dt);

        // ── One-shot actions (IsKeyPressed: true only on the first frame) ─────
        if (IsKeyPressed(Key::Space))
            player_->SetPosition(0.0f, 0.0f, 0.0f);   // teleport to origin

        if (IsKeyPressed(Key::Escape))
            Quit();

        // ── Live HUD: refresh position label every frame ──────────────────────
        Vector3 pos = player_->GetPosition();
        char buf[80];
        std::snprintf(buf, sizeof(buf),
                      "Position: (%.2f, %.2f, %.2f)",
                      pos.x_, pos.y_, pos.z_);
        posLabel_->SetText(buf);
    }
};

int main() {
    return CreateGame<KeyInputDemo>()->Run();
}

How it works

IsKeyDown vs IsKeyPressed

simple-3d provides two keyboard polling methods. They look similar but have a critical difference in how often they return true.

Method While key is held Frame the key is released Best used for
IsKeyDown(Key) true every frame false Continuous actions — movement, acceleration, holding a button
IsKeyPressed(Key) true frame 1 only, then false false One-shot events — jump, fire, toggle, reset, quit

IsKeyDown simply asks "is the key physically down right now?" It returns true on every call while the player keeps the key held. This makes it perfect for movement: every frame we add a small offset proportional to dt, producing smooth, continuous motion.

IsKeyPressed is a rising-edge detector. It tracks the key's previous state and only fires on the transition from up to down. No matter how long the player holds the key, IsKeyPressed returns true for exactly one frame. This makes it ideal for actions that should happen once per intentional keypress — teleporting to the origin, opening a menu, firing a bullet, or quitting the application.

Continuous movement — use IsKeyDown

// Fires every frame the key is held.
// The player moves smoothly for as
// long as the key stays pressed.
if (IsKeyDown(Key::W))
    player_->TranslateWorld(
        Vector3(0, 0, speed_ * dt));

One-shot reset — use IsKeyPressed

// Fires only on the FIRST frame of
// the keypress, regardless of how
// long the player holds Space.
if (IsKeyPressed(Key::Space))
    player_->SetPosition(
        0, 0, 0);

Common mistake: using IsKeyPressed for movement

If you write if (IsKeyPressed(Key::W)) player_->TranslateWorld(...);, the box will jump exactly one step forward, then stop dead — even if the player keeps W held. The move triggers only on the first frame of the press. Always use IsKeyDown for anything that should repeat while the key is held.

The Key enum

All keyboard constants live in the Key enum. Include <Simple3D/Simple3D.h> to access them. This tutorial uses the most common movement and control keys:

Constant Physical key Typical use
Key::W W Move forward
Key::A A Move left (strafe)
Key::S S Move backward
Key::D D Move right (strafe)
Key::Up ↑ Arrow-key alternative for forward
Key::Down ↓ Arrow-key alternative for backward
Key::Left ← Arrow-key alternative for left
Key::Right → Arrow-key alternative for right
Key::Space Space Jump, confirm, reset
Key::Escape Escape Quit, cancel, open/close menu

The full enum also covers letters Key::A–Key::Z, digits Key::Num0–Key::Num9, function keys Key::F1–Key::F12, modifier keys (Key::LShift, Key::LCtrl, Key::LAlt, etc.), and the numpad. See the C++ API reference for the complete list.

Binding the same action to two different keys — WASD and arrow keys — is done with a simple logical OR. Both alternatives are polled in the same if statement so neither overrides the other: 

// Either W or the up-arrow moves forward — whichever the player prefers.
if (IsKeyDown(Key::W) || IsKeyDown(Key::Up))
    move.z_ += 1.0f;

Prefer Input Actions for real games

For a shipping game, consider using the higher-level Input Actions system (covered in Part 16) instead of raw key polling. BindAction("Move") lets you wire both keyboard keys and a gamepad button to the same named action, and players can remap controls without touching your code. The raw IsKeyDown / IsKeyPressed API shown here is the right foundation to understand before tackling that system.

Frame-rate independent movement with dt

The parameter dt passed to Update(float dt) is the delta time: the number of seconds that elapsed since the previous frame. On a display running at 60 Hz, dt is approximately 0.0167. On a 120 Hz display it is roughly 0.0083 — half as large.

If you move the player by a fixed amount per frame, the game runs twice as fast on the faster display. Multiplying by dt converts the movement into units per second so the result is identical regardless of frame rate: 

// BAD: frame-rate dependent
// 60 fps  → 30 units/sec
// 120 fps → 60 units/sec ← 2x faster!
player_->TranslateWorld(
    Vector3(0, 0, 0.5f));  // per frame
// GOOD: frame-rate independent
// 60 fps  → 5 units/sec
// 120 fps → 5 units/sec ← same!
player_->TranslateWorld(
    Vector3(0, 0, 1) * speed_ * dt);

The idiom is always: direction × speed × dt. speed_ is in world-units per second (e.g. 5.0f means five metres per second). Multiplying by dt converts that to the correct per-frame distance at whatever frame rate the player's hardware achieves.

dt also accounts for pauses and slow-motion

Because dt is the actual elapsed time — not a constant — it also correctly handles frames that take longer than usual (e.g. a stutter), and it honours SetTimeScale() automatically. At SetTimeScale(0.5f), dt is halved every frame, slowing all dt-based motion to half speed with no extra work from you.

Normalizing diagonal input

When the player holds two direction keys at once — say W and D — the raw move vector accumulates contributions from both axes: 

// W pressed: move += (0, 0, +1)  →  move = (0, 0, 1)
// D pressed: move += (+1, 0, 0)  →  move = (1, 0, 1)
// Length of (1, 0, 1) = sqrt(1² + 0² + 1²) = sqrt(2) ≈ 1.414

Without normalization the diagonal distance travelled per second is 1.414 × speed_ ≈ 7.07 instead of the intended 5.0 — a 41 % speed boost just for moving diagonally. Players notice this immediately in competitive games and often exploit it.

Calling Normalized() scales the vector to unit length before multiplying by speed and dt: 

// After accumulating W + D, move = (1, 0, 1), length ≈ 1.414
Vector3 dir = move.Normalized();   // dir = (0.707, 0, 0.707), length = 1.0
player_->TranslateWorld(dir * speed_ * dt);  // always travels at speed_ u/s

The guard if (move.Length() > 0.0f) prevents calling Normalized() on the zero vector, which would produce a NaN. If no keys are pressed, move is (0, 0, 0) and we skip the translation entirely. 

if (move.Length() > 0.0f)
    player_->TranslateWorld(move.Normalized() * speed_ * dt);

Live HUD label showing entity position

A Label created with CreateLabel() renders text on the screen at a fixed pixel position. Its content is static until you call SetText(). By calling SetText() inside Update(), the label refreshes every frame, turning it into a live readout of any game value.

This tutorial reads the player entity's current world-space position with GetPosition(), which returns a Vector3 whose x_, y_, and z_ members hold the three coordinates. We then format them to two decimal places using std::snprintf: 

// 1. Read the entity's current world-space position each frame.
Vector3 pos = player_->GetPosition();

// 2. Format the three components into a fixed-width string.
//    "%.2f" prints the float with exactly two decimal places.
char buf[80];
std::snprintf(buf, sizeof(buf),
              "Position: (%.2f, %.2f, %.2f)",
              pos.x_, pos.y_, pos.z_);

// 3. Push the new string to the label.  The old text is replaced immediately.
posLabel_->SetText(buf);

std::snprintf writes at most sizeof(buf) bytes, including the null terminator, making it safe even if the numbers grow large. The string is then passed directly to SetText(), which accepts a const char* as well as a std::string.

When to use a live HUD label

Position readouts are invaluable during development — they let you verify that your movement speed is correct, check that physics is behaving as expected, and debug teleport/spawn logic. In a shipped game you would typically hide or remove the raw position label and replace it with styled UI elements that represent meaningful game state (health, score, minimap coordinates, etc.).

Lua equivalent

Lua note

In Lua scripts attached to an entity via entity->AddScript("..."), the Input global provides keyboard polling. The method names are Input:IsKeyDown("KeyName") and Input:IsKeyPress("KeyName") (note: IsKeyPress, not IsKeyPressed). Key names are plain strings matching the enum value without the Key:: prefix. Vector3 component access in Lua uses .x, .y, .z (no trailing underscores). To update a HUD label from a Lua entity script, store a reference to it in a Lua global from C++ before calling AddScript.

The script below is a direct Lua translation of the C++ example. Save it as Data/Scripts/keyboard_input.lua and attach it with player_->AddScript("Scripts/keyboard_input.lua"); in Start(). The posLabel global must be set from C++ before the script runs (see the note inside the code). 

-- Scripts/keyboard_input.lua
-- Attach with: player_->AddScript("Scripts/keyboard_input.lua");
--
-- To update the HUD label, set the global from C++ before calling AddScript:
--   lua_["posLabel"] = posLabel_;   -- expose the Label* to Lua

local speed = 5.0   -- world units per second

function Start()
    entity = self.node   -- simple-3d convention: alias self.node for cleaner syntax
end

function Update(dt)
    -- ── Continuous movement: IsKeyDown fires every frame the key is held ──
    local move = Vector3(0, 0, 0)

    if Input:IsKeyDown("W") or Input:IsKeyDown("Up")    then move.z = move.z + 1 end
    if Input:IsKeyDown("S") or Input:IsKeyDown("Down")  then move.z = move.z - 1 end
    if Input:IsKeyDown("A") or Input:IsKeyDown("Left")  then move.x = move.x - 1 end
    if Input:IsKeyDown("D") or Input:IsKeyDown("Right") then move.x = move.x + 1 end

    -- Normalise diagonal movement, then scale by speed and delta time
    if move:Length() > 0 then
        entity:TranslateWorld(move:Normalized() * speed * dt)
    end

    -- ── One-shot actions: IsKeyPress fires only on the first frame ─────────
    if Input:IsKeyPress("Space") then
        entity:SetPosition(0, 0, 0)   -- teleport to origin
    end

    -- ── Live HUD: update position label if the global reference is set ─────
    local pos = entity:GetPosition()
    if posLabel then
        posLabel:SetText(string.format(
            "Position: (%.2f, %.2f, %.2f)",
            pos.x, pos.y, pos.z))
    end
end

Three differences from the C++ version are worth noting:

  • Method name: Lua uses Input:IsKeyPress("Space") (without the trailing d) to match the Urho3D Lua binding convention. The underlying behaviour — fires exactly once per keypress — is identical to C++ IsKeyPressed(Key::Space).
  • Vector3 components: In Lua, access the components as pos.x, pos.y, pos.z (no trailing underscores). The Lua binding strips the Urho3D C++ underscore convention.
  • String formatting: Lua's built-in string.format with "%.2f" is the equivalent of C++ std::snprintf with the same format specifier. Both produce a string with two decimal places.

Common mistakes

Pitfalls to avoid

1. Using IsKeyPressed for movement
IsKeyPressed returns true for exactly one frame per keypress. If you use it for movement, the entity jolts forward one step per keypress and then stops, even while the key is held. Use IsKeyDown for anything that should repeat continuously while held.

2. Forgetting to multiply by dt
player_->TranslateWorld(Vector3(0, 0, 0.1f)); moves a constant per frame. At 60 fps that is 6 units/sec; at 144 fps it is 14.4 units/sec. The game behaves differently on every machine. Always write direction * speed * dt.

3. Not normalizing diagonal movement
Accumulating two perpendicular unit contributions produces a vector of length √2 ≈ 1.414. Without calling Normalized(), diagonal movement is 41 % faster than axis-aligned movement. Always guard with if (move.Length() > 0.0f) before normalizing to avoid dividing a zero vector.

4. Not updating the label every frame
A Label does not automatically track a variable. If you call SetText() only in Start(), the HUD will always show the initial string. Position display requires SetText() inside Update() so it refreshes with each new frame.

5. Calling game API from the constructor
CreateEntity(), CreateLabel(), and all other scene-creation methods require the engine to be fully initialised. Do not call them from KeyInputDemo() — put everything in Start().

6. Comparing floating-point position for exact equality
After moving an entity with TranslateWorld, do not test if (pos.x_ == 5.0f). Floating-point accumulation introduces tiny rounding errors. Use a tolerance check: if (std::abs(pos.x_ - 5.0f) < 0.01f).

What you learned

  • IsKeyDown(Key) returns true on every frame the key is physically held, making it the right choice for continuous actions like movement, camera rotation, and acceleration.
  • IsKeyPressed(Key) returns true on the first frame of a keypress only — a rising-edge detector. Use it for one-shot events: jumping, firing, toggling menus, and quitting.
  • The Key enum provides named constants for every keyboard key: Key::W, Key::A, Key::S, Key::D, Key::Up, Key::Down, Key::Left, Key::Right, Key::Space, Key::Escape, and many more.
  • Providing two key alternatives (WASD and arrow keys) requires only a logical OR: if (IsKeyDown(Key::W) || IsKeyDown(Key::Up)).
  • Multiplying the direction by dt converts per-frame movement into per-second movement, making the game run identically at any frame rate. The formula is always direction * speed * dt.
  • Calling Vector3::Normalized() before scaling prevents diagonal movement from being faster than axis-aligned movement. Guard with if (move.Length() > 0.0f) to avoid normalizing the zero vector.
  • entity->GetPosition() returns the entity's current world-space Vector3. Its x_, y_, and z_ members can be formatted with std::snprintf and displayed live via label->SetText() inside Update().
  • In Lua, the equivalents are Input:IsKeyDown("W") / Input:IsKeyPress("Space"), and vector components are accessed as .x, .y, .z (no trailing underscores).
