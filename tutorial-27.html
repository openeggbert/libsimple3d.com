Background: what debug draw is and how it works

During development it is invaluable to see invisible data: the exact shape of a physics collider, the waypoints a nav agent computed, the detection radius of an enemy, the bounds of a trigger zone, the current AI state printed above an NPC's head. Debug draw provides exactly these overlays without requiring you to place persistent scene geometry, add UI elements, or rebuild assets.

All DrawDebug* geometry calls submit wireframe primitives to Urho3D's built-in DebugRenderer. The renderer auto-clears them at the start of each new frame, so you simply call them every Update tick and they appear for exactly one frame. DrawDebugText works slightly differently: it creates a temporary Text3D scene node that is also removed at the beginning of the next frame.

Crucially, when SetDebugDrawEnabled(false) (the default), every DrawDebug* call returns immediately without touching the renderer or allocating memory. Calling SetDebugDrawEnabled(true) only in debug builds — or behind a runtime toggle — means shipping code pays exactly zero overhead from debug draw calls left in your Update loops.

Geometry primitives

Lines, boxes, and spheres are rendered by Urho3D's DebugRenderer as single-pixel wireframes. They appear on top of scene geometry and are automatically cleared every frame. No persistent nodes are created; submit the calls each tick in Update().

Billboard text

DrawDebugText creates a Text3D node at the given world position. The node always faces the camera (billboard mode) and is removed at the start of the next HandleUpdate. Disabling debug draw mid-frame with SetDebugDrawEnabled(false) also removes any pending text nodes immediately.

Complete working example

The example below builds a small scene with a player, several AI patrol enemies, a trigger zone, and a navigation target. Press F1 to toggle the debug overlay on and off. When enabled, you will see collider boxes and spheres on every entity, a path line from the first enemy to the player, detection radius circles on all enemies, floating state labels above each enemy, and a box outline around the trigger zone. 

#include <Simple3D/Simple3D.h>

using namespace Simple3D;

// ────────────────────────────────────────────────────────────────────
// DebugDrawDemo — showcases all DrawDebug* helpers in one scene.
// Press F1 to toggle the debug overlay at runtime.
// ────────────────────────────────────────────────────────────────────
class DebugDrawDemo : public Game
{
public:
    // ── Scene entities ────────────────────────────────────────────────
    Entity*  player_        = nullptr;
    Entity*  navTarget_     = nullptr;
    Entity*  triggerZone_   = nullptr;
    Camera*  cam_           = nullptr;

    // ── Enemies — tagged "enemy" for batch retrieval ──────────────────
    std::vector<Entity*> enemies_;

    // ── AI states for the enemies ─────────────────────────────────────
    std::vector<String>  enemyStates_;

    // ── Detection radius that each enemy uses ─────────────────────────
    static constexpr float DETECT_RADIUS = 5.0f;

    // ── Trigger zone half-size ────────────────────────────────────────
    static constexpr float TRIGGER_HALF = 3.0f;

    // ── Player speed ─────────────────────────────────────────────────
    static constexpr float SPEED = 5.0f;

    // ────────────────────────────────────────────────────────────────
    // Start() — build the scene and enable debug draw in debug builds.
    // ────────────────────────────────────────────────────────────────
    void Start() override
    {
        SetAppName("DebugDrawDemo");

        // ── Debug draw: enable only in debug builds. ─────────────────
        // When disabled (the default) every DrawDebug* call is a
        // zero-cost no-op, so it is safe to leave them in Update()
        // unconditionally for shipping code.
#ifdef _DEBUG
        SetDebugDrawEnabled(true);
#endif

        // ── Ground plane ─────────────────────────────────────────────
        Entity* ground = CreateEntity("Ground");
        ground->AddModel("Models/Plane.mdl");
        ground->SetScale(Vector3(30.0f, 1.0f, 30.0f));
        ground->AddRigidBody(0.0f);                       // mass 0 = static
        ground->AddCollisionShape(CollisionShape::Box,
                                    Vector3(15.0f, 0.1f, 15.0f));

        // ── Player ───────────────────────────────────────────────────
        player_ = CreateEntity("Player");
        player_->AddModel("Models/Box.mdl");
        player_->SetScale(0.8f);
        player_->SetPosition(Vector3(0.0f, 0.5f, 0.0f));
        player_->AddCharacterController(0.4f, 0.8f);

        // ── Trigger zone (invisible box at the far end of the arena) ─
        triggerZone_ = CreateEntity("TriggerZone");
        triggerZone_->SetPosition(Vector3(10.0f, 1.5f, 10.0f));
        triggerZone_->AddRigidBody(0.0f);
        triggerZone_->AddCollisionShape(CollisionShape::Box,
                                         Vector3(TRIGGER_HALF,
                                                 TRIGGER_HALF,
                                                 TRIGGER_HALF));
        triggerZone_->SetTrigger(true);
        triggerZone_->SetOnTriggerEnter([this](Entity* other) {
            if (other == player_)
                Log("Player entered trigger zone!");
        });

        // ── Navigation target marker ──────────────────────────────────
        navTarget_ = CreateEntity("NavTarget");
        navTarget_->SetPosition(Vector3(8.0f, 0.0f, 8.0f));

        // ── Spawn three patrol enemies ────────────────────────────────
        Vector3 spawnPositions[] = {
            Vector3(-4.0f, 0.5f,  4.0f),
            Vector3( 4.0f, 0.5f, -4.0f),
            Vector3(-6.0f, 0.5f, -6.0f)
        };
        String enemyNames[] = { "EnemyA", "EnemyB", "EnemyC" };

        for (int i = 0; i < 3; ++i)
        {
            Entity* e = CreateEntity(enemyNames[i]);
            e->SetTag("enemy");
            e->AddModel("Models/Cone.mdl");
            e->SetScale(0.7f);
            e->SetPosition(spawnPositions[i]);
            e->AddRigidBody(1.0f);
            e->AddCollisionShape(CollisionShape::Capsule,
                                  Vector3(0.35f, 0.7f, 0.35f));
            enemies_.push_back(e);
            enemyStates_.push_back("Patrol");
        }

        // ── Lighting ─────────────────────────────────────────────────
        SetAmbientLight(Color(0.3f, 0.3f, 0.3f));
        Entity* sun = CreateEntity("Sun");
        sun->AddDirectionalLight(Color(1.0f, 0.95f, 0.8f), 1.2f);
        sun->SetRotation(Quaternion(45.0f, 30.0f, 0.0f));

        // ── Camera ───────────────────────────────────────────────────
        cam_ = CreateCamera();
        cam_->Follow(player_, 10.0f, 4.0f);
        cam_->SetSmoothFollow(true, 5.0f);

        // ── HUD hint ─────────────────────────────────────────────────
        Label* hint = CreateLabel(
            "WASD = move   F1 = toggle debug draw");
        hint->SetPosition(10, 10);
        hint->SetFontSize(15);
        hint->SetColor(Color::WHITE);
    }

    // ────────────────────────────────────────────────────────────────
    // Update() — player movement, AI logic, and all debug draw calls.
    // ────────────────────────────────────────────────────────────────
    void Update(float dt) override
    {
        // ── Toggle debug draw with F1 ─────────────────────────────────
        if (IsKeyPressed(Key::F1))
            SetDebugDrawEnabled(!IsDebugDrawEnabled());

        if (IsKeyPressed(Key::Escape))
            Quit();

        // ── Player movement ───────────────────────────────────────────
        HandleMovement(dt);

        // ── Minimal AI: decide state per enemy ───────────────────────
        Vector3 playerPos = player_->GetPosition();
        for (int i = 0; i < (int)enemies_.size(); ++i)
        {
            float dist = (enemies_[i]->GetPosition() - playerPos).Length();
            enemyStates_[i] = (dist < DETECT_RADIUS) ? "Chase!" : "Patrol";
        }

        // ── All debug draw calls ──────────────────────────────────────
        // When SetDebugDrawEnabled(false) all calls below are no-ops.
        // In release builds compiled with _DEBUG undefined and
        // SetDebugDrawEnabled never called, they cost nothing.

        DrawColliderOverlays();
        DrawEnemyDebug();
        DrawNavPath();
        DrawTriggerZoneOverlay();
    }

    // ────────────────────────────────────────────────────────────────
    // DrawColliderOverlays() — draw a white box around the player and
    // cyan boxes around each static obstacle to make collider shapes
    // visible without opening a physics debugger.
    // ────────────────────────────────────────────────────────────────
    void DrawColliderOverlays()
    {
        // Player has a capsule controller; approximate it with a box overlay.
        DrawDebugBox(
            player_->GetPosition(),
            Vector3(0.4f, 0.8f, 0.4f),
            Color::WHITE);

        // Ground plane — show its thin box collider in grey.
        DrawDebugBox(
            Vector3(0.0f, -0.05f, 0.0f),
            Vector3(15.0f, 0.1f, 15.0f),
            Color(0.5f, 0.5f, 0.5f, 1.0f));
    }

    // ────────────────────────────────────────────────────────────────
    // DrawEnemyDebug() — for each enemy draw:
    //   • a red sphere at their feet (capsule collider approximation)
    //   • a yellow sphere showing their detection radius
    //   • a floating text label with their name and AI state
    // ────────────────────────────────────────────────────────────────
    void DrawEnemyDebug()
    {
        for (int i = 0; i < (int)enemies_.size(); ++i)
        {
            Entity* e     = enemies_[i];
            Vector3 pos   = e->GetPosition();
            bool    chasing = (enemyStates_[i] == "Chase!");

            // Capsule collider — show as a small red sphere at the base.
            DrawDebugSphere(pos, 0.35f, Color::RED);

            // Detection radius — yellow when patrolling, magenta when chasing.
            Color radiusCol = chasing
                ? Color(1.0f, 0.0f, 1.0f, 1.0f)   // magenta
                : Color::YELLOW;
            DrawDebugSphere(pos, DETECT_RADIUS, radiusCol);

            // Line from enemy to player when chasing — shows the pursuit vector.
            if (chasing)
            {
                DrawDebugLine(
                    pos,
                    player_->GetPosition(),
                    Color(1.0f, 0.4f, 0.0f, 1.0f));   // orange
            }

            // Billboard text label 2.2 units above the entity origin.
            Vector3 labelPos = pos + Vector3(0.0f, 2.2f, 0.0f);
            String  label    = e->GetName() + ": " + enemyStates_[i];
            Color   textCol  = chasing
                ? Color(1.0f, 0.3f, 0.3f, 1.0f)     // red-ish when chasing
                : Color::WHITE;
            DrawDebugText(labelPos, label, textCol);
        }
    }

    // ────────────────────────────────────────────────────────────────
    // DrawNavPath() — compute the nav mesh path from the first enemy
    // to the nav target and draw each path segment as a cyan line.
    // This makes it easy to verify that pathfinding is working.
    // ────────────────────────────────────────────────────────────────
    void DrawNavPath()
    {
        if (enemies_.empty()) return;

        auto path = enemies_[0]->FindPath(navTarget_->GetPosition());

        // Draw each segment of the computed path.
        for (int i = 0; i + 1 < (int)path.size(); ++i)
            DrawDebugLine(path[i], path[i + 1], Color::CYAN);

        // Mark each waypoint with a small yellow sphere.
        for (const Vector3& wp : path)
            DrawDebugSphere(wp, 0.12f, Color::YELLOW);

        // Label the destination.
        if (!path.empty())
        {
            DrawDebugText(
                navTarget_->GetPosition() + Vector3(0.0f, 1.5f, 0.0f),
                "NavTarget",
                Color::CYAN);
        }
    }

    // ────────────────────────────────────────────────────────────────
    // DrawTriggerZoneOverlay() — draw a green wire box at the trigger's
    // position and size so artists can see the invisible volume.
    // ────────────────────────────────────────────────────────────────
    void DrawTriggerZoneOverlay()
    {
        DrawDebugBox(
            triggerZone_->GetPosition(),
            Vector3(TRIGGER_HALF, TRIGGER_HALF, TRIGGER_HALF),
            Color::GREEN);

        DrawDebugText(
            triggerZone_->GetPosition() + Vector3(0.0f, TRIGGER_HALF + 0.5f, 0.0f),
            "Trigger Zone",
            Color::GREEN);
    }

    // ────────────────────────────────────────────────────────────────
    void HandleMovement(float dt)
    {
        CharacterController* cc =
            player_->GetComponent<CharacterController>();
        if (!cc) return;

        float mx = 0.0f, mz = 0.0f;
        if (IsKeyDown(Key::A) || IsKeyDown(Key::Left))  mx -= 1.0f;
        if (IsKeyDown(Key::D) || IsKeyDown(Key::Right)) mx += 1.0f;
        if (IsKeyDown(Key::W) || IsKeyDown(Key::Up))    mz -= 1.0f;
        if (IsKeyDown(Key::S) || IsKeyDown(Key::Down))  mz += 1.0f;

        cc->Move(Vector3(mx, 0.0f, mz) * SPEED, dt);

        if (IsKeyPressed(Key::Space) && cc->IsOnGround())
            cc->TryJump(8.0f);
    }
};

int main()
{
    return Simple3D::CreateGame<DebugDrawDemo>()->Run();
}

Step-by-step explanation

SetDebugDrawEnabled(bool)

This is the master switch for the debug draw subsystem. When set to false (the default), all DrawDebug* calls are complete no-ops: they do not touch the renderer, do not allocate memory, and return immediately. The flag is checked at the entry point of each helper, so even thousands of DrawDebugSphere calls scattered through a large Update loop cost nothing in a build where the flag has never been set to true.

Call it once in Start(). The canonical pattern is to gate it behind the preprocessor so release builds never enable it accidentally: 

void Start() override
{
#ifdef _DEBUG
    SetDebugDrawEnabled(true);
#endif
}

You can also allow the player (or an internal developer menu) to toggle it at runtime using IsDebugDrawEnabled() as the current state query: 

void Update(float dt) override
{
    // F1 flips the current state — no explicit true/false needed.
    if (IsKeyPressed(Key::F1))
        SetDebugDrawEnabled(!IsDebugDrawEnabled());
}

When you call SetDebugDrawEnabled(false) while debug draw is active, any Text3D billboard nodes that were created by DrawDebugText in the current frame are removed immediately. Geometry submitted to the DebugRenderer is cleared at the end of the frame regardless.

DrawDebugLine(from, to, color=WHITE)

Submits a single line segment to Urho3D's DebugRenderer. The line exists for exactly one frame and is automatically cleared before the next Update runs. You must call it every frame for a persistent visual. It is the most versatile primitive: use it for velocity arrows, ray cast paths, bone chains, field-of-view edges, or any directed relationship between two world positions. 

// Simplest form — white line between two points
DrawDebugLine(Vector3(0.0f, 0.0f, 0.0f),
              Vector3(5.0f, 0.0f, 0.0f));

// Colored line between two entity positions
DrawDebugLine(shooter->GetPosition(),
              target->GetPosition(),
              Color::RED);

// Velocity vector: draw from entity position scaled by velocity
Vector3 vel = rb->GetLinearVelocity();
DrawDebugLine(entity->GetPosition(),
              entity->GetPosition() + vel,
              Color::YELLOW);

// Visualise a nav path — call in a loop across path waypoints
auto path = agent->FindPath(destination);
for (int i = 0; i + 1 < (int)path.size(); ++i)
    DrawDebugLine(path[i], path[i + 1], Color::CYAN);

DrawDebugBox(center, halfSize, color=WHITE)

Draws a wireframe axis-aligned bounding box (AABB) centered at center with half-extents halfSize. Because the box is axis-aligned, its edges always run parallel to the world X, Y, and Z axes. This is the right tool for visualizing CollisionShape::Box colliders, spawn area bounds, grid cells, and trigger volumes.

The halfSize parameter uses half extents, matching the convention used by AddCollisionShape. If your collision shape was created with Vector3(2.0f, 1.0f, 2.0f), pass the same vector directly to DrawDebugBox to see the exact collider boundary: 

// Mirror a box collider exactly — same half-extents as AddCollisionShape
Vector3 halfExtents(2.0f, 1.0f, 2.0f);
entity->AddCollisionShape(CollisionShape::Box, halfExtents);
DrawDebugBox(entity->GetPosition(), halfExtents, Color::GREEN);

// Trigger zone — green so it stands out from physics colliders
DrawDebugBox(
    triggerEntity->GetPosition(),
    Vector3(3.0f, 3.0f, 3.0f),
    Color::GREEN);

// Grid cell in a procedural level — iterate and draw each cell
for (int z = 0; z < GRID_H; ++z)
    for (int x = 0; x < GRID_W; ++x)
    {
        Vector3 cellCenter(x * CELL_SIZE, 0.0f, z * CELL_SIZE);
        DrawDebugBox(cellCenter,
                     Vector3(CELL_SIZE * 0.5f, 0.05f, CELL_SIZE * 0.5f),
                     Color(0.2f, 0.2f, 0.8f, 1.0f));
    }

DrawDebugSphere(center, radius, color=WHITE)

Draws a wireframe sphere at center with the given radius. Internally this renders several latitude and longitude circle arcs to approximate the sphere shell. It is ideal for visualizing sphere colliders, detection radii, explosion blast radii, audio attenuation ranges, and any other concept that is fundamentally circular in all directions. 

// Show a sphere collider in red
DrawDebugSphere(entity->GetPosition(), 0.5f, Color::RED);

// Enemy detection radius — yellow when idle, magenta when alert
Color col = enemy->IsAlert()
    ? Color(1.0f, 0.0f, 1.0f, 1.0f)
    : Color::YELLOW;
DrawDebugSphere(enemy->GetPosition(), DETECT_RADIUS, col);

// Explosion blast radius — briefly visible on detonation frame
DrawDebugSphere(blastCenter, blastRadius,
               Color(1.0f, 0.5f, 0.0f, 1.0f));   // orange

// Nav path waypoints — tiny spheres at each position
for (const Vector3& wp : path)
    DrawDebugSphere(wp, 0.1f, Color::CYAN);

DrawDebugText(worldPos, text, color=WHITE)

Places a Text3D scene node at worldPos that always faces the camera (billboard mode). The node is removed at the start of the next HandleUpdate, so calling this once per frame gives a persistent label with no manual cleanup. This is the quickest way to put readable information directly in 3D space: entity names, health points, physics body states, AI state machine transitions, or any string value you want to inspect in context.

Position the text slightly above the entity so it clears the mesh. A Y offset between 1.5f and 2.5f works well for human-scale characters: 

// Entity name floating above its head
DrawDebugText(
    entity->GetPosition() + Vector3(0.0f, 2.2f, 0.0f),
    entity->GetName());

// Health value with a warm-color tint
DrawDebugText(
    entity->GetPosition() + Vector3(0.0f, 2.0f, 0.0f),
    "HP: " + String(currentHp),
    Color(1.0f, 0.5f, 0.0f, 1.0f));

// AI state — colour-coded: white=patrol, red=chase, green=search
Color stateColor =
    (state == AIState::Chase)  ? Color::RED   :
    (state == AIState::Search) ? Color::GREEN :
                                 Color::WHITE;
DrawDebugText(
    enemy->GetPosition() + Vector3(0.0f, 2.5f, 0.0f),
    aiStateToString(state),
    stateColor);

// Batch-label all entities tagged "interactable"
for (Entity* e : FindEntitiesByTag("interactable"))
    DrawDebugText(
        e->GetPosition() + Vector3(0.0f, 1.8f, 0.0f),
        e->GetName(),
        Color::WHITE);

Zero overhead in release builds

Every DrawDebug* function begins with a single branch on the enabled flag. When the flag is false, the function returns immediately. No vector math, no renderer submissions, no string construction. This means you can write expressive debug draw calls spread throughout your Update, AI, and physics code and leave them there permanently: in a release build compiled without calling SetDebugDrawEnabled(true) they cost one branch each at most — typically far less than a cache miss or a renderer state change.

For the most conservative approach, combine the runtime flag with a preprocessor guard so the calls are completely elided in release compilation: 

void Start() override
{
#ifdef _DEBUG
    SetDebugDrawEnabled(true);
#endif
}

void Update(float dt) override
{
    // These calls are in shipping code.
    // In _DEBUG builds: rendered as wireframes.
    // In release builds: SetDebugDrawEnabled was never called, so
    //                    each call returns immediately, costing ~1 ns.
    DrawDebugSphere(player_->GetPosition(), 0.5f, Color::WHITE);
    DrawDebugBox(triggerZone_->GetPosition(),
                 Vector3(3.0f, 3.0f, 3.0f),
                 Color::GREEN);
}

Practical visualization patterns

Visualizing physics colliders

Draw a debug box or sphere in your Update at the same position and size as each entity's collision shape. This lets you spot mismatches between the visual mesh and the physics volume without opening a separate physics debugger tool. Use distinct colors for different collision layers: white for the player, grey for static geometry, red for enemies, green for triggers. 

// Show all entities' approximate collider boxes by iterating tagged groups
for (Entity* e : FindEntitiesByTag("obstacle"))
    DrawDebugBox(e->GetPosition(), e->GetScale() * 0.5f,
                 Color(0.6f, 0.6f, 0.6f, 1.0f));

Visualizing nav mesh paths

Call FindPath on your nav agent each frame and draw lines between consecutive waypoints. This confirms that the nav mesh is baked correctly, that the agent's target is reachable, and lets you watch the path update in real time as dynamic obstacles move: 

void Update(float dt) override
{
    auto path = navAgent_->FindPath(player_->GetPosition());

    for (int i = 0; i + 1 < (int)path.size(); ++i)
        DrawDebugLine(path[i], path[i + 1], Color::CYAN);

    // Show the end of the path
    if (!path.empty())
        DrawDebugSphere(path.back(), 0.2f, Color::YELLOW);
}

Visualizing AI state and attention

Print the current FSM state above each AI entity and color-code detection spheres by alert level. You can also draw a line from the AI to its current target when chasing, and draw the last known player position as a small sphere to debug search behavior: 

void DrawAIDebug(Entity* ai, AIState state,
                   Vector3 lastKnownPlayerPos)
{
    Vector3 pos = ai->GetPosition();

    // Detection radius — color shows alert level
    Color col = (state == AIState::Chase)  ? Color::RED
              : (state == AIState::Search) ? Color::YELLOW
              :                                Color::GREEN;
    DrawDebugSphere(pos, 6.0f, col);

    // Current state text
    DrawDebugText(pos + Vector3(0, 2.5f, 0),
                  ai->GetName() + ": " + AIStateToStr(state), col);

    // Last known player position during Search state
    if (state == AIState::Search)
    {
        DrawDebugSphere(lastKnownPlayerPos, 0.4f,
                        Color(1.0f, 0.5f, 0.0f, 1.0f));
        DrawDebugLine(pos, lastKnownPlayerPos,
                      Color(1.0f, 0.5f, 0.0f, 1.0f));
        DrawDebugText(lastKnownPlayerPos + Vector3(0, 0.6f, 0),
                      "Last seen",
                      Color::YELLOW);
    }
}

Visualizing trigger volumes

Trigger volumes are invisible by design, which makes them easy to misconfigure. Draw a colored box or sphere in Update at each trigger's position and size so level designers can see exactly where a volume begins and ends. Use a different color from physics colliders so the two are easy to distinguish at a glance: 

// Draw every entity tagged "trigger" in green
for (Entity* t : FindEntitiesByTag("trigger"))
{
    Vector3 half = t->GetScale() * 0.5f;
    DrawDebugBox(t->GetPosition(), half, Color::GREEN);
    DrawDebugText(t->GetPosition() + Vector3(0, half.y_ + 0.3f, 0),
                  t->GetName(), Color::GREEN);
}

Quick reference

Method Returns Notes

IsDebugDrawEnabled() bool Query the current state; use for runtime toggle logic.
DrawDebugLine(from, to, color=WHITE) — One-frame wireframe line. Submit every Update tick for persistence.
DrawDebugBox(center, halfSize, color=WHITE) — Axis-aligned wireframe box. halfSize matches AddCollisionShape convention.
DrawDebugSphere(center, radius, color=WHITE) — Wireframe sphere. Good for detection radii and sphere colliders.

Lua equivalent

All debug draw methods are available on the game global inside Lua scripts. Signatures mirror C++ exactly. The one-frame rule applies identically: call them every Update tick from Lua for a persistent overlay, or call once for a single-frame flash. 

-- Scripts/debug_draw_demo.lua
-- Attach this script to any persistent entity.
-- Press F1 to toggle the debug overlay at runtime.

local player   = nil
local enemies  = {}
local DETECT   = 5.0

function Start()
    -- Enable debug draw only in debug builds.
-- (No Lua-side preprocessor; use a runtime config flag instead.)
    game:SetDebugDrawEnabled(true)

    player   = game:FindEntity("Player")
    enemies  = game:FindEntitiesByTag("enemy")
end

function Update(dt)
    -- Runtime toggle
    if game:IsKeyPressed(Key.F1) then
        game:SetDebugDrawEnabled(not game:IsDebugDrawEnabled())
    end

    -- Player collider box
    game:DrawDebugBox(
        player:GetPosition(),
        Vector3(0.4, 0.8, 0.4),
        Color.WHITE)

    -- Enemy overlays
    for _, e in ipairs(enemies) do
        local pos  = e:GetPosition()
        local dist = (pos - player:GetPosition()):Length()
        local chasing = dist < DETECT

        -- Capsule approximation
        game:DrawDebugSphere(pos, 0.35, Color.RED)

        -- Detection radius: yellow or magenta
        local col = chasing
            and Color(1, 0, 1, 1)
            or  Color.YELLOW
        game:DrawDebugSphere(pos, DETECT, col)

        -- Chase line
        if chasing then
            game:DrawDebugLine(pos, player:GetPosition(),
                                Color(1, 0.4, 0, 1))
        end

        -- Billboard label
        local label = e:GetName() .. ": "
                   .. (chasing and "Chase!" or "Patrol")
        game:DrawDebugText(
            pos + Vector3(0, 2.2, 0),
            label,
            chasing and Color(1, 0.3, 0.3, 1) or Color.WHITE)
    end

    -- Nav path from first enemy to player
    if #enemies > 0 then
        local path = enemies[1]:FindPath(player:GetPosition())
        for i = 1, #path - 1 do
            game:DrawDebugLine(path[i], path[i + 1], Color.CYAN)
        end
        for _, wp in ipairs(path) do
            game:DrawDebugSphere(wp, 0.1, Color.YELLOW)
        end
    end

    -- Trigger zone (named "TriggerZone" in the scene)
    local tz = game:FindEntity("TriggerZone")
    if tz then
        game:DrawDebugBox(tz:GetPosition(),
                           Vector3(3, 3, 3),
                           Color.GREEN)
        game:DrawDebugText(tz:GetPosition() + Vector3(0, 3.5, 0),
                           "Trigger Zone", Color.GREEN)
    end
end

Common mistakes

  • Calling DrawDebug* only once instead of every frame. All geometry calls and text calls survive for exactly one frame. If you call DrawDebugSphere in Start() or once when an event fires, the sphere appears for a single frame and then vanishes. Move all debug draw calls into Update() so they are resubmitted each tick.
  • Forgetting that SetDebugDrawEnabled is false by default. Nothing will appear until you call SetDebugDrawEnabled(true). If your overlays are invisible, check that the call is actually being reached. A common culprit is placing it inside a conditional block (#ifdef _DEBUG) without defining _DEBUG in the build.
  • Using halfSize incorrectly for DrawDebugBox. The second argument is half-extents, not full size. A call of DrawDebugBox(pos, Vector3(1,1,1)) draws a 2×2×2 metre box centered at pos. If your box looks twice as large as the mesh, divide the scale or the collision shape size by two.
  • Enabling debug draw in release builds by accident. If you call SetDebugDrawEnabled(true) unconditionally in Start() without a preprocessor guard, the debug overlay will appear in shipped builds. Gate the call inside #ifdef _DEBUG or behind a developer-only runtime flag checked against a build configuration.
  • Expecting DrawDebugText to accumulate across frames. Each call to DrawDebugText creates a new Text3D node. Calling it twice in the same frame for the same position creates two overlapping labels. Call it once per entity per frame.
  • Not accounting for entity position when sizing debug boxes. DrawDebugBox draws an axis-aligned box centered at the provided position. If your entity is offset from the origin (e.g. the collision shape is centered differently from the entity pivot), you may need to offset the center argument accordingly to match the actual collider location.
  • Calling DrawDebugLine with identical from and to points. A zero-length line is valid but invisible. If your lines disappear, verify that from != to. This often happens when referencing a position that was not yet updated (e.g. a nav target entity at the default origin).

What you learned

In this part you learned how to use simple-3d's debug draw system to overlay real-time diagnostic information directly in your 3D world:

  • Enabling the system: SetDebugDrawEnabled(true) activates the subsystem; IsDebugDrawEnabled() reads the current state so you can build a runtime toggle with a single key press. Gate the enable call behind #ifdef _DEBUG to guarantee zero impact on release builds.
  • Lines: DrawDebugLine(from, to, color) draws a one-frame world-space line segment. Resubmit every Update tick for a persistent overlay. Essential for velocity vectors, ray casts, chase lines, and nav path segments.
  • Boxes: DrawDebugBox(center, halfSize, color) draws an axis-aligned wireframe box using the same half-extents convention as AddCollisionShape. Use it to expose invisible box colliders and trigger zone boundaries to level designers and programmers alike.
  • Spheres: DrawDebugSphere(center, radius, color) draws a wireframe sphere. Ideal for sphere colliders, detection radii, blast radii, and audio attenuation boundaries.
  • Text: DrawDebugText(worldPos, text, color) places a camera-facing Text3D node for one frame. Use it to display entity names, HP values, and AI state machine transitions floating above entities in world space without any persistent UI setup.
  • Zero release overhead: all calls are no-ops when the flag is false, so you can leave debug draw calls in your shipping code paths without any measurable runtime cost.
  • Practical patterns: you now know how to visualize physics colliders, nav mesh paths, AI detection radii and states, and trigger volumes using a consistent color-coding convention that makes complex runtime behavior readable at a glance.
