Background: what is a BehaviorFSM?

A finite-state machine is a model in which an entity is always in exactly one named state, and transitions between states are driven by explicit conditions. For game AI, this maps perfectly onto enemy behaviour: an enemy is either patrolling, chasing, attacking, or stunned — never two at once, and always with a clear rule governing when it switches.

Simple3D::BehaviorFSM is a lightweight FSM built on top of an entity. It integrates directly with the engine's NavMesh layer (introduced in Part 14) so you can get patrol routes and path-following for free with a single call, or write completely custom onEnter / onUpdate(dt) / onExit callbacks for any state that needs bespoke logic.

Built-in nav states

AddPatrolState() and AddChaseState() wrap the NavMesh movement API from Part 14. The FSM internally calls entity->MoveToward() and entity->StopMoving() at the right moments. You get NavMesh path-finding with zero extra boilerplate.

Custom callback states

AddState() accepts three lambdas: one called on entry, one called every frame with dt, and one called on exit. Pass {} for any callback you do not need. This handles everything that the built-in states cannot: playing animations, running timers, spawning projectiles, and so on.

Transitions are evaluated in the order they were registered. Each frame, fsm->Update(dt) checks all transitions from the current state and fires the first one whose condition returns true. This makes ordering matter: put the most urgent transitions first.

NavMesh prerequisite

AddPatrolState() and AddChaseState() both rely on the engine's NavMesh to plan paths. You must call BuildNavMesh() in Start() after placing all static level geometry (floors, walls, obstacles), but before calling any AddPatrolState or AddChaseState. Custom states created with AddState() do not require the NavMesh. 

void Start() override
{
    // 1. Build static geometry first.
    Entity* floor = CreateEntity("Floor");
    floor->AddModel("Models/Plane.mdl");
    floor->AddBoxCollider(Vector3(40.0f, 0.2f, 40.0f));
    floor->AddRigidBody(0.0f); // static
    floor->SetCollisionLayer(CollisionLayer::StaticGeometry);

    // 2. Bake the NavMesh — agents 0.4 m radius, 1.8 m height.
    BuildNavMesh(0.4f, 1.8f);

    // 3. Now safe to create enemies with patrol and chase states.
    SpawnEnemy();
}

Complete working example

The example below implements a full enemy with four states:

  1. Idle — the enemy stands still and scans. It transitions to patrol when the alert timer expires.
  2. Patrol — the enemy walks a NavMesh-backed waypoint loop. It switches to chase when the player comes within detection range.
  3. Chase — the enemy runs toward the player along a NavMesh path. It switches to attack when close enough, or falls back to patrol when the player escapes far enough.
  4. Attack — the enemy stops moving and swings. Once the attack animation finishes (detected by a timer) it re-evaluates whether to chase or return to patrol.

A fifth state, Stunned, can be triggered externally at any time by calling fsm->SetState("stunned"). This demonstrates the manual override pattern for hit-reactions and interrupts. A HUD label always shows the current FSM state so you can see transitions in real time. 

#include <Simple3D/Simple3D.h>

using namespace Simple3D;

// ────────────────────────────────────────────────────────────────────────────
// AIBehaviorGame — demonstrates the full BehaviorFSM API.
// Controls: WASD = move player, Space = jump, E = punch enemy (triggers stun).
// ────────────────────────────────────────────────────────────────────────────
class AIBehaviorGame : public Game
{
public:
    // ── Scene objects ────────────────────────────────────────────────────────
    Entity*       player_    = nullptr;
    Entity*       enemy_     = nullptr;
    Camera*       cam_       = nullptr;

    // ── AI state ─────────────────────────────────────────────────────────────
    BehaviorFSM*  fsm_       = nullptr;

    // ── FSM timer variables — captured by FSM callbacks ──────────────────────
    // idleTimer_:   how long the enemy has been standing still
    // attackTimer_: how long the current attack has been running
    // stunTimer_:   how long the enemy has been stunned
    float idleTimer_   = 0.0f;
    float attackTimer_ = 0.0f;
    float stunTimer_   = 0.0f;

    // ── AI tuning constants ───────────────────────────────────────────────────
    static constexpr float DETECT_RANGE  = 10.0f; // patrol → chase threshold (m)
    static constexpr float LOSE_RANGE    = 15.0f; // chase → patrol threshold (m)
    static constexpr float ATTACK_RANGE  =  2.0f; // chase → attack threshold (m)
    static constexpr float IDLE_DURATION =  2.0f; // seconds before patrol resumes
    static constexpr float ATTACK_DURATION= 1.2f; // seconds per attack swing
    static constexpr float STUN_DURATION =  1.5f; // seconds the enemy is stunned

    // ── HUD ───────────────────────────────────────────────────────────────────
    Label*        stateLabel_  = nullptr;
    Label*        hintLabel_   = nullptr;
    UI::Panel*    hudPanel_    = nullptr;

    // ────────────────────────────────────────────────────────────────────────
    // Helper — distance between two entities
    // ────────────────────────────────────────────────────────────────────────
    float DistToPlayer() const
    {
        return (enemy_->GetPosition() - player_->GetPosition()).Length();
    }

    // ────────────────────────────────────────────────────────────────────────
    // Start() — build world, player, enemy + FSM.
    // ────────────────────────────────────────────────────────────────────────
    void Start() override
    {
        SetAppName("AIBehaviorDemo");
        SetClearColor(Color(0.15f, 0.18f, 0.22f, 1.0f));

        // ── Lighting ──────────────────────────────────────────────────────────
        Entity* sun = CreateEntity("Sun");
        sun->AddDirectionalLight(Color(1.0f, 0.95f, 0.85f), 1.2f);
        sun->SetRotation(Quaternion(45.0f, 35.0f, 0.0f));

        // ── Floor ─────────────────────────────────────────────────────────────
        Entity* floor = CreateEntity("Floor");
        floor->AddModel("Models/Plane.mdl");
        floor->SetScale(40.0f);
        floor->AddBoxCollider(Vector3(40.0f, 0.5f, 40.0f));
        floor->AddRigidBody(0.0f);
        floor->SetCollisionLayer(CollisionLayer::StaticGeometry);

        // ── Obstacles (break up patrol path) ──────────────────────────────────
        auto MakeWall = [this](Vector3 pos, Vector3 size) {
            Entity* w = CreateEntity("Wall");
            w->AddModel("Models/Box.mdl");
            w->SetPosition(pos);
            w->SetScale(size.x_, size.y_, size.z_);
            w->AddBoxCollider(size);
            w->AddRigidBody(0.0f);
            w->SetCollisionLayer(CollisionLayer::StaticGeometry);
        };
        MakeWall(Vector3( 5.0f, 1.0f,  0.0f), Vector3(1.5f, 2.0f, 6.0f));
        MakeWall(Vector3(-5.0f, 1.0f,  3.0f), Vector3(1.5f, 2.0f, 4.0f));
        MakeWall(Vector3( 0.0f, 1.0f, -7.0f), Vector3(6.0f, 2.0f, 1.5f));

        // ── NavMesh ───────────────────────────────────────────────────────────
        // Must be called AFTER static geometry is placed, BEFORE FSM patrol/chase
        // states are created. Agent radius 0.4 m, height 1.8 m, cell size 0.25 m.
        BuildNavMesh(0.4f, 1.8f, 0.25f);

        // ── Player ────────────────────────────────────────────────────────────
        player_ = CreateEntity("Player");
        player_->AddModel("Models/Box.mdl");
        player_->SetScale(0.7f);
        player_->SetPosition(0.0f, 1.0f, 10.0f);

        CharacterController* cc = player_->AddCharacterController(0.35f, 1.6f);
        cc->SetGravity(22.0f);
        player_->SetCollisionLayer(CollisionLayer::Actor);
        player_->SetCollisionMask(MakeCollisionMask({
            CollisionLayer::StaticGeometry
        }));

        // ── Enemy ─────────────────────────────────────────────────────────────
        enemy_ = CreateEntity("Enemy");
        enemy_->AddModel("Models/Box.mdl");
        enemy_->SetScale(0.8f);
        enemy_->SetPosition(0.0f, 1.0f, 0.0f);
        enemy_->AddRigidBody(60.0f);
        enemy_->AddCapsuleCollider(0.4f, 1.6f);
        enemy_->SetCollisionLayer(CollisionLayer::Actor);
        enemy_->SetCollisionMask(MakeCollisionMask({
            CollisionLayer::StaticGeometry
        }));

        // ── Build the FSM ─────────────────────────────────────────────────────
        // AddBehaviorFSM() attaches the component and returns a raw pointer.
        // The FSM is owned by the entity; do not delete it manually.
        fsm_ = enemy_->AddBehaviorFSM();

        SetupFSM();

        // ── Camera ────────────────────────────────────────────────────────────
        cam_ = CreateCamera();
        cam_->Follow(player_, 9.0f, 4.0f);
        cam_->SetSmoothFollow(true, 5.0f);
        cam_->SetCollisionEnabled(true);

        // ── HUD ───────────────────────────────────────────────────────────────
        hudPanel_ = CreatePanel();
        hudPanel_->SetSize(280, 72);
        hudPanel_->SetAnchor(UI::Anchor::TopLeft);
        hudPanel_->SetPosition(8, 8);
        hudPanel_->SetColor(Color(0.0f, 0.0f, 0.0f, 0.60f));

        stateLabel_ = CreateLabel("Enemy state: idle");
        stateLabel_->SetPosition(10, 8);
        stateLabel_->SetFontSize(16);
        stateLabel_->SetColor(Color(0.5f, 1.0f, 0.5f, 1.0f));
        hudPanel_->AddChild(stateLabel_);

        hintLabel_ = CreateLabel("WASD=move  Space=jump  E=stun enemy");
        hintLabel_->SetPosition(10, 40);
        hintLabel_->SetFontSize(12);
        hintLabel_->SetColor(Color(0.7f, 0.7f, 0.7f, 1.0f));
        hudPanel_->AddChild(hintLabel_);
    }

    // ────────────────────────────────────────────────────────────────────────
    // SetupFSM() — defines all five states and their transitions.
    // Called once from Start() after the NavMesh is ready.
    //
    // State priority map (first match in transition list wins each frame):
    //
    //   idle ──────────────────────────────── (timer) ──────────────► patrol
    //   patrol ─────────────────────────── (dist < 10) ──────────────► chase
    //   chase ──────────────────────────── (dist < 2) ───────────────► attack
    //   chase ──────────────────────────── (dist > 15) ──────────────► patrol
    //   attack ─────────────────────────── (dist > 2.5 || timer done) ► chase
    //   * ──── SetState("stunned") (manual override at any time) ────► stunned
    //   stunned ─────────────────────────── (timer) ─────────────────► patrol
    // ────────────────────────────────────────────────────────────────────────
    void SetupFSM()
    {
        // ── STATE: idle ───────────────────────────────────────────────────────
        // Enemy stands still and resets its alert timer.
        // First registered state = default starting state.
        fsm_->AddState("idle",
            /*onEnter*/ [this]{
                enemy_->StopMoving();
                idleTimer_ = 0.0f;
                // TODO: trigger idle animation: anim->Play("idle")
            },
            /*onUpdate*/ [this](float dt){
                idleTimer_ += dt;
            },
            /*onExit*/ {}  // no cleanup needed on exit
        );

        // ── STATE: patrol ─────────────────────────────────────────────────────
        // AddPatrolState() creates a built-in NavMesh patrol loop.
        // The enemy walks each waypoint in order and repeats from the start.
        // Speed is in metres per second.
        fsm_->AddPatrolState("patrol", {
            Vector3(-8.0f, 0.0f,  0.0f),
            Vector3(-8.0f, 0.0f, -6.0f),
            Vector3( 0.0f, 0.0f, -6.0f),
            Vector3( 8.0f, 0.0f, -6.0f),
            Vector3( 8.0f, 0.0f,  6.0f),
            Vector3( 0.0f, 0.0f,  6.0f),
        }, 2.5f);

        // ── STATE: chase ──────────────────────────────────────────────────────
        // AddChaseState() creates a built-in NavMesh chase behaviour.
        // The enemy moves toward 'player_' at 4.5 m/s and stops
        // when within 1.8 m (stopDist).
        fsm_->AddChaseState("chase", player_, 4.5f, 1.8f);

        // ── STATE: attack ─────────────────────────────────────────────────────
        // Custom state: enemy stops, plays attack animation, waits for swing to end.
        fsm_->AddState("attack",
            /*onEnter*/ [this]{
                enemy_->StopMoving();
                attackTimer_ = 0.0f;
                // TODO: trigger attack animation: anim->Play("attack")
                // TODO: enable hitbox, deal damage to player
            },
            /*onUpdate*/ [this](float dt){
                attackTimer_ += dt;
            },
            /*onExit*/ [this]{
                // TODO: disable hitbox
            }
        );

        // ── STATE: stunned ────────────────────────────────────────────────────
        // Manual-override state: triggered by SetState("stunned") from outside,
        // e.g. when the player punches the enemy. Uses a timer to auto-recover.
        fsm_->AddState("stunned",
            /*onEnter*/ [this]{
                enemy_->StopMoving();
                stunTimer_ = 0.0f;
                // TODO: play hit-stun animation, flash material red, etc.
            },
            /*onUpdate*/ [this](float dt){
                stunTimer_ += dt;
                // The stun state self-transitions via SetState() rather than
                // AddTransition() — this is valid and fires immediately.
                if (stunTimer_ >= STUN_DURATION)
                    fsm_->SetState("patrol");
            },
            /*onExit*/ {}
        );

        // ── TRANSITIONS ───────────────────────────────────────────────────────
        // Evaluation order: first match wins.  More urgent rules first.

        // idle → patrol: after the enemy has stood still long enough.
        fsm_->AddTransition("idle", "patrol",
            [this]{ return idleTimer_ >= IDLE_DURATION; });

        // patrol → chase: player stepped into detection range.
        fsm_->AddTransition("patrol", "chase",
            [this]{ return DistToPlayer() < DETECT_RANGE; });

        // chase → attack: player is within melee reach.
        // Register BEFORE chase → patrol so a close player triggers attack, not retreat.
        fsm_->AddTransition("chase", "attack",
            [this]{ return DistToPlayer() <= ATTACK_RANGE; });

        // chase → patrol: player got away.
        fsm_->AddTransition("chase", "patrol",
            [this]{ return DistToPlayer() > LOSE_RANGE; });

        // attack → chase: swing animation finished AND player still somewhat close.
        fsm_->AddTransition("attack", "chase",
            [this]{
                return attackTimer_ >= ATTACK_DURATION
                    && DistToPlayer() <= LOSE_RANGE;
            });

        // attack → patrol: swing finished AND player got away.
        fsm_->AddTransition("attack", "patrol",
            [this]{
                return attackTimer_ >= ATTACK_DURATION
                    && DistToPlayer() > LOSE_RANGE;
            });

        // NOTE: "stunned" has NO AddTransition rules — it self-transitions via
        // SetState() inside its own onUpdate callback.
    }

    // ────────────────────────────────────────────────────────────────────────
    // Update() — player movement, stun input, FSM tick, HUD refresh.
    // ────────────────────────────────────────────────────────────────────────
    void Update(float dt) override
    {
        HandlePlayerMovement(dt);

        // E key: manually override the FSM to "stunned" from outside.
        // This is the pattern for external events (damage, cutscenes, etc.).
        if (IsKeyPressed(Key::E))
        {
            if (!fsm_->IsInState("stunned"))
                fsm_->SetState("stunned");
        }

        // Tick the FSM — must be called once every frame.
        // Evaluates all transitions from the current state and calls
        // onUpdate(dt) for whichever state is currently active.
        fsm_->Update(dt);

        // Refresh HUD — GetCurrentState() returns the active state name.
        std::string cur = fsm_->GetCurrentState();
        stateLabel_->SetText("Enemy state: " + cur);

        // Tint the label per state for quick visual feedback.
        if      (cur == "patrol")  stateLabel_->SetColor(Color(0.4f, 0.8f, 1.0f, 1.0f));
        else if (cur == "chase")   stateLabel_->SetColor(Color(1.0f, 0.8f, 0.2f, 1.0f));
        else if (cur == "attack")  stateLabel_->SetColor(Color(1.0f, 0.3f, 0.3f, 1.0f));
        else if (cur == "stunned") stateLabel_->SetColor(Color(0.8f, 0.4f, 1.0f, 1.0f));
        else                        stateLabel_->SetColor(Color(0.5f, 1.0f, 0.5f, 1.0f));

        if (IsKeyPressed(Key::Escape))
            Quit();
    }

    void HandlePlayerMovement(float dt)
    {
        CharacterController* cc = player_->GetComponent<CharacterController>();
        if (!cc) return;

        float mx = 0.0f, mz = 0.0f;
        if (IsKeyDown(Key::A) || IsKeyDown(Key::Left))  mx -= 1.0f;
        if (IsKeyDown(Key::D) || IsKeyDown(Key::Right)) mx += 1.0f;
        if (IsKeyDown(Key::W) || IsKeyDown(Key::Up))    mz -= 1.0f;
        if (IsKeyDown(Key::S) || IsKeyDown(Key::Down))  mz += 1.0f;

        cc->Move(Vector3(mx, 0.0f, mz) * 5.5f, dt);

        if (IsKeyPressed(Key::Space))
            cc->TryJump(11.0f);
    }
};

int main()
{
    return Simple3D::CreateGame<AIBehaviorGame>()->Run();
}

Step-by-step explanation

entity->AddBehaviorFSM()

AddBehaviorFSM() attaches a BehaviorFSM component to the entity and returns a raw pointer to it. The FSM is owned by the entity — you must never delete the pointer yourself. The component is automatically destroyed when the entity is destroyed. Calling AddBehaviorFSM() a second time on the same entity is safe and returns the existing component without creating a duplicate.

If you need to retrieve an FSM that was added earlier (for example in a different function or from a script), use entity->GetBehaviorFSM(), which returns nullptr if no FSM has been added. 

// Attach a new FSM and keep the pointer for configuration.
BehaviorFSM* fsm = enemy_->AddBehaviorFSM();

// Later, from a different function, retrieve it safely.
BehaviorFSM* fsm = enemy_->GetBehaviorFSM(); // nullptr if not added
if (fsm)
    fsm->SetState("stunned");

fsm->AddState(name, onEnter, onUpdate, onExit)

AddState() registers a custom state identified by a string name. It accepts three callbacks:

  • onEnter — a std::function<void()> called exactly once when the FSM transitions into this state. Use it to stop movement, start an animation, reset a timer, or enable a hitbox.
  • onUpdate — a std::function<void(float dt)> called every frame while this state is active. The dt argument is the frame delta in seconds, identical to the value passed to Game::Update(dt).
  • onExit — a std::function<void()> called exactly once when the FSM transitions out of this state. Use it to clean up: disable hitboxes, stop sounds, restore material colours.

Any callback you do not need can be replaced with an empty braced initialiser {}. The first AddState() call (or AddPatrolState / AddChaseState call) sets the default starting state. The FSM begins in this state without calling onEnter — it is already "in" that state from the start. 

// Minimal idle state — stop moving on entry, nothing on update or exit.
fsm_->AddState("idle",
    [this]{ enemy_->StopMoving(); },  // onEnter
    {},                                   // onUpdate — not needed
    {}                                    // onExit — not needed
);

// Attack state with all three callbacks used.
fsm_->AddState("attack",
    [this]{
        enemy_->StopMoving();
        attackTimer_ = 0.0f;
        // enable damage hitbox here
    },
    [this](float dt){
        attackTimer_ += dt;
    },
    [this]{
        // disable damage hitbox here
    }
);

fsm->AddPatrolState(name, waypoints, speed)

AddPatrolState() is a convenience constructor for a state that walks a list of world-space waypoints in order, looping back to the first when the last is reached. Under the hood it calls entity->MoveToward(waypoint, speed, dt) each frame and advances to the next waypoint when the entity is within stopping distance of the current one. The NavMesh is used to route around obstacles automatically; BuildNavMesh() must have been called before this state is ever entered.

The waypoint list is a std::initializer_list<Vector3> or any std::vector<Vector3>. The speed is in metres per second and defaults to 3.0f if omitted. 

// Three-waypoint patrol rectangle at speed 2 m/s.
fsm_->AddPatrolState("patrol", {
    Vector3(-8.0f, 0.0f,  0.0f),
    Vector3( 8.0f, 0.0f,  0.0f),
    Vector3( 8.0f, 0.0f, -8.0f),
}, 2.0f);

// Dynamic waypoint list built at runtime (e.g. read from scene file).
std::vector<Vector3> pts;
pts.push_back(Vector3( 0, 0,  5));
pts.push_back(Vector3( 0, 0, -5));
fsm_->AddPatrolState("patrol", pts, 3.0f);

fsm->AddChaseState(name, target, speed, stopDist)

AddChaseState() creates a state that continuously moves the entity toward a target (an Entity*) along a NavMesh path. Each frame the FSM recomputes the path if the target has moved significantly, so the enemy correctly navigates around walls and obstacles rather than walking through them. The entity stops and does not overshoot when it comes within stopDist metres of the target. The default speed is 4.0f m/s and the default stop distance is 1.5f m. 

// Chase the player at 4.5 m/s; stop once within 1.8 m (melee range).
fsm_->AddChaseState("chase", player_, 4.5f, 1.8f);

// Slow, persistent chaser that gets very close.
fsm_->AddChaseState("chase", player_, 2.0f, 0.5f);

fsm->AddTransition(from, to, conditionFn)

AddTransition() registers a condition-based edge in the state graph. Each call to fsm->Update(dt) evaluates all transitions registered from the current state, in the order they were added. The first condition that returns true fires: the FSM calls onExit for the current state, sets the new current state, and calls onEnter for the new state. Only one transition can fire per Update call.

The condition is a std::function<bool()>. Lambda captures let you read game state directly. Keep condition lambdas cheap: they run every frame on every enemy. 

// The condition captures local variables by reference — very common pattern.
float alertTimer = 0.0f;

fsm_->AddTransition("idle", "patrol",
    [&alertTimer]{ return alertTimer >= 2.0f; });

// Distance-based transition — captures 'this' for helper access.
fsm_->AddTransition("patrol", "chase",
    [this]{ return DistToPlayer() < 10.0f; });

// Compound condition — both requirements must be true.
// Because first-match wins, register more specific rules before general ones.
fsm_->AddTransition("attack", "chase",
    [this]{
        return attackTimer_ >= ATTACK_DURATION
            && DistToPlayer() <= LOSE_RANGE;
    });

fsm->SetState(name)

SetState() performs an immediate, unconditional state change regardless of any registered transitions. It calls onExit on the current state and onEnter on the named state, then sets the new state as current. It can be called from anywhere: from inside an onUpdate callback (as in the stunned timer example), from a trigger callback, or from Game::Update().

SetState() is the correct tool for interrupt events — things that should override whatever the AI is currently doing: being hit, a scripted event, a cutscene start, or a death. Never use it to replace a transition that could be expressed as a condition, since SetState() bypasses all ordering rules. 

// From Game::Update() — player punches enemy with E key.
if (IsKeyPressed(Key::E) && !fsm_->IsInState("stunned"))
    fsm_->SetState("stunned");

// From a trigger callback — an explosion sends all nearby enemies to "flee".
explosion->SetOnTriggerEnter([this](Entity* other) {
    BehaviorFSM* otherFsm = other->GetBehaviorFSM();
    if (otherFsm)
        otherFsm->SetState("flee");
});

// Self-transition from inside onUpdate — stun timer recovery.
fsm_->AddState("stunned",
    [this]{ stunTimer_ = 0.0f; enemy_->StopMoving(); },
    [this](float dt){
        stunTimer_ += dt;
        if (stunTimer_ >= 1.5f)
            fsm_->SetState("patrol");
    },
    {}
);

fsm->GetCurrentState() and fsm->IsInState(name)

GetCurrentState() returns a std::string naming the state that is currently active. It returns an empty string only if no states have been added yet. IsInState(name) is a convenience shorthand that returns true when GetCurrentState() == name; it is more readable than a string comparison and slightly cheaper because it avoids constructing a temporary string object.

Both methods are safe to call at any point in Update(), before or after fsm->Update(dt). 

// Display the state name in a HUD label each frame.
stateLabel_->SetText("AI: " + fsm_->GetCurrentState());

// Guard an action to only run in a specific state.
if (fsm_->IsInState("attack"))
    CheckMeleeDamage();

// Prevent double-stun from multiple overlapping triggers.
if (!fsm_->IsInState("stunned"))
    fsm_->SetState("stunned");

fsm->Update(dt)

fsm->Update(dt) is the engine that drives the FSM every frame. It performs three things in order:

  1. Calls the current state's onUpdate(dt) callback (if one was set).
  2. Evaluates all transitions registered from the current state in order. If a condition returns true, it fires that transition and stops evaluating.
  3. For built-in patrol and chase states, handles NavMesh movement internally.

You must call fsm->Update(dt) exactly once per frame in Game::Update(), passing the same dt that Game::Update() received. Failing to call it means the FSM never transitions and no movement occurs. 

void Update(float dt) override
{
    // ... other per-frame logic ...

    // Always call Update(dt) once per frame — no earlier, no later.
    fsm_->Update(dt);
}

Managing multiple enemies

When you have many enemies each with their own FSM, the recommended pattern is to store them in a container alongside their pointers and call GetBehaviorFSM() to update each one, rather than storing raw FSM pointers separately: 

class EnemyWaveGame : public Game
{
    std::vector<Entity*> enemies_;

    void Start() override
    {
        BuildNavMesh();

        for (int i = 0; i < 5; ++i)
        {
            Entity* e = CreateEntity("Enemy");
            e->SetPosition(Vector3(i * 4.0f - 8.0f, 1.0f, 0.0f));
            e->AddModel("Models/Box.mdl");

            BehaviorFSM* fsm = e->AddBehaviorFSM();

            // Each enemy gets its own FSM, configured identically.
            fsm->AddPatrolState("patrol", {
                e->GetPosition() + Vector3( 5, 0, 0),
                e->GetPosition() + Vector3(-5, 0, 0),
            }, 2.0f);
            fsm->AddChaseState("chase", player_, 4.0f, 1.5f);

            fsm->AddTransition("patrol", "chase",
                [e, this]{ return (e->GetPosition() - player_->GetPosition()).Length() < 8.0f; });
            fsm->AddTransition("chase", "patrol",
                [e, this]{ return (e->GetPosition() - player_->GetPosition()).Length() > 14.0f; });

            enemies_.push_back(e);
        }
    }

    void Update(float dt) override
    {
        for (Entity* e : enemies_)
        {
            BehaviorFSM* fsm = e->GetBehaviorFSM();
            if (fsm) fsm->Update(dt);
        }
    }
};

Four-state alert cycle (patrol → alert → chase → attack)

A common refinement is to add an alert state between patrol and chase: the enemy pauses, plays a "!" reaction, then begins the chase. This makes the AI feel less mechanical and gives the player a brief warning. 

float alertTimer_ = 0.0f;

// After patrol and before chase.
fsm_->AddState("alert",
    [this]{ enemy_->StopMoving(); alertTimer_ = 0.0f; },
    [this](float dt){ alertTimer_ += dt; },
    [this]{ alertTimer_ = 0.0f; }
);

// patrol → alert when player first enters detection range.
fsm_->AddTransition("patrol", "alert",
    [this]{ return DistToPlayer() < 10.0f; });

// alert → chase after brief pause OR if player runs very close.
fsm_->AddTransition("alert", "chase",
    [this]{ return alertTimer_ > 0.8f || DistToPlayer() < 4.0f; });

// alert → patrol if player left immediately without triggering chase.
fsm_->AddTransition("alert", "patrol",
    [this]{ return alertTimer_ > 3.0f && DistToPlayer() > 10.0f; });

Lua equivalent

The BehaviorFSM API is fully available from Lua scripts. Because Lua callbacks are closures, you can capture local variables exactly as you would with C++ lambdas. Note that BuildNavMesh() must still be called from C++ (or from Lua's global game object) before entering any patrol or chase state. 

-- Scripts/enemy_ai.lua
-- Attach this script to the enemy entity.
-- The global `game` and `self.node` (the entity) are available automatically.

local fsm          = nil
local player        = nil
local idleTimer     = 0
local attackTimer   = 0
local stunTimer     = 0

local DETECT_RANGE  = 10.0
local LOSE_RANGE    = 15.0
local ATTACK_RANGE  =  2.0
local ATTACK_DUR    =  1.2
local STUN_DUR      =  1.5
local IDLE_DUR      =  2.0

local function dist()
    return (self.node:GetPosition() - player:GetPosition()):Length()
end

function Start()
    player = game:FindEntity("Player")

    -- Attach and configure the FSM.
    fsm = self.node:AddBehaviorFSM()

    -- Idle state — first registered, so it is the default.
    fsm:AddState("idle",
        function()                  -- onEnter
            self.node:StopMoving()
            idleTimer = 0
        end,
        function(dt) idleTimer = idleTimer + dt end,  -- onUpdate
        nil                         -- onExit — nil is the Lua equivalent of {}
    )

    -- Built-in patrol: NavMesh waypoint loop.
    fsm:AddPatrolState("patrol", {
        Vector3(-8, 0,  0),
        Vector3( 8, 0,  0),
        Vector3( 8, 0, -8),
    }, 2.5)

    -- Built-in chase: NavMesh target follow.
    fsm:AddChaseState("chase", player, 4.5, 1.8)

    -- Custom attack state.
    fsm:AddState("attack",
        function()
            self.node:StopMoving()
            attackTimer = 0
        end,
        function(dt) attackTimer = attackTimer + dt end,
        nil
    )

    -- Custom stunned state with self-transition.
    fsm:AddState("stunned",
        function()
            self.node:StopMoving()
            stunTimer = 0
        end,
        function(dt)
            stunTimer = stunTimer + dt
            if stunTimer >= STUN_DUR then
                fsm:SetState("patrol")
            end
        end,
        nil
    )

    -- Transitions (first-match wins each frame).
    fsm:AddTransition("idle", "patrol",
        function() return idleTimer >= IDLE_DUR end)

    fsm:AddTransition("patrol", "chase",
        function() return dist() < DETECT_RANGE end)

    fsm:AddTransition("chase", "attack",
        function() return dist() <= ATTACK_RANGE end)

    fsm:AddTransition("chase", "patrol",
        function() return dist() > LOSE_RANGE end)

    fsm:AddTransition("attack", "chase",
        function()
            return attackTimer >= ATTACK_DUR and dist() <= LOSE_RANGE
        end)

    fsm:AddTransition("attack", "patrol",
        function()
            return attackTimer >= ATTACK_DUR and dist() > LOSE_RANGE
        end)
end

function Update(dt)
    -- Tick the FSM — required every frame.
    fsm:Update(dt)
end

-- Called externally (e.g. from a trigger or game script) to stun the enemy.
function OnStun()
    if not fsm:IsInState("stunned") then
        fsm:SetState("stunned")
    end
end

Quick reference

Method Returns Notes
entity->AddBehaviorFSM() BehaviorFSM* Attaches the FSM component; idempotent (returns existing if called twice).
entity->GetBehaviorFSM() BehaviorFSM* Retrieves the component; nullptr if not yet added.
fsm->AddState(name, onEnter, onUpdate, onExit) — Custom state with three callbacks; first call sets the default state. Pass {} or nil for unused callbacks.
fsm->AddPatrolState(name, waypoints, speed=3) — Built-in NavMesh waypoint loop. Requires BuildNavMesh() to have been called first.
fsm->AddChaseState(name, target, speed=4, stopDist=1.5) — Built-in NavMesh target follow. Recomputes path automatically when target moves. Requires BuildNavMesh().
fsm->AddTransition(from, to, conditionFn) — std::function<bool()> condition. Transitions are evaluated in registration order; first match fires per frame.
fsm->SetState(name) — Immediate override: calls onExit then onEnter. Bypasses all transition conditions.
fsm->Update(dt) — Ticks the FSM: calls current state onUpdate(dt), then evaluates transitions. Call once per frame.
fsm->GetCurrentState() std::string Name of the active state. Empty string if no states have been registered.
fsm->IsInState(name) bool Shorthand for GetCurrentState() == name. Prefer this in conditions for clarity.

Common mistakes

  • Forgetting to call BuildNavMesh() before AddPatrolState / AddChaseState. These two built-in states route movement through the NavMesh. If the NavMesh has not been baked yet when the patrol or chase state is first entered, the enemy will stand still. Call BuildNavMesh() in Start() after all static level geometry exists, but before any patrol or chase state is registered.
  • Forgetting to call fsm->Update(dt) every frame. If you omit the Update call, the FSM never evaluates transitions and never calls onUpdate. The enemy will stay frozen in its initial state. Put fsm->Update(dt) at a consistent point in Game::Update() and never skip it.
  • Registering transitions in the wrong order. Transitions are checked in the order they were registered, and the first match fires. If you register chase → patrol before chase → attack, an enemy that is close enough to attack will retreat instead, because the patrol condition fires first. Always register more specific (higher-priority) transitions before broader ones.
  • Using SetState() where AddTransition() would be better. SetState() bypasses the FSM's transition graph, making it harder to follow the logic when debugging. Reserve it for external interrupts (damage, cutscene triggers, death). Distance thresholds, timer expirations, and animation events should all go through AddTransition() conditions.
  • Calling SetState() inside onEnter or onExit of the same state. Calling SetState("X") from within state X's own onEnter creates an infinite enter/exit loop. If you need to skip immediately to another state upon entry, use a zero-duration transition instead, or set a flag and handle it in the next Update.
  • Capturing entity pointers in transition lambdas after the entity is destroyed. If an enemy entity is destroyed while its FSM is still being updated, any lambda that captures enemy_ or this will dereference a dangling pointer. Always call enemy->SetActive(false) or DestroyEntity(enemy) only after the FSM update loop has finished, and null out the pointer immediately after.
  • Registering a state with the same name twice. Registering two states with the same name (e.g. calling AddState("patrol", ...) and then AddPatrolState("patrol", ...)) results in undefined behaviour. Every state name in a given FSM must be unique.
  • Expecting the first state's onEnter to be called at startup. The FSM starts already "in" the first registered state. Its onEnter callback is not called at setup time — only on transitions. If you need initialisation logic for the starting state, put it in Start() directly rather than relying on onEnter.

What you learned

In this part you built a fully functional AI enemy using simple-3d's BehaviorFSM system:

  • Attaching and retrieving the component: entity->AddBehaviorFSM() attaches a new FSM and returns its pointer; entity->GetBehaviorFSM() retrieves an existing one safely.
  • Custom states: fsm->AddState(name, onEnter, onUpdate, onExit) registers any bespoke behaviour with three optional callbacks. The first registered state becomes the default.
  • Built-in patrol: fsm->AddPatrolState(name, waypoints, speed) creates a NavMesh-backed waypoint loop automatically, routing around obstacles.
  • Built-in chase: fsm->AddChaseState(name, target, speed, stopDist) makes the enemy follow a moving target along the NavMesh, stopping within melee range.
  • Condition transitions: fsm->AddTransition(from, to, fn) wires the graph edges. Transitions are checked in registration order; the first matching condition fires. Putting high-priority rules first is critical for correct behaviour.
  • Manual overrides: fsm->SetState(name) performs an unconditional state change for external events like damage or scripted triggers, bypassing all transition rules.
  • Querying state: fsm->GetCurrentState() returns the active state name; fsm->IsInState(name) is a concise boolean check.
  • Ticking the FSM: fsm->Update(dt) must be called exactly once per frame to drive transitions and run the active state's update callback.
  • Four-state cycle: You assembled an Idle → Patrol → Chase → Attack loop with a Stunned interrupt, demonstrating how to layer states and combine built-in nav states with custom callback states in a single coherent FSM.
