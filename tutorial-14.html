How navigation meshes work in simple-3d

A navigation mesh (navmesh) is a simplified polygon surface that covers the walkable areas of your level. Pathfinding algorithms can route an agent across this surface efficiently because they only ever need to reason about polygons, not raw triangle soup.

simple-3d uses the industry-standard Recast library to bake the navmesh and Detour to query paths through it at runtime. The typical workflow has three steps:

  1. Place all static level geometry (ground, walls, crates) in the scene and assign them CollisionLayer::StaticGeometry with mass 0 rigid bodies so the engine can find them.
  2. Call BuildNavMesh() once in Start() after setup. Recast voxelises the geometry, erodes the walkable surface by the agent radius, and stores the resulting polygon mesh. For a normal-sized level this takes a fraction of a second.
  3. Every frame, call entity->MoveToward() or entity->FindPath() to navigate agents across the cached mesh. These calls are cheap because the heavy baking work is already done.

Dynamic obstacles

The navmesh baked in Start() is static. If you add or remove a wall at runtime (for example, a door that opens), call BuildNavMesh() again with the same parameters to rebuild it. Rebuilding is synchronous and safe to call at any time, though it will cause a brief stall on very large levels.

Static geometry: what the nav mesh sees

Recast only includes entities that have both a zero-mass rigid body and a collision shape assigned to CollisionLayer::StaticGeometry (layer 2). A decorative entity with no physics components will be ignored entirely; a dynamic entity with mass > 0 will also be ignored because it can move at any time.

The minimum required setup for any object you want to block navigation: 

Entity* wall = CreateEntity("Wall");
wall->SetPosition(Vector3(5.0f, 1.5f, 0.0f));
wall->SetScale(Vector3(1.0f, 3.0f, 8.0f));
wall->AddModel("Models/Box.mdl");

// Step 1: Mass 0 — static body. Never moves. Nav mesh can use it.
wall->AddRigidBody(0.0f);

// Step 2: Collision shape so physics (and Recast) know the wall's extent.
wall->AddBoxCollider(Vector3(1.0f, 3.0f, 8.0f));

// Step 3: Layer 2 — Recast collects only this layer during baking.
wall->SetCollisionLayer(CollisionLayer::StaticGeometry);

Any entity that does not follow this pattern (wrong layer, missing rigid body, or mass > 0) will be transparent to the nav mesh and agents will walk right through it.

Complete example

The demo below creates a 30×30 m arena with four wall obstacles, a player (controlled with the arrow keys) and a red enemy that automatically navigates around the walls to reach the player. Press C to switch the enemy between chasing the player and heading to the center of the arena. Press F to toggle a yellow debug visualisation of the current path waypoints. Press Escape to quit. 

// NavDemo.cpp — Part 14 of the simple-3d tutorial series
#include <Simple3D/Simple3D.h>
#include <string>
#include <vector>

using namespace Simple3D;

class NavDemo : public Game {
public:
    Entity*              player_   = nullptr;
    Entity*              enemy_    = nullptr;
    CharacterController* playerCC_ = nullptr;
    Label*               statusLbl_= nullptr;
    bool                 debugPath_= false;
    bool                 chaseMode_= true;

    // Fixed rally point: center of the arena, at ground level.
    Vector3 rallyPoint_ = Vector3(0.0f, 0.0f, 0.0f);

    // ── Helper: create one static obstacle ───────────────────────
    void MakeWall(const char* name, Vector3 pos, Vector3 size) {
        Entity* w = CreateEntity(name);
        w->SetPosition(pos);
        w->SetScale(size);
        w->AddModel("Models/Box.mdl");
        w->SetMaterialColor(Color(0.55f, 0.50f, 0.45f));
        w->AddRigidBody(0.0f);              // static — included in nav mesh bake
        w->AddBoxCollider(size);
        w->SetCollisionLayer(CollisionLayer::StaticGeometry);
    }

    void Start() override {
        SetWindowTitle("Part 14 — Navigation and Pathfinding");
        SetClearColor(Color(0.12f, 0.14f, 0.18f));

        // ── Camera: angled overhead view of the arena ─────────────
        Camera* cam = CreateCamera();
        cam->SetPosition(Vector3(0.0f, 22.0f, -18.0f));
        cam->LookAt(Vector3(0.0f, 0.0f, 3.0f));
        cam->SetFOV(50.0f);

        // ── Directional light ─────────────────────────────────────
        Entity* sun = CreateEntity("Sun");
        sun->SetRotation(Quaternion(50.0f, 20.0f, 0.0f));
        sun->AddDirectionalLight(Color::WHITE, 1.0f);

        // ════════════════════════════════════════════════════════
        // STEP 1: Place all static geometry BEFORE calling BuildNavMesh.
        // Recast only sees entities on CollisionLayer::StaticGeometry
        // with a zero-mass rigid body.
        // ════════════════════════════════════════════════════════

        // Ground plane — 30 m × 30 m, 0.5 m thick, top surface at y = +0.25
        Entity* ground = CreateEntity("Ground");
        ground->SetPosition(Vector3(0.0f, 0.0f, 0.0f));
        ground->SetScale(Vector3(30.0f, 0.5f, 30.0f));
        ground->AddModel("Models/Box.mdl");
        ground->SetMaterialColor(Color(0.28f, 0.48f, 0.28f));
        ground->AddRigidBody(0.0f);
        ground->AddBoxCollider(Vector3(30.0f, 0.5f, 30.0f));
        ground->SetCollisionLayer(CollisionLayer::StaticGeometry);

        // Four wall obstacles arranged to create a winding path through the arena.
        // Wall centers sit at y = 1.5 so the base rests on the ground (y = 0.25)
        // and the top reaches y = 2.75.
        MakeWall("WallA", Vector3(-5.0f, 1.5f,  2.0f), Vector3(8.0f, 2.5f, 1.0f));
        MakeWall("WallB", Vector3( 4.0f, 1.5f, -2.0f), Vector3(1.0f, 2.5f, 8.0f));
        MakeWall("WallC", Vector3(-2.0f, 1.5f, -5.0f), Vector3(6.0f, 2.5f, 1.0f));
        MakeWall("WallD", Vector3( 6.0f, 1.5f,  5.0f), Vector3(1.0f, 2.5f, 5.0f));

        // ── PLAYER — controlled by arrow keys ─────────────────────
        player_ = CreateEntity("Player");
        player_->SetPosition(Vector3(-8.0f, 1.15f, -8.0f));
        player_->SetScale(Vector3(0.7f, 1.8f, 0.7f));
        player_->AddModel("Models/Box.mdl");
        player_->SetMaterialColor(Color(0.2f, 0.55f, 0.9f));
        playerCC_ = player_->AddCharacterController(0.35f, 1.8f);
        playerCC_->SetGravity(22.0f);
        player_->SetCollisionLayer(CollisionLayer::Actor);
        player_->SetCollisionMask(MakeCollisionMask({CollisionLayer::StaticGeometry}));

        // ── ENEMY — auto-navigated by MoveToward each frame ───────
        // A dynamic RigidBody (mass > 0) is required. MoveToward drives
        // the body's velocity to follow the computed path.
        // A CharacterController also works; kinematic bodies use Translate().
        enemy_ = CreateEntity("Enemy");
        enemy_->SetPosition(Vector3(8.0f, 1.15f, 8.0f));
        enemy_->SetScale(Vector3(0.7f, 1.8f, 0.7f));
        enemy_->AddModel("Models/Box.mdl");
        enemy_->SetMaterialColor(Color(0.9f, 0.25f, 0.2f));
        enemy_->AddRigidBody(75.0f);         // dynamic — nav system sets velocity
        enemy_->AddCapsuleCollider(0.35f, 1.8f);
        enemy_->SetCollisionLayer(CollisionLayer::Actor);
        enemy_->SetCollisionMask(MakeCollisionMask({CollisionLayer::StaticGeometry}));

        // ════════════════════════════════════════════════════════
        // STEP 2: Build the nav mesh.
        // Parameters:
        //   0.4f  — agent radius in metres (clearance from obstacle edges)
        //   1.8f  — agent height in metres (minimum overhead clearance)
        //   0.3f  — voxel cell size in metres (resolution; smaller = more
        //            accurate corners but longer bake time)
        //
        // This call is synchronous and typically takes < 100 ms for a small
        // level. The result is cached in the scene until the next call.
        // ════════════════════════════════════════════════════════
        BuildNavMesh(0.4f, 1.8f, 0.3f);

        // ── HUD ───────────────────────────────────────────────────
        statusLbl_ = CreateLabel("Enemy: chasing player");
        statusLbl_->SetPosition(12, 12);
        statusLbl_->SetFontSize(18);
        statusLbl_->SetColor(Color(0.92f, 0.88f, 0.78f));

        Label* helpLbl = CreateLabel(
            "Arrow keys = move  |  C = chase / rally  |  F = debug path  |  ESC = quit");
        helpLbl->SetPosition(12, 40);
        helpLbl->SetFontSize(14);
        helpLbl->SetColor(Color(0.6f, 0.6f, 0.6f));

        // Enable debug renderer so DrawDebugSphere / DrawDebugLine work.
        SetDebugDrawEnabled(true);
    }

    void Update(float dt) override {
        if (IsKeyPressed(Key::Escape))
            Quit();

        // ── Player movement (arrow keys) ──────────────────────────
        Vector3 dir;
        if (IsKeyDown(Key::Up))    dir += Vector3( 0.0f, 0.0f,  1.0f);
        if (IsKeyDown(Key::Down))  dir += Vector3( 0.0f, 0.0f, -1.0f);
        if (IsKeyDown(Key::Left))  dir += Vector3(-1.0f, 0.0f,  0.0f);
        if (IsKeyDown(Key::Right)) dir += Vector3( 1.0f, 0.0f,  0.0f);
        if (dir.LengthSquared() > 0.01f)
            dir.Normalize();
        playerCC_->Move(dir * 6.0f, dt);

        // ── C — toggle between chasing the player and the rally point
        if (IsKeyPressed(Key::C)) {
            chaseMode_ = !chaseMode_;
            if (!chaseMode_) {
                // Cancel the current path before switching targets.
                enemy_->StopMoving();
            }
        }

        // ── F — toggle path debug visualisation
        if (IsKeyPressed(Key::F))
            debugPath_ = !debugPath_;

        // ════════════════════════════════════════════════════════
        // STEP 3: Navigate the enemy every frame with MoveToward.
        //
        // MoveToward computes a path to the target on the first call,
        // caches it, and then drives the entity's velocity along the path.
        // It recomputes automatically when the target moves more than 0.5 m.
        //
        // Return value:
        //   true  — currently navigating (has a valid path, not yet arrived)
        //   false — arrived at the destination, or no valid path exists
        // ════════════════════════════════════════════════════════
        bool moving;
        if (chaseMode_) {
            // Pass an Entity* — the nav system tracks the entity's position
            // and recomputes whenever the player moves significantly.
            moving = enemy_->MoveToward(player_, 4.0f, dt);
        } else {
            // Pass a Vector3 — navigate to a fixed world-space point.
            moving = enemy_->MoveToward(rallyPoint_, 3.0f, dt);
        }

        std::string modeStr = chaseMode_ ? "chasing player" : "going to center";
        statusLbl_->SetText(
            "Enemy: " + modeStr + (moving ? "" : "  [arrived]")
        );

        // ── Debug: draw the current path as yellow spheres + lines ──
        // FindPath queries the nav mesh without starting movement.
        // It returns the same waypoint list that MoveToward uses internally,
        // so the visualisation always matches the route being followed.
        if (debugPath_) {
            Vector3 target = chaseMode_
                ? player_->GetPosition()
                : rallyPoint_;

            std::vector<Vector3> path = enemy_->FindPath(target);

            Vector3 prev = enemy_->GetPosition();
            for (const Vector3& wp : path) {
                DrawDebugSphere(wp, 0.18f, Color::YELLOW);
                DrawDebugLine(prev, wp, Color::YELLOW);
                prev = wp;
            }
        }
    }
};

int main() {
    return Simple3D::CreateGame<NavDemo>()->Run();
}

Step-by-step explanation

BuildNavMesh(agentRadius, agentHeight, cellSize)

BuildNavMesh is a method on the Game class (not on any single entity). It scans the entire scene for static geometry, voxelises it into a grid of the given cell size, erodes the walkable surface inward by the agent radius to ensure agents can pass without clipping obstacle edges, and stores the resulting polygon navmesh in memory.

Call it once in Start(), after every static entity you want included has been created. The call is synchronous — it blocks the main thread for a moment, which is acceptable in a loading/startup context but should not be called inside Update() unless you explicitly need a runtime rebuild.

BuildNavMesh(float agentRadius = 0.4f, float agentHeight = 1.8f, float cellSize = 0.3f);

Agent radius (SetNavMeshAgentRadius) — the first parameter

The agent radius (first parameter to BuildNavMesh) is the half-width of your navigating character in metres. Recast erodes the walkable surface by this amount from every obstacle edge, so agents computed by Detour always have at least this much clearance.

A value of 0.4f suits a roughly human-sized character. If you make it too small, agents may pathfind through gaps that they would actually clip in 3D space. If you make it too large, agents may refuse to walk through narrow corridors that would realistically fit them.

Match the agent radius to the collision capsule

Set the agent radius equal to (or slightly greater than) the radius of the capsule collider you attach to your enemies. In the example above both the nav mesh agent and the enemy's AddCapsuleCollider use 0.35–0.4 m. This prevents the nav system from routing the enemy through openings that the physics capsule would then block.

Cell size (SetNavMeshCellSize) — the third parameter

The cell size (third parameter) is the edge length of each voxel in the Recast grid. Smaller values produce a more accurate navmesh that tracks the true shape of obstacles more closely, but they also increase bake time and memory use quadratically.

The default 0.3f (30 cm per voxel) is a good starting point for levels with obstacles at least 1 m wide. For very small levels or tight corridors, try 0.15f. For large open outdoor levels where exact obstacle fitting is less important, 0.5f or even 1.0f is usually fine and bakes in milliseconds.

Parameter Smaller value Larger value Typical range
agentRadius Agents walk closer to edges; may clip narrow gaps Agents avoid more clearance; narrow gaps blocked 0.2–0.6 m
agentHeight Agents pass under lower ceilings Agents need more overhead clearance 1.5–2.2 m
cellSize More accurate corners; slower bake Blockier navmesh; faster bake 0.1–0.5 m

entity->MoveToward(target, speed, dt)

MoveToward is the primary high-level navigation call. On the first call it asks the Detour library to compute a path from the entity's current position to target and caches the resulting waypoint list. On subsequent calls it drives the entity's velocity along the cached path at speed metres per second, automatically advancing to the next waypoint when the current one is reached. When the final waypoint is reached the entity stops and the call returns false.

Path recomputation: if the target is an Entity*, the system compares the target's current position to the position recorded when the last path was computed. If the difference is more than 0.5 m it recomputes the path. This makes the call suitable for chasing a moving player without incurring a full Detour query every frame.

// Chase an entity bool moving = entity->MoveToward(Entity* target, float speed, float dt); // Navigate to a fixed world position bool moving = entity->MoveToward(Vector3 targetPos, float speed, float dt);

Return value: true while the entity is actively moving along a path, false when it has arrived at the destination or when no valid path could be found (e.g. the target is surrounded by walls with no reachable route). Always check the return value if you need to trigger logic when the agent arrives.

The entity must have either a dynamic RigidBody (mass > 0) or a CharacterController so that MoveToward has a velocity field to write into. Entities with no physics component are moved via Translate().

entity->FindPath(target) → std::vector<Vector3>

FindPath queries Detour for the shortest path from the entity's current position to target and returns the full waypoint list as a std::vector<Vector3>. It does not start movement. Use it when you need to inspect the route (for AI decisions, length calculation, or debug visualisation) before or instead of following it.

// Returns waypoints including the destination; empty if no path exists std::vector<Vector3> path = entity->FindPath(Vector3 targetPos); std::vector<Vector3> path = entity->FindPath(Entity* target);

The first element of the returned vector is the first turn waypoint on the path; the last element is the destination. If the target is directly reachable without any turns, the vector contains only the destination. An empty vector means the target is not reachable from the entity's current position (for example, it is outside the navmesh boundary, or completely surrounded by obstacles).

Because FindPath calls Detour directly it costs a tiny amount of CPU each frame if called every Update(). For continuous chasing, prefer MoveToward which caches the path internally. Use FindPath for one-shot queries, debug drawing, or periodic AI checks.

entity->StopMoving()

Clears the entity's cached path and sets its horizontal velocity to zero. Call it whenever you want to interrupt navigation early: when the enemy enters an idle state, when the player is caught, or when switching to a new target. Without calling StopMoving(), the entity would continue following the old cached path for another frame before MoveToward computes a new one.

entity->StopMoving(); // clears path, zeroes horizontal velocity

Manual path following with FindPath

Sometimes you need direct control over every step of navigation — for example, an AI that inspects the path length before deciding whether to chase, or a preview arrow drawn before the player confirms a movement order. FindPath covers both cases: 

// In Update(dt) — example: only chase if the path is shorter than 15 m
std::vector<Vector3> path = enemy_->FindPath(player_);

if (path.empty()) {
    // No route to the player — stay put
    enemy_->StopMoving();
    return;
}

// Estimate total path length
float totalLength = 0.0f;
Vector3 prev = enemy_->GetPosition();
for (const Vector3& wp : path) {
    totalLength += (wp - prev).Length();
    prev = wp;
}

if (totalLength < 15.0f) {
    // Player is reachable and close enough — chase
    enemy_->MoveToward(player_, 4.0f, dt);
} else {
    // Player is too far away — patrol
    enemy_->StopMoving();
}

You can also follow the waypoints yourself if you need custom steering logic. Advance through the vector by tracking the current waypoint index and checking whether the entity is close enough to move to the next one: 

// Manual step-by-step path following
static std::vector<Vector3> s_path;
static int s_wpIndex = 0;

// Re-query once when needed (e.g. on target change)
if (recomputeNeeded) {
    s_path    = enemy_->FindPath(targetPos);
    s_wpIndex = 0;
}

if (s_wpIndex < (int)s_path.size()) {
    Vector3 toWp = s_path[s_wpIndex] - enemy_->GetPosition();
    toWp.y_ = 0.0f;  // ignore vertical component

    if (toWp.LengthSquared() < 0.25f) {
        ++s_wpIndex;   // close enough — advance
    } else {
        toWp.Normalize();
        enemy_->SetLinearVelocity(toWp * 4.0f);
    }
}

This is exactly the pattern that MoveToward implements internally. In most games the built-in method is simpler to use and handles edge cases such as path replanning on target drift.

Lua equivalent

All navigation functions are exposed to Lua with the same names. BuildNavMesh() is a global function; FindPath, MoveToward, and StopMoving are methods on entity objects. The returned path from FindPath is a Lua table (1-indexed array) of Vector3 values. 

-- NavDemo.lua — Lua equivalent of the navigation demo
-- Attach to a "Game" script entity, or use as a standalone Lua game.

local player, enemy, playerCC, statusLbl
local debugPath  = false
local chaseMode  = true
local rallyPoint = Vector3(0, 0, 0)

    w:SetScale(size)

    w:AddRigidBody(0.0)           -- static body: included in nav mesh
    w:AddBoxCollider(size)
    w:SetCollisionLayer(CollisionLayer.StaticGeometry)
end

    SetClearColor(Color(0.12, 0.14, 0.18))

    local cam = CreateCamera()
    cam:SetPosition(Vector3(0, 22, -18))
    cam:LookAt(Vector3(0, 0, 3))
    cam:SetFOV(50)

    local sun = CreateEntity("Sun")
    sun:SetRotation(Quaternion(50, 20, 0))
    sun:AddDirectionalLight(Color.WHITE, 1.0)

    -- Ground (static — baked into nav mesh)
    local ground = CreateEntity("Ground")
    ground:SetPosition(Vector3(0, 0, 0))
    ground:SetScale(Vector3(30, 0.5, 30))
    ground:AddModel("Models/Box.mdl")
    ground:SetMaterialColor(Color(0.28, 0.48, 0.28))
    ground:AddRigidBody(0.0)
    ground:AddBoxCollider(Vector3(30, 0.5, 30))
    ground:SetCollisionLayer(CollisionLayer.StaticGeometry)

    -- Obstacles
    MakeWall("WallA", Vector3(-5, 1.5,  2), Vector3(8, 2.5, 1))
    MakeWall("WallB", Vector3( 4, 1.5, -2), Vector3(1, 2.5, 8))
    MakeWall("WallC", Vector3(-2, 1.5, -5), Vector3(6, 2.5, 1))
    MakeWall("WallD", Vector3( 6, 1.5,  5), Vector3(1, 2.5, 5))

    -- Player
    player = CreateEntity("Player")
    player:SetPosition(Vector3(-8, 1.15, -8))
    player:SetScale(Vector3(0.7, 1.8, 0.7))
    player:AddModel("Models/Box.mdl")
    player:SetMaterialColor(Color(0.2, 0.55, 0.9))
    playerCC = player:AddCharacterController(0.35, 1.8)
    playerCC:SetGravity(22.0)
    player:SetCollisionLayer(CollisionLayer.Actor)

    -- Enemy (dynamic rigid body — MoveToward writes its velocity)
    enemy = CreateEntity("Enemy")
    enemy:SetPosition(Vector3(8, 1.15, 8))
    enemy:SetScale(Vector3(0.7, 1.8, 0.7))
    enemy:AddModel("Models/Box.mdl")
    enemy:SetMaterialColor(Color(0.9, 0.25, 0.2))
    enemy:AddRigidBody(75.0)
    enemy:AddCapsuleCollider(0.35, 1.8)
    enemy:SetCollisionLayer(CollisionLayer.Actor)
    enemy:SetCollisionMask(MakeCollisionMask({CollisionLayer.StaticGeometry}))

    -- Build nav mesh AFTER all static geometry is placed
    BuildNavMesh(0.4, 1.8, 0.3)

    statusLbl = CreateLabel("Enemy: chasing player")
    statusLbl:SetPosition(12, 12)
    statusLbl:SetFontSize(18)
    statusLbl:SetColor(Color(0.92, 0.88, 0.78))

    SetDebugDrawEnabled(true)
end

function Update(dt)
    if IsKeyPressed(Key.Escape) then Quit() end

    -- Player movement
    local dir = Vector3(0, 0, 0)
    if IsKeyDown(Key.Up)    then dir = dir + Vector3(0, 0,  1) end
    if IsKeyDown(Key.Down)  then dir = dir + Vector3(0, 0, -1) end
    if IsKeyDown(Key.Left)  then dir = dir + Vector3(-1, 0, 0) end
    if IsKeyDown(Key.Right) then dir = dir + Vector3( 1, 0, 0) end
    if dir:LengthSquared() > 0.01 then dir:Normalize() end
    playerCC:Move(dir * 6.0, dt)

    -- Toggle chase / rally
    if IsKeyPressed(Key.C) then
        chaseMode = not chaseMode
        if not chaseMode then
            enemy:StopMoving()
        end
    end
    if IsKeyPressed(Key.F) then debugPath = not debugPath end

    -- Enemy navigation
    local moving
    if chaseMode then
        moving = enemy:MoveToward(player, 4.0, dt)
    else
        moving = enemy:MoveToward(rallyPoint, 3.0, dt)
    end

    local modeStr = chaseMode and "chasing player" or "going to center"
    statusLbl:SetText("Enemy: " .. modeStr .. (moving and "" or "  [arrived]"))

    -- Debug path (FindPath returns a Lua table of Vector3)
    if debugPath then
        local target = chaseMode and player:GetPosition() or rallyPoint
        local path = enemy:FindPath(target)
        local prev = enemy:GetPosition()
        for i = 1, #path do
            DrawDebugSphere(path[i], 0.18, Color.YELLOW)
            DrawDebugLine(prev, path[i], Color.YELLOW)
            prev = path[i]
        end
    end
end

Lua note

BuildNavMesh() is a global function in Lua — no colon syntax. Entity navigation methods (MoveToward, FindPath, StopMoving) use the colon (enemy:MoveToward(player, 4.0, dt)). FindPath returns a 1-indexed Lua table, so iterate with for i = 1, #path do. The table is empty (length zero) when no path exists.

Navigation API quick reference

Call Where Notes
BuildNavMesh(radius, height, cellSize) Start() Bake navmesh from all StaticGeometry entities. Synchronous. Call again if static geometry changes.
entity->MoveToward(target, speed, dt) Update(dt) Auto-navigate every frame. Returns true while moving, false on arrival or no path.
entity->FindPath(target) Anywhere Returns std::vector<Vector3> waypoints. Empty if unreachable. Does not start movement.
entity->StopMoving() Anywhere Clears cached path, zeroes horizontal velocity. Call when switching targets or entering idle state.

Common mistakes

Common mistakes

  • Calling BuildNavMesh() before placing all static geometry. Any wall or obstacle created after BuildNavMesh() is invisible to the navmesh. Agents will walk through it. Always finalise your level layout first, then call BuildNavMesh() as the last setup step in Start().
  • Forgetting to set CollisionLayer::StaticGeometry on obstacles. Recast only collects entities on layer 2 (StaticGeometry). An obstacle on a different layer (or with no layer set at all) will be transparent to the nav mesh even though it physically blocks the player via normal Bullet physics. The result is an enemy that walks straight through walls.
  • Using a dynamic (mass > 0) rigid body on a wall. Dynamic bodies can move, so Recast does not include them in the baked navmesh. If a wall has AddRigidBody(5.0f) instead of AddRigidBody(0.0f), it will be excluded and agents will ignore it during pathfinding.
  • Calling MoveToward without a rigid body or CharacterController. MoveToward drives the entity's velocity or translates it. An entity with no physics component will be moved via Translate() which ignores gravity and other forces. Make sure navigated entities have either a dynamic rigid body or a CharacterController attached before calling MoveToward.
  • Switching targets without calling StopMoving() first. If you change chaseMode_ and immediately call MoveToward with a new target, the enemy will follow the old cached path for one more frame before recomputing. On fast-moving targets or tight level layouts this can cause a visible direction hiccup. Call StopMoving() before switching targets to ensure a clean transition.
  • Treating a non-empty FindPath result as guaranteed reachability. Detour snaps the start and end positions to the nearest navmesh polygon before computing the path. If both positions snap to the same polygon, the returned path may be very short even if the target is behind a wall in 3D space. Always verify the path length or use physics line-of-sight checks for game logic decisions.
  • Setting cellSize too large for narrow corridors. If your corridors are only 1.5 m wide and you set cellSize = 1.0f, Recast may fail to generate any walkable polygon through the corridor at all. As a rule, your corridor width should be at least 3× the cell size to guarantee reliable coverage.
  • Calling BuildNavMesh() inside Update(). Recast baking is synchronous. Calling it every frame will stall the game. Call it once in Start() or explicitly when geometry changes (e.g. after a door opens).

What you learned

By the end of this part you have a complete working navigation system and understand how each layer of it works:

  • simple-3d uses Recast to bake a navmesh from static scene geometry and Detour to query paths through it at runtime.
  • Every entity you want the navmesh to block must have a zero-mass rigid body, a collision shape, and SetCollisionLayer(CollisionLayer::StaticGeometry).
  • BuildNavMesh(agentRadius, agentHeight, cellSize) must be called once in Start() after all static geometry is in place. Call it again only when static geometry changes at runtime.
  • The agent radius (first parameter) controls how much clearance agents keep from obstacle edges; match it to your enemy's capsule collider radius.
  • The cell size (third parameter) trades baking accuracy for speed; smaller values are needed for tight corridors, larger values are fine for open areas.
  • entity->MoveToward(target, speed, dt) is called every frame and drives the entity's velocity along the cached Detour path. It caches the path and recomputes automatically when the target drifts by more than 0.5 m. Returns true while moving, false on arrival.
  • entity->FindPath(target) returns the waypoint list as a std::vector<Vector3> without starting movement — useful for AI decisions, length checks, and debug visualisation.
  • entity->StopMoving() cancels the current path and zeroes horizontal velocity. Call it before switching navigation targets.
