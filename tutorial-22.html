  • How to express a child entity's intended offset using SetLocalPosition() and SetLocalRotation() rather than world-space setters

  • The weapon socket pattern — attaching an empty transform entity to the character mesh, then parenting the weapon model beneath it so equipping and unequipping require only reparenting
  • How SetActive(false) propagates through a subtree, suspending the entire branch at once

Overview

Every entity in simple-3d is a node in a scene graph — an implicit tree rooted at the invisible scene root. When you call CreateEntity("Player") you add a node directly beneath that root. When you subsequently call player->CreateChild("PlayerVisual") you add another node one level deeper, making PlayerVisual a child of Player.

The most important consequence of this tree is transform inheritance. A child entity's position, rotation, and scale are always expressed relative to its parent in local space. When the engine computes where to draw the child on screen it walks up the tree, composing transforms, to arrive at a final world-space position. This means that moving the parent entity automatically moves every descendant — you never need to write per-frame code to keep a weapon glued to a hand or a camera target glued to a character's mid-body.

This tutorial builds a three-level hierarchy that demonstrates all of these ideas in a single running scene:

  • Player — root entity with a physics character controller
  • PlayerVisual — child of Player; carries the renderable mesh, offset 0.8 m above the capsule centre
  • WeaponSocket — child of PlayerVisual; an empty transform entity positioned at the right-hand grip point
  • Sword — child of WeaponSocket; the actual weapon model

When you walk the player around with WASD, the visual mesh, the socket, and the sword all follow automatically. Pressing F hides the entire weapon subtree with a single SetActive(false) call. Pressing Q/E raises and lowers the socket using TranslateLocal(), which moves in the socket's own rotated coordinate frame — a clean demonstration of local versus world translation.

Complete source

The listing below is the complete, self-contained program for this tutorial. Read through it once in full, then follow the step-by-step explanation that breaks each new concept down in detail. 

// main.cpp — Part 22: Entity Hierarchy
#include <Simple3D/Simple3D.h>
#include <string>
#include <vector>
#include <sstream>

using namespace Simple3D;

// ─────────────────────────────────────────────────────────────────────────────
// HierarchyDemo — demonstrates parent/child entity relationships, local-space
// transforms, and the weapon socket pattern used in real action games.
// ─────────────────────────────────────────────────────────────────────────────
class HierarchyDemo : public Game {
public:

    void Start() override {
        SetWindowTitle("Entity Hierarchy — simple-3d Part 22");
        SetClearColor(Color(0.12f, 0.15f, 0.20f, 1.0f));

        // ── Directional light ────────────────────────────────────────────────
        Entity* sun = CreateEntity("Sun");
        sun->AddDirectionalLight(Color(1.0f, 0.93f, 0.82f), 1.2f);
        sun->SetRotation(Quaternion(50.0f, 30.0f, 0.0f));

        // ── Ground plane (static geometry) ──────────────────────────────────
        Entity* ground = CreateEntity("Ground");
        ground->AddModel("Models/Plane.mdl");
        ground->SetScale(12.0f);
        ground->AddBoxCollider(Vector3(24.0f, 0.2f, 24.0f));
        ground->SetCollisionLayer(CollisionLayer::StaticGeometry);

        // ── Level 1: Player body (the physics root) ──────────────────────────
        //
        // The player entity is a ROOT entity — it lives directly under the
        // scene root.  It carries the CharacterController for physics-driven
        // movement and is the anchor point for the entire player hierarchy.
        //
        // Its position represents the BOTTOM of the capsule; we start at y=1
        // so the capsule (1.8 m tall) straddles the ground plane cleanly.
        player_ = CreateEntity("Player");
        player_->SetPosition(0.0f, 1.0f, 0.0f);

        cc_ = player_->AddCharacterController(0.35f, 1.8f);
        cc_->SetGravity(20.0f);

        player_->SetCollisionLayer(CollisionLayer::Actor);
        player_->SetCollisionMask(MakeCollisionMask({
            CollisionLayer::StaticGeometry
        }));

        // ── Level 2: PlayerVisual child ──────────────────────────────────────
        //
        // CreateChild() attaches a new entity directly beneath player_ in the
        // scene graph.  The child is owned by the engine — never delete it.
        //
        // We set the LOCAL position to (0, 0.8, 0) so the mesh sits 0.8 m
        // above the capsule centre, placing the character model correctly on
        // top of the physics body rather than at the feet.
        visual_ = player_->CreateChild("PlayerVisual");
        visual_->AddModel("Models/Character.mdl");
        visual_->SetLocalPosition(0.0f, 0.8f, 0.0f);  // LOCAL offset from parent

        // ── Level 3: WeaponSocket grandchild ─────────────────────────────────
        //
        // An EMPTY transform entity acts as the attachment point for the weapon.
        // Its local position is tuned to match the character's right-hand grip.
        // Tilting it by -20 degrees on Z makes the sword angle outward naturally.
        //
        // This "socket" pattern decouples the weapon from the mesh skeleton.
        // To swap weapons, reparent a different weapon model under this socket.
        weaponSocket_ = visual_->CreateChild("WeaponSocket");
        weaponSocket_->SetLocalPosition(0.45f, 0.55f, 0.25f);
        weaponSocket_->SetLocalRotation(Quaternion(0.0f, 0.0f, -20.0f));

        // ── Level 4: Sword (child of the socket) ─────────────────────────────
        //
        // The actual weapon model hangs beneath WeaponSocket.  Its own local
        // offset positions the blade correctly relative to the grip point.
        sword_ = weaponSocket_->CreateChild("Sword");
        sword_->AddModel("Models/Sword.mdl");
        sword_->SetLocalPosition(0.0f, 0.35f, 0.0f);
        sword_->SetScale(0.75f);                   // scale propagates to children

        // ── Camera orbiting the visual child ─────────────────────────────────
        //
        // We orbit the VISUAL child, not the physics body.  This centres the
        // camera at mid-body (0.8 m above feet) for a nicer third-person view.
        cam_ = CreateCamera();
        cam_->SetOrbitMode(visual_, 6.0f);
        cam_->SetOrbitAngles(0.0f, 15.0f);
        cam_->SetOrbitPitchLimits(-10.0f, 60.0f);
        cam_->SetSmoothFollow(true, 6.0f);

        // ── HUD information label ────────────────────────────────────────────
        label_ = CreateLabel("");
        label_->SetPosition(16, 16);
        label_->SetFontSize(15);
        label_->SetColor(Color::WHITE);

        // Print hierarchy statistics to the console at startup
        PrintHierarchyInfo();
    }

    void Update(float dt) override {
        if (IsKeyPressed(Key::Escape)) Quit();

        // ── Player movement (WASD) ───────────────────────────────────────────
        Vector3 moveDir(0.0f, 0.0f, 0.0f);
        if (IsKeyDown(Key::W)) moveDir.z_ += 1.0f;
        if (IsKeyDown(Key::S)) moveDir.z_ -= 1.0f;
        if (IsKeyDown(Key::A)) moveDir.x_ -= 1.0f;
        if (IsKeyDown(Key::D)) moveDir.x_ += 1.0f;
        cc_->Move(moveDir * 5.0f, dt);

        if (IsKeyPressed(Key::Space)) cc_->TryJump(10.0f);

        // ── F: toggle sword visibility ───────────────────────────────────────
        // SetActive(false) on a parent propagates to all children — the socket
        // AND the sword both disappear with a single call on weaponSocket_.
        if (IsKeyPressed(Key::F)) {
            swordEquipped_ = !swordEquipped_;
            weaponSocket_->SetActive(swordEquipped_);
        }

        // ── Q / E: raise / lower the weapon socket using TranslateLocal ──────
        //
        // TranslateLocal() moves the entity along ITS OWN rotated axes.
        // Because weaponSocket_ is rotated -20 degrees on Z, "up" in local
        // space is slightly tilted relative to world Y.  Compare this with
        // TranslateWorld() below, which always uses the fixed world axes.
        if (IsKeyDown(Key::Q))
            weaponSocket_->TranslateLocal(Vector3(0.0f,  0.8f * dt, 0.0f));
        if (IsKeyDown(Key::E))
            weaponSocket_->TranslateLocal(Vector3(0.0f, -0.8f * dt, 0.0f));

        // ── G / H: push socket forward / back using TranslateWorld ───────────
        //
        // TranslateWorld() moves in WORLD axes regardless of the entity's own
        // rotation.  World +Z always points in the same scene direction even
        // if weaponSocket_ is rotated.
        if (IsKeyDown(Key::G))
            weaponSocket_->TranslateWorld(Vector3(0.0f, 0.0f,  0.8f * dt));
        if (IsKeyDown(Key::H))
            weaponSocket_->TranslateWorld(Vector3(0.0f, 0.0f, -0.8f * dt));

        // ── R: reset socket to default local position ────────────────────────
        if (IsKeyPressed(Key::R)) {
            weaponSocket_->SetLocalPosition(0.45f, 0.55f, 0.25f);
            weaponSocket_->SetLocalRotation(Quaternion(0.0f, 0.0f, -20.0f));
        }

        // ── Update HUD ────────────────────────────────────────────────────────
        UpdateLabel();
    }

private:
    // ── Scene entities ───────────────────────────────────────────────────────
    Entity*              player_       = nullptr;   // Level 1 — physics root
    Entity*              visual_       = nullptr;   // Level 2 — renderable mesh
    Entity*              weaponSocket_ = nullptr;   // Level 3 — attachment point
    Entity*              sword_        = nullptr;   // Level 4 — weapon model
    Camera*              cam_          = nullptr;
    Label*               label_        = nullptr;
    CharacterController* cc_           = nullptr;
    bool                 swordEquipped_ = true;

    // ── PrintHierarchyInfo ───────────────────────────────────────────────────
    // Demonstrates GetChildCount(), GetChild(index), GetChildren(), and
    // GetParent() by walking parts of the hierarchy at startup.
    void PrintHierarchyInfo() {
        // GetChildCount() — number of direct (first-level) children
        int playerChildren = player_->GetChildCount();   // → 1  (visual_)
        int visualChildren = visual_->GetChildCount();   // → 1  (weaponSocket_)
        int socketChildren = weaponSocket_->GetChildCount();  // → 1  (sword_)
        int swordChildren  = sword_->GetChildCount();    // → 0  (leaf node)

        // GetChild(index) — access by zero-based position; returns nullptr if out of range
        Entity* firstChild = player_->GetChild(0);   // → visual_
        Entity* noChild    = player_->GetChild(5);   // → nullptr (out of range)

        // GetChildren() — returns a vector of ALL direct children
        std::vector<Entity*> playerKids = player_->GetChildren();
        // playerKids.size() == 1;  playerKids[0] == visual_

        // GetParent() — navigate upward; returns nullptr at scene root
        Entity* visualParent  = visual_->GetParent();       // → player_
        Entity* socketParent  = weaponSocket_->GetParent();  // → visual_
        Entity* playerParent  = player_->GetParent();       // → nullptr (root entity)

        // Iterate all of visual_'s direct children and shrink them slightly
        for (Entity* child : visual_->GetChildren()) {
            child->SetScale(0.9f);   // only weaponSocket_ is affected here
        }

        // Silence unused-variable warnings in minimal builds
        (void)playerChildren; (void)visualChildren;
        (void)socketChildren; (void)swordChildren;
        (void)firstChild;     (void)noChild;
        (void)playerKids;
        (void)visualParent;   (void)socketParent; (void)playerParent;
    }

    // ── UpdateLabel ──────────────────────────────────────────────────────────
    // Rebuilds the HUD each frame to show live hierarchy state.
    void UpdateLabel() {
        Vector3 playerPos  = player_->GetPosition();         // world-space
        Vector3 socketLocal = weaponSocket_->GetLocalPosition(); // parent-relative
        Vector3 socketWorld = weaponSocket_->GetPosition();    // absolute world

        std::ostringstream ss;
        ss.precision(2);
        ss << std::fixed;
        ss << "Player world:  ("
           << playerPos.x_ << ", " << playerPos.y_ << ", " << playerPos.z_ << ")\n"
           << "Socket local:  ("
           << socketLocal.x_ << ", " << socketLocal.y_ << ", " << socketLocal.z_ << ")\n"
           << "Socket world:  ("
           << socketWorld.x_ << ", " << socketWorld.y_ << ", " << socketWorld.z_ << ")\n"
           << "Player children: " << player_->GetChildCount() << "\n"
           << "Sword equipped: " << (swordEquipped_ ? "yes" : "no") << "\n\n"
           << "WASD=move  Space=jump  F=toggle sword\n"
           << "Q/E=socket up/down (local)  G/H=forward/back (world)  R=reset";

        label_->SetText(ss.str());
    }
};

int main(int, char**) {
    auto game = Simple3D::CreateGame<HierarchyDemo>();
    game->SetResourcePrefixPaths("/path/to/u3d/bin");
    return game->Run();
}

Step-by-step explanation

1. The scene graph — entities as a tree

Before diving into individual API calls it helps to picture the tree this demo builds: 

Scene root (invisible)
 └─ Player              ← CreateEntity("Player")          — Level 1
     └─ PlayerVisual    ← player_→CreateChild("...")      — Level 2
         └─ WeaponSocket← visual_→CreateChild("...")      — Level 3
             └─ Sword   ← weaponSocket_→CreateChild("...") — Level 4

Every entity you create via CreateEntity() is a direct child of the scene root — a root entity. Every entity you create via parent->CreateChild() is one level deeper, owned by that parent. The engine owns the lifetime of both kinds — never call delete on any entity pointer.

The key invariant of the scene graph: a child entity's world-space transform is the result of composing its local transform with its parent's world-space transform, recursively up to the root. This composition happens automatically every frame before rendering. You never write any sync code.

2. CreateChild(name)

visual_ = player_->CreateChild("PlayerVisual");

CreateChild(name) allocates a new entity, immediately inserts it as a direct child of the calling entity in the scene graph, and returns a raw non-owning pointer. The returned entity is just as capable as any entity created through CreateEntity() — you can attach models, physics, scripts, lights, and further children to it.

The name argument is used for debugging and can be retrieved with entity->GetName(). Names are not required to be unique, but giving each node a descriptive unique name makes hierarchy dumps much easier to read during development.

You can chain calls to build arbitrarily deep trees: 

Entity* body   = CreateEntity("Robot");
Entity* torso  = body->CreateChild("Torso");
Entity* lArm   = torso->CreateChild("LeftArm");
Entity* lHand  = lArm->CreateChild("LeftHand");
Entity* rArm   = torso->CreateChild("RightArm");  // sibling of lArm

3. Local space versus world space

This is the single most important concept in this tutorial. Every entity in the hierarchy has two coordinate frames:

FrameMethodsMeaning
World space SetPosition(), GetPosition(), SetRotation(), GetRotation() Absolute coordinates relative to the scene origin; unaffected by where the parent is
Local space SetLocalPosition(), GetLocalPosition(), SetLocalRotation(), GetLocalRotation() Coordinates relative to the parent entity's own origin and axes

For root entities (direct children of the scene root) local space and world space are identical — there is no parent transform to compose with. For child entities the two frames differ whenever the parent has a non-zero position or rotation.

The golden rule: always use local-space setters to express a child's intended offset. If you call visual_->SetPosition(0, 0.8f, 0) in world space, the engine immediately overwrites that world position with the result of composing visual_'s local transform with the parent's new world transform on the very next frame. The apparent position will jump. Use SetLocalPosition() instead to declare the permanent offset relative to the parent: 

// Wrong — world position gets overridden by the parent transform every frame
visual_->SetPosition(0.0f, 0.8f, 0.0f);   // do NOT do this on a child entity

// Correct — the local offset is fixed relative to the parent; never drifts
visual_->SetLocalPosition(0.0f, 0.8f, 0.0f);

Reading GetPosition() on a child still gives a useful result — it returns the world-space position the engine computed after composing the full chain of parent transforms. The HUD in this tutorial prints both GetLocalPosition() and GetPosition() for the weapon socket so you can observe them changing independently as you move the player around.

Scale also composes through the hierarchy, but there is only one SetScale() method. For a child entity, SetScale(0.5f) sets the child's scale in its own local frame, and the final world-space scale is the product of all ancestor scales. In the demo, calling sword_->SetScale(0.75f) makes the sword three-quarters of the weapon socket's natural size.

4. SetLocalPosition() and SetLocalRotation()

// Offset the visual mesh 0.8 m above the capsule centre
visual_->SetLocalPosition(0.0f, 0.8f, 0.0f);

// Position weapon socket at the character's right-hand grip point
weaponSocket_->SetLocalPosition(0.45f, 0.55f, 0.25f);

// Tilt the socket so the sword angles outward naturally
weaponSocket_->SetLocalRotation(Quaternion(0.0f, 0.0f, -20.0f));

SetLocalPosition() accepts either three floats or a Vector3. The coordinates are in the parent's local space — so (0.45, 0.55, 0.25) means 0.45 m to the right, 0.55 m up, and 0.25 m forward relative to the visual child's origin and facing direction, not relative to the world.

SetLocalRotation() takes a Quaternion, most conveniently constructed as Quaternion(pitch, yaw, roll) in degrees. The rotation is applied relative to the parent's rotation. If the player rotates to face a different direction, the weapon socket's local rotation (0, 0, -20) will continue to tilt the sword 20 degrees inward regardless of which direction the player faces — the local frame turns with the parent automatically.

You can read local values back with GetLocalPosition() and GetLocalRotation(), which return the same values you set (unmodified by any parent transform). Use these when you need to save or compare an entity's offset independent of where its parent currently happens to be.

5. Transform propagation in action

Consider the chain of transforms when the player is at world position (3, 1, 5) and facing along the -Z axis: 

Player     world pos: (3.00, 1.00, 5.00)   — set directly via SetPosition()

PlayerVisual
  local pos: (0.00, 0.80, 0.00)            — set via SetLocalPosition()
  world pos: (3.00, 1.80, 5.00)            — parent pos + rotated local offset

WeaponSocket
  local pos: (0.45, 0.55, 0.25)            — set via SetLocalPosition()
  world pos: ≈ (2.55, 2.35, 4.75)         — composed through two parent transforms

Sword
  local pos: (0.00, 0.35, 0.00)            — set via SetLocalPosition()
  world pos: ≈ (2.55, 2.70, 4.75)         — composed through three parent transforms

Every time the player moves, all four world positions are recomputed automatically by the engine. If you press WASD in the demo and watch the HUD, the socket's world position tracks the player's position exactly — no per-frame update code was written for that.

Rotation composes the same way. If the player turns 90 degrees to the right, the visual child turns with it, which rotates the socket's local frame, which in turn changes where the sword sits in world space — all without a single line of sync code.

6. GetParent()

Entity* parent = visual_->GetParent();       // → player_
Entity* root   = player_->GetParent();       // → nullptr (root entity)

GetParent() returns the direct parent entity in the scene graph, or nullptr if the entity is a root entity (parented directly to the scene root via CreateEntity()). You should always null-check the result before dereferencing it.

A common pattern is to walk the hierarchy upward from any entity to locate the root of an articulated character: 

// Walk up to the root entity from an arbitrary descendant
Entity* findRoot(Entity* e) {
    while (Entity* p = e->GetParent())
        e = p;
    return e;   // first entity with no parent
}

7. GetChildCount(), GetChild(index), and GetChildren()

// GetChildCount — number of DIRECT children only
int n = player_->GetChildCount();   // → 1 (only visual_; socket and sword are grandchildren)

// GetChild(index) — zero-based; returns nullptr when index >= GetChildCount()
Entity* first  = player_->GetChild(0);   // → visual_
Entity* bad    = player_->GetChild(9);   // → nullptr

// GetChildren — returns a vector of ALL direct children
std::vector<Entity*> kids = player_->GetChildren();
for (Entity* child : kids) {
    child->SetActive(false);   // hide every direct child
}

All three methods report only direct children — not grandchildren or deeper descendants. GetChild(0) on player_ returns visual_ but does not see weaponSocket_ or sword_. If you need to walk the entire subtree you must recurse: 

// Recursive subtree walk: collect all descendants
void collectAll(Entity* root, std::vector<Entity*>& out) {
    for (Entity* child : root->GetChildren()) {
        out.push_back(child);
        collectAll(child, out);   // recurse into grandchildren
    }
}

// Usage: every entity beneath player_ (visual_, socket_, sword_)
std::vector<Entity*> all;
collectAll(player_, all);   // all.size() == 3

8. TranslateLocal() versus TranslateWorld()

// TranslateLocal — moves along the ENTITY'S OWN rotated axes
weaponSocket_->TranslateLocal(Vector3(0.0f, 0.8f * dt, 0.0f));

// TranslateWorld — moves along the fixed WORLD axes (Y is always up in the scene)
weaponSocket_->TranslateWorld(Vector3(0.0f, 0.0f, 0.8f * dt));

Both methods add a positional delta each frame, but they interpret the delta differently. TranslateLocal() applies the delta in the entity's own local coordinate frame — "forward" means along the entity's current facing direction, even if that direction is rotated relative to the world. This is the natural choice for AI movement, weapon recoil effects, or any motion that should follow an entity's own orientation.

TranslateWorld() always interprets the delta in the fixed world coordinate frame. Vector3(0, 0, 1) always moves the entity one unit in the world's +Z direction, regardless of where the entity is facing. This is useful when you want motion that is absolute — for instance sliding a platform along a fixed track axis, or moving a head-bob entity strictly up and down in the world.

You can see the contrast clearly in the demo: press Q/E to raise and lower the socket along its own (possibly tilted) local Y axis with TranslateLocal(). Then press G/H to push it along world Z with TranslateWorld(). The trajectories diverge whenever the socket's local frame is rotated away from the world axes.

Legacy alias

The API also exposes entity->MoveRelative(delta), which is an alias for TranslateWorld(). It exists for backward compatibility. In new code prefer the explicit TranslateWorld() name.

9. The weapon socket pattern

The weapon socket is the single most reused hierarchy pattern in 3D action games. The idea is simple: instead of parenting a weapon model directly to a character bone, you first create an empty transform entity — the "socket" — at the desired attachment point. The weapon model then hangs beneath the socket. 

// Create the socket at the character's right-hand grip position
weaponSocket_ = visual_->CreateChild("WeaponSocket");
weaponSocket_->SetLocalPosition(0.45f, 0.55f, 0.25f);
weaponSocket_->SetLocalRotation(Quaternion(0.0f, 0.0f, -20.0f));

// Attach the sword under the socket
sword_ = weaponSocket_->CreateChild("Sword");
sword_->AddModel("Models/Sword.mdl");
sword_->SetLocalPosition(0.0f, 0.35f, 0.0f);   // blade offset within grip
sword_->SetScale(0.75f);

The advantages of this indirection become clear when you want to swap weapons. Rather than destroying and recreating entities with careful coordinate adjustments, you simply hide the old weapon and show the new one (or reparent a different model to the same socket): 

// Equip an axe instead of the sword — same socket, different weapon child
sword_->SetActive(false);           // hide sword (subtree suspended)
Entity* axe = weaponSocket_->CreateChild("Axe");
axe->AddModel("Models/Axe.mdl");
axe->SetLocalPosition(0.0f, 0.30f, 0.0f);

The socket's local position and rotation remain unchanged — the new weapon inherits the same grip placement automatically. You can also define multiple sockets in a single setup (one per hand, one on the back for a sheathed weapon) and toggle which socket is active depending on game state.

In the demo, pressing F toggles weaponSocket_->SetActive(false). This hides both the socket and the sword in one call, because activating or deactivating a parent entity always propagates to every entity in its subtree.

10. SetActive() propagation through the subtree

// Deactivate the entire weapon subtree (socket + sword) in one call
weaponSocket_->SetActive(false);

// Re-activate
weaponSocket_->SetActive(true);

// Check the state of any individual node
bool on = weaponSocket_->IsActive();

Deactivating a parent entity suspends rendering, physics, and scripting for the entire subtree. You do not need to iterate through children to hide them individually. SetActive(true) restores all descendants that were active before the parent was deactivated. This makes SetActive ideal for toggling groups of related objects — weapon rigs, cutscene props, enemy spawners — without destroying and recreating them.

Note that SetActive(false) is not the same as DestroyEntity(). Destroyed entities are permanently removed and their pointers become invalid. Deactivated entities remain in memory and can be restored. Use SetActive for temporary visibility changes (equipping/unequipping, opening/closing doors, object pooling) and DestroyEntity() only when an object should never return.

11. Persistent entities across scene transitions

When you call LoadScene() the engine destroys all non-persistent entities. If your player hierarchy should survive a level transition, mark only the root entity as persistent: 

// Both player_ and ALL its descendants (visual_, socket_, sword_) will survive
SetPersistentEntity(player_);   // call this BEFORE the first LoadScene()

The entire subtree of a persistent entity is automatically preserved. You do not need to mark each child individually.

The body / visual split in depth

The demo's player hierarchy follows the pattern used in the blupi_proto reference implementation and in many commercial character-controller games. Understanding why the split exists will help you apply it correctly.

A CharacterController attaches to the entity at its origin. The capsule's centre is at the entity's position; a 1.8 m tall capsule has its bottom at y - 0.9 and its top at y + 0.9. If you place the player entity at (0, 1, 0) the capsule bottom is exactly at y = 0.1 — just above ground zero.

If you attach the character model directly to the physics entity, the mesh origin (usually the character's feet) sits at the capsule's centre, making the character float 0.9 m in the air. By instead attaching the model to a child entity with a local Y offset of −0.9 (or whatever offset your art requires), the feet of the mesh align with the ground while the capsule's centre remains at the correct physics height.

A second advantage: the camera can follow the visual child rather than the physics body. The visual child sits at mid-body height, producing a natural third-person orbit target centred on the character's torso. If the camera followed the physics body directly it would orbit a point at the capsule's centre, which often feels too low. 

// The full body/visual split pattern:

Entity* player = CreateEntity("Player");
player->SetPosition(0.0f, 0.9f, 0.0f);      // capsule centre 0.9 m above ground
player->AddCharacterController(0.35f, 1.8f);   // radius 0.35, height 1.8
player->SetCollisionLayer(CollisionLayer::Actor);
player->SetCollisionMask(MakeCollisionMask({
    CollisionLayer::StaticGeometry,
    CollisionLayer::Trigger
}));

Entity* visual = player->CreateChild("PlayerVisual");
visual->AddModel("Models/Character.mdl");
visual->SetLocalPosition(0.0f, -0.9f, 0.0f);  // feet at ground level

Camera* cam = CreateCamera();
cam->Follow(visual);                            // orbit mid-body, not feet

Lua equivalent

All of the hierarchy API is available in Lua. The example below builds the same three-level player hierarchy and demonstrates GetParent(), GetChildren(), and TranslateLocal(). 

-- game.lua — Lua equivalent of the Entity Hierarchy tutorial (Part 22)

local player, visual, weaponSocket, sword, cam, label
local cc
local swordEquipped = true

function Start()
    game:SetWindowTitle("Entity Hierarchy — Lua")
    game:SetClearColor(Color(0.12, 0.15, 0.20, 1.0))

    -- Light
    local sun = game:CreateEntity("Sun")
    sun:AddDirectionalLight(Color(1.0, 0.93, 0.82), 1.2)
    sun:SetRotation(Quaternion(50, 30, 0))

    -- Level 1: Player (root)
    player = game:CreateEntity("Player")
    player:SetPosition(0, 1, 0)
    cc = player:AddCharacterController(0.35, 1.8)

    -- Level 2: Visual child — CreateChild() attaches a new entity to player
    visual = player:CreateChild("PlayerVisual")
    visual:AddModel("Models/Character.mdl")
    visual:SetLocalPosition(0, 0.8, 0)   -- LOCAL offset, not world!

    -- Level 3: Weapon socket (empty transform entity)
    weaponSocket = visual:CreateChild("WeaponSocket")
    weaponSocket:SetLocalPosition(0.45, 0.55, 0.25)
    weaponSocket:SetLocalRotation(Quaternion(0, 0, -20))

    -- Level 4: Sword (child of socket)
    sword = weaponSocket:CreateChild("Sword")
    sword:AddModel("Models/Sword.mdl")
    sword:SetLocalPosition(0, 0.35, 0)
    sword:SetScale(0.75)

    -- Camera orbiting the mid-body visual child
    cam = game:CreateCamera()
    cam:SetOrbitMode(visual, 6.0)
    cam:SetOrbitAngles(0, 15)

    -- Demonstrate hierarchy enumeration in Lua
    local count = player:GetChildCount()   -- → 1
    local first = player:GetChild(0)       -- → visual
    local kids  = player:GetChildren()     -- table of direct children
    local par   = visual:GetParent()       -- → player
    -- par:GetName() == "Player"

    -- Iterate all direct children of visual and print names
    for _, child in ipairs(visual:GetChildren()) do
        print("Visual child: " .. child:GetName())
    end

    label = game:CreateLabel("")
    label:SetPosition(16, 16)
    label:SetFontSize(15)
    label:SetColor(Color.WHITE)
end

function Update(dt)
    if Input:IsKeyPressed("Escape") then game:Quit() end

    -- Movement
    local move = Vector3(0, 0, 0)
    if Input:IsKeyDown("W") then move.z = move.z + 1 end
    if Input:IsKeyDown("S") then move.z = move.z - 1 end
    if Input:IsKeyDown("A") then move.x = move.x - 1 end
    if Input:IsKeyDown("D") then move.x = move.x + 1 end
    cc:Move(move * 5, dt)

    if Input:IsKeyPressed("Space") then cc:TryJump(10) end

    -- Toggle sword
    if Input:IsKeyPressed("F") then
        swordEquipped = not swordEquipped
        weaponSocket:SetActive(swordEquipped)
    end

    -- Raise / lower socket using TranslateLocal (entity's own axes)
    if Input:IsKeyDown("Q") then
        weaponSocket:TranslateLocal(Vector3(0, 0.8 * dt, 0))
    end
    if Input:IsKeyDown("E") then
        weaponSocket:TranslateLocal(Vector3(0, -0.8 * dt, 0))
    end

    -- Push socket along world Z using TranslateWorld (fixed world axes)
    if Input:IsKeyDown("G") then
        weaponSocket:TranslateWorld(Vector3(0, 0, 0.8 * dt))
    end
    if Input:IsKeyDown("H") then
        weaponSocket:TranslateWorld(Vector3(0, 0, -0.8 * dt))
    end

    -- HUD update
    local wpos  = weaponSocket:GetPosition()
    local lpos  = weaponSocket:GetLocalPosition()
    label:SetText(
        "Children of player: " .. player:GetChildCount() .. "\n" ..
        "Socket local:  ("  .. string.format("%.2f %.2f %.2f", lpos.x, lpos.y, lpos.z) .. ")\n" ..
        "Socket world:  ("  .. string.format("%.2f %.2f %.2f", wpos.x, wpos.y, wpos.z) .. ")\n" ..
        "Sword: " .. (swordEquipped and "equipped" or "unequipped") .. "\n" ..
        "WASD=move  Space=jump  F=toggle  Q/E=local  G/H=world"
    )
end

In Lua, GetChildren() returns a Lua table (array) rather than a std::vector. Iterate it with ipairs(). All other method names and argument types are identical to the C++ API.

API quick reference — this part

MethodReturnsDescription
entity->CreateChild(string name) Entity* Creates a new entity immediately beneath entity in the scene graph; engine owns the lifetime
entity->GetParent() Entity* Direct parent entity, or nullptr for root entities; always null-check the result
entity->GetChildCount() int Number of direct children only; grandchildren are not counted
entity->GetChild(int index) Entity* Zero-based direct child lookup; returns nullptr when index >= GetChildCount()
entity->GetChildren() vector<Entity*> All direct children as a vector; empty vector if the entity is a leaf node
entity->SetLocalPosition(x, y, z) — Sets position in the parent's coordinate frame; also accepts Vector3
entity->GetLocalPosition() Vector3 Parent-relative position, unaffected by where the parent currently sits in world space
entity->SetLocalRotation(Quaternion) — Rotation relative to the parent's facing direction; use Quaternion(pitch, yaw, roll)
entity->GetLocalRotation() Quaternion Parent-relative rotation; the value set by SetLocalRotation()
entity->TranslateLocal(Vector3 delta) — Moves the entity by delta in its own rotated local axes
entity->TranslateWorld(Vector3 delta) — Moves the entity by delta in the fixed world axes; ignores entity rotation
entity->SetActive(bool) — Shows/hides the entity and all descendants; false suspends rendering and physics
entity->IsActive() bool Current activation state of this individual entity node
SetPersistentEntity(Entity*) — Marks the entity (and its entire subtree) to survive LoadScene() calls

Common mistakes

Common mistakes — Part 22

  • Using SetPosition() on a child entity to set its offset. Calling child->SetPosition(0, 0.8f, 0) writes a world-space coordinate. The parent's transform will override it the next frame, causing the child to snap back or drift erratically. Use SetLocalPosition() to declare a permanent parent-relative offset that persists across all parent movements.
  • Confusing TranslateLocal() and TranslateWorld(). When an entity's local frame is aligned with the world (zero rotation) the two methods produce the same result. They diverge only once the entity is rotated. A common symptom: an AI character "strafes" diagonally instead of moving straight ahead because the code used TranslateWorld(Vector3::FORWARD) without accounting for the character's facing rotation. Use TranslateLocal() when you want motion to follow the entity's own facing direction.
  • Calling GetChildren() expecting grandchildren. All enumeration methods (GetChildCount(), GetChild(), GetChildren()) report only the direct children of the queried entity. They never descend into nested levels. If you need all descendants, write a recursive walk using GetChildren() on each entity in turn.
  • Dereferencing a nullptr returned by GetParent() or GetChild(). GetParent() returns nullptr for root entities. GetChild(index) returns nullptr when the index is out of range. Both must be null-checked before use. The pattern if (Entity* p = child->GetParent()) { p->DoSomething(); } is the safest idiom.
  • Positioning the weapon in world space and wondering why it floats in place. The weapon socket and the weapon beneath it must both use SetLocalPosition(). If you accidentally call sword->SetPosition(worldCoord), the sword will snap to that world coordinate — and then the parent's transform will try to move it somewhere else, resulting in a visually glitched, oscillating position.
  • Parenting physics bodies to each other as a substitute for constraints. Child entities do not contribute their own physics response to the parent. If you need two physics bodies to move together rigidly, use a Fixed physics constraint (Part 23), not a parent/child relationship. Parent/child hierarchy is purely a transform relationship — it does not simulate physical forces between the linked bodies.
  • Calling SetPersistentEntity() on a child instead of the root. Only the subtree root needs to be marked persistent. Marking an individual child (such as the visual or the weapon) persistent while leaving the parent non-persistent will result in the parent being destroyed at scene transition and the child becoming an orphaned, dangling subtree. Always call SetPersistentEntity() on the topmost entity you want to preserve.
  • Using SetActive(false) after calling DestroyEntity(). After DestroyEntity(ptr) the pointer is invalid. Any subsequent access — including ptr->SetActive() — is undefined behaviour. Set the pointer to nullptr immediately after calling DestroyEntity() and guard all subsequent uses with a null-check.

Try it yourself

Once the example is running, try these exercises to deepen your understanding:

  1. Add a second weapon socket on the left hand by calling visual_->CreateChild("WeaponSocketLeft") with a mirrored X position (negative X). Attach a shield model beneath it. Verify that moving the player carries both weapons along.
  2. Change the camera to follow player_ (the physics body) instead of visual_ (the mid-body child). Observe how the orbit pivot changes — it drops to the capsule centre instead of the torso height.
  3. In Update(), iterate all of visual_->GetChildren() and print their names and world positions to the label. Extend the list by adding a "BackScabbard" child to the visual and watch it appear in the loop automatically.
  4. Use GetParent() to navigate from the sword_ entity back up to the scene root and print each ancestor's name to trace the full chain: Sword → WeaponSocket → PlayerVisual → Player.
  5. Replace the F key binding with a timer that toggles the sword on and off every two seconds by calling weaponSocket_->SetActive(). This simulates a "flicker" damage-invincibility effect applied to the entire weapon sub-hierarchy at once.
  6. Add a point light as a child of weaponSocket_ so the sword casts a coloured glow on nearby surfaces. Because the light is a child of the socket, it follows the sword automatically without any per-frame positioning code.

What you learned

Entity hierarchy is the structural backbone of every complex scene in simple-3d. Here is a summary of what this part covered:

  • The scene graph is an implicit tree. CreateEntity() creates root nodes; entity->CreateChild(name) creates child nodes one level deeper. The engine owns all nodes regardless of how they were created.
  • Transform inheritance is automatic. Moving or rotating a parent entity repositions all descendants in world space. No per-frame sync code is ever needed.
  • Local space vs. world space is the central distinction. Use SetLocalPosition() and SetLocalRotation() to express a child's fixed offset relative to its parent. Use GetPosition() to read back the computed absolute world coordinate.
  • TranslateLocal() moves an entity along its own rotated axes — correct for orientation-aware motion like character movement or weapon recoil. TranslateWorld() moves along fixed world axes — correct for motion that must stay aligned with the scene regardless of entity rotation.
  • Hierarchy enumeration via GetChildCount(), GetChild(index), GetChildren(), and GetParent() lets you inspect and traverse the tree at runtime. All methods operate on direct children only; recurse for full subtree walks.
  • The body / visual child split is the idiomatic pattern for characters with a physics capsule: place the physics root at the capsule centre, attach the renderable mesh as a child with a local Y offset so the feet align with the ground, and orbit the camera around the visual child for a natural mid-body pivot.
  • The weapon socket pattern inserts an empty transform entity at the desired grip point. Weapon models hang beneath this socket. Swapping weapons is then a matter of toggling child entities rather than repositioning and re-orienting models from scratch.
  • SetActive(false) on any node propagates to its entire subtree, suspending rendering and physics for the whole branch in one call — ideal for pooled objects, hidden loot, and equipment toggles.
  • SetPersistentEntity() on the root of a subtree preserves the entire hierarchy across LoadScene() transitions.

The next tutorial, Part 23, introduces physics constraints — hinges, sliders, and ball-socket joints that connect rigid bodies in ways the scene-graph parent/child relationship alone cannot, opening the door to swinging doors, motorised platforms, and chain-link simulations.
