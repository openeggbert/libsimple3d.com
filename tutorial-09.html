Part 09 of 30Animation State Machine

New in this part

entity->AddAnimStateMachine() — attach a declarative animation controller to an entity.
asm->AddState(name, clipPath, loop) — register a named animation state.
asm->AddTransition(from, to, conditionKey) — fire on a bool or trigger condition.
asm->AddTransitionBool(from, to, key, expectValue) — fire when a bool equals a specific value.
asm->AddTransitionFloat(from, to, key, threshold, greaterThan) — fire on a float comparison.
asm->AddAutoTransition(from, to) — automatically advance when a non-looped animation finishes.
asm->SetBool(key, value) / SetFloat(key, value) / SetTrigger(key) — drive condition variables.
asm->GetCurrentState() / asm->IsInState(name) — runtime state queries.

What you will learn

  • Why a declarative Animation State Machine (ASM) is cleaner than manual if/else animation logic.
  • How to register animation states with AddState() and make the first-added state the default.
  • The three transition types: bool/trigger (AddTransition), explicit bool value (AddTransitionBool), and float comparison (AddTransitionFloat).
  • How AddAutoTransition() lets a one-shot animation return to a looping state without any game-logic code.
  • How to drive the machine with SetBool, SetFloat, and SetTrigger — and the key difference between a trigger and a bool.
  • How to query the active state name at runtime for UI feedback, sound cues, or gameplay logic.
  • A complete locomotion + special-action example covering idle, walk, run, jump, and a one-shot custom gesture.
  • The equivalent Lua scripting approach.

The problem with manual animation logic

In Part 08 you used AnimationController directly and called anim->Play() and anim->Stop() by hand. For a single walk/idle cycle that is fine, but real characters have many more states. Once you add run, jump, attack, hurt, and die, the logic inside Update() grows into a nest of conditionals that is hard to read and even harder to extend.

The AnimStateMachine solves this by separating declaration from execution. You describe the states and the rules for switching between them once in Start(). Every frame you simply update the condition variables and call sm->Update(dt). The machine evaluates transitions, picks the new state, and blends the animation — no manual play/stop calls needed.

Concepts at a glance

TermMeaning
State A named wrapper around one animation clip. e.g. "idle" plays Anims/Idle.ani.
Transition A directed edge from one state to another, guarded by a condition.
Condition variable A named bool, float, or trigger stored inside the machine. Transitions test these.
Trigger Like a bool, but automatically reset to false after the first transition that consumes it fires.
Auto-transition A conditionless edge that fires automatically when a non-looped clip reaches its end.
Default state The state the machine starts in — always the first state passed to AddState().

Complete working example

The example below builds a character with five animation states: idle, walk, run, jump, and a one-shot tongue gesture. It also displays the current state name on screen so you can observe transitions in real time. 

// AnimStateMachineDemo.cpp
// Part 09 — Animation State Machine
// Build: link against simple-3d, place animation files under Data/Anims/

#include <Simple3D/Simple3D.h>
using namespace Simple3D;

class AnimDemo : public Game {
public:
    void Start() override
    {
        // ── Ground plane ─────────────────────────────────────
        Entity* ground = CreateEntity("Ground");
        ground->AddModel("Models/Plane.mdl");
        ground->AddRigidBody(0.0f);                        // static
        ground->AddBoxCollider(Vector3(20, 0.2f, 20));
        ground->SetCollisionLayer(CollisionLayer::StaticGeometry);

        // ── Player entity ─────────────────────────────────────
        player_ = CreateEntity("Player");
        player_->SetPosition(0, 1, 0);
        player_->AddModel("Models/Hero.mdl");

        // Character controller (gravity + slope handling)
        cc_ = player_->AddCharacterController(0.35f, 1.8f);
        cc_->SetGravity(25.0f);
        player_->SetCollisionLayer(CollisionLayer::Actor);
        player_->SetCollisionMask(MakeCollisionMask({
            CollisionLayer::StaticGeometry
        }));

        // ── Animation State Machine ───────────────────────────
        // AddAnimStateMachine() also creates an AnimationController
        // on the entity automatically — no separate call needed.
        sm_ = player_->AddAnimStateMachine();

        // States — the FIRST state added is always the default.
        sm_->AddState("idle",   "Anims/Idle.ani",   true);   // looped
        sm_->AddState("walk",   "Anims/Walk.ani",   true);   // looped
        sm_->AddState("run",    "Anims/Run.ani",    true);   // looped
        sm_->AddState("jump",   "Anims/Jump.ani",   false);  // one-shot
        sm_->AddState("tongue", "Anims/Tongue.ani", false);  // one-shot custom gesture

        // ── Transitions ───────────────────────────────────────

        // idle -> walk when bool "moving" becomes true
        sm_->AddTransition("idle", "walk", "moving");

        // walk -> idle when bool "moving" is false
        sm_->AddTransitionBool("walk", "idle", "moving", false);

        // walk -> run when float "speed" exceeds 4.0 m/s
        sm_->AddTransitionFloat("walk", "run",  "speed", 4.0f, true);

        // run -> walk when float "speed" falls below 4.0 m/s
        sm_->AddTransitionFloat("run",  "walk", "speed", 4.0f, false);

        // idle or walk -> jump on trigger "jump"
        sm_->AddTransition("idle", "jump", "jump");
        sm_->AddTransition("walk", "jump", "jump");
        sm_->AddTransition("run",  "jump", "jump");

        // jump automatically returns to idle when the clip finishes
        sm_->AddAutoTransition("jump", "idle");

        // idle -> tongue on trigger "doTongue"
        sm_->AddTransition("idle", "tongue", "doTongue");

        // tongue automatically returns to idle when the clip finishes
        sm_->AddAutoTransition("tongue", "idle");

        // ── Input bindings ────────────────────────────────────
        BindAxis2D("Move")
            .Keys(Key::Left, Key::Right, Key::Down, Key::Up)
            .GamepadStick(GamepadAxis::LeftX, GamepadAxis::LeftY)
            .InvertY(true);

        BindAction("Jump")
            .Key(Key::Space)
            .GamepadButton(GamepadButton::A);

        BindAction("Tongue")
            .Key(Key::T)
            .GamepadButton(GamepadButton::Y);

        BindAction("Sprint")
            .Key(Key::LShift)
            .GamepadButton(GamepadButton::LB);

        BindAction("Quit")
            .Key(Key::Escape);

        // ── Camera ────────────────────────────────────────────
        Camera* cam = CreateCamera();
        cam->Follow(player_, 6.0f, 2.0f);
        cam->LookAt(player_);
        cam->SetSmoothFollow(true, 5.0f);

        // ── Light ─────────────────────────────────────────────
        Entity* sun = CreateEntity("Sun");
        sun->SetRotation(Quaternion(45, 30, 0));
        sun->AddDirectionalLight(Color::WHITE, 1.2f);

        // ── HUD label ─────────────────────────────────────────
        stateLabel_ = CreateLabel("State: idle");
        stateLabel_->SetPosition(20, 20);
        stateLabel_->SetFontSize(22);
        stateLabel_->SetColor(Color::WHITE);

        hintLabel_ = CreateLabel("Arrows=move  Shift=sprint  Space=jump  T=tongue  Esc=quit");
        hintLabel_->SetPosition(20, 50);
        hintLabel_->SetFontSize(16);
        hintLabel_->SetColor(Color(0.8f, 0.8f, 0.8f, 1));
    }

    void Update(float dt) override
    {
        if (IsActionPressed("Quit")) { Quit(); return; }

        // ── Movement ──────────────────────────────────────────
        Vector2 input = GetAxis2D("Move");
        bool    sprint = IsActionDown("Sprint");
        float   topSpeed = sprint ? 7.0f : 3.0f;

        cc_->Move(Vector3(input.x_, 0, input.y_) * topSpeed, dt);

        // Actual horizontal speed from the physics body
        Vector3 vel   = player_->GetLinearVelocity();
        float   speed = Vector2(vel.x_, vel.z_).Length();

        // ── Condition variables ───────────────────────────────
        sm_->SetBool ("moving", speed > 0.15f);
        sm_->SetFloat("speed",  speed);

        // Triggers are consumed once (one-shot events)
        if (IsActionPressed("Jump"))   sm_->SetTrigger("jump");
        if (IsActionPressed("Tongue")) sm_->SetTrigger("doTongue");

        // TryJump checks IsOnGround() internally
        if (IsActionPressed("Jump")) cc_->TryJump(10.0f);

        // ── Tick the machine ──────────────────────────────────
        // This call evaluates all transitions and advances
        // the current animation — must be called every frame.
        sm_->Update(dt);

        // ── HUD refresh ───────────────────────────────────────
        stateLabel_->SetText("State: " + sm_->GetCurrentState());
    }

private:
    Entity*           player_      = nullptr;
    CharacterController* cc_       = nullptr;
    AnimStateMachine* sm_          = nullptr;
    Label*            stateLabel_  = nullptr;
    Label*            hintLabel_   = nullptr;
};

int main()
{
    return Simple3D::CreateGame<AnimDemo>()->Run();
}

Step-by-step explanation

entity->AddAnimStateMachine()

Calling AddAnimStateMachine() on an entity does two things at once: it creates an AnimationController component (exactly as in Part 08), and it wraps it inside a new AnimStateMachine that you control declaratively. You should not call AddAnimationController() separately — the state machine sets one up for you. The returned pointer is valid for the lifetime of the entity. After creation you can retrieve it at any time with entity->GetAnimStateMachine(), which returns nullptr if AddAnimStateMachine() was never called.

sm->AddState(name, clipPath, loop)

Each call registers one named animation state. The name is a free-form string — it becomes the key used by transitions and by GetCurrentState(). The clipPath is the path to an .ani animation file inside your Data/ folder, exactly as you would pass to AnimationController::Play(). Setting loop to true makes the animation repeat indefinitely; false makes it a one-shot clip. The first state you add becomes the default — the machine starts in that state as soon as the first Update(dt) runs. If you want a different starting state, call sm->SetDefaultState(name) after all states are registered.

sm->AddTransition(from, to, conditionKey)

This is the most common transition type. The machine checks whether conditionKey names a trigger that was set or a bool that is currently true. If either is the case and the machine is in state from, it blends into state to. The default blend time is 0.2 seconds; you can pass a fourth argument to override it: sm->AddTransition("idle", "walk", "moving", 0.1f).

sm->AddTransitionBool(from, to, key, expectValue)

Works like AddTransition but fires only when the bool key equals expectValue. The typical use is expectValue = false to transition away from a state when a condition stops being true. In the example, AddTransitionBool("walk", "idle", "moving", false) returns the character to idle the moment the player releases the movement keys.

sm->AddTransitionFloat(from, to, key, threshold, greaterThan)

Fires when the float condition variable named key crosses threshold. When greaterThan is true the condition is key > threshold; when false it is key < threshold. In the example, AddTransitionFloat("walk", "run", "speed", 4.0f, true) upgrades the character to a run animation as soon as horizontal speed exceeds 4 m/s, and AddTransitionFloat("run", "walk", "speed", 4.0f, false) drops them back to walking below that threshold.

sm->AddAutoTransition(from, to)

An auto-transition has no condition variable. It fires automatically when the animation in state from reaches its end — but only when that state's animation was registered with loop = false. This is the canonical way to return from a one-shot action to a looping base state. In the example, both "jump" and "tongue" are non-looped states, and each has an auto-transition back to "idle". The optional third parameter controls blend time (default 0.1 s).

sm->SetBool(key, value)

Sets a persistent named boolean condition inside the machine. The value remains until you explicitly change it. SetBool is the right choice for continuous states like "is the player moving" or "is the player crouching". In the example it is called every frame with the result of a speed threshold check. Read the value back with sm->GetBool(key).

sm->SetFloat(key, value)

Sets a named floating-point condition inside the machine. Any AddTransitionFloat rules that reference this key will re-evaluate on the next sm->Update(dt). In the example, the character's actual horizontal speed is written here each frame so the walk/run threshold transition can pick it up. Read the current value with sm->GetFloat(key).

sm->SetTrigger(key)

Marks a named trigger as "set". The trigger behaves like a boolean true but is automatically reset to false after the first transition that tests it actually fires. This makes it perfect for discrete events such as jumping or attacking: even if the player holds the jump key for multiple frames, the jump animation will start only once. You can call SetTrigger before the machine is in the right state — the trigger will wait until a matching transition is applicable.

sm->Update(dt)

This must be called once per frame, after you have finished writing all condition variables. It evaluates every registered transition in the order they were added, picks the first one whose guard is satisfied, and if the current state changes it starts the crossfade blend. It also advances the underlying AnimationController playback. Forgetting this call means no transition will ever fire and the animation will freeze on the default state.

sm->GetCurrentState() and sm->IsInState(name)

GetCurrentState() returns a std::string with the name of the currently active state. Use this to drive UI labels, spawn sound effects on state entry, or make gameplay decisions based on what the character is doing. IsInState(name) is a convenience shortcut that returns true when the active state matches name.

Transition evaluation order

On each Update(dt) call, the machine checks only transitions that originate from the current state. Among those, the first one added whose guard is satisfied wins. Transitions from other states are ignored. This means the order you call AddTransition* matters when multiple guards could be simultaneously true — put higher-priority transitions first.

A transition to a state that is already the active state is a no-op; the machine will not restart the animation needlessly.

One-shot pattern in detail

Non-looped animations that must play to completion and then return to a base state always follow the same three-step pattern:

  1. Register the state with loop = false.
  2. Add an inbound transition (usually a trigger) from the base state.
  3. Add an AddAutoTransition back to the base state.
// One-shot pattern — generalized
sm_->AddState("myAction", "Anims/MyAction.ani", false); // step 1

sm_->AddTransition("idle", "myAction", "doMyAction");   // step 2
sm_->AddAutoTransition("myAction", "idle");             // step 3

// In Update(dt):
if (IsActionPressed("MyKey")) sm_->SetTrigger("doMyAction");

Because AddAutoTransition fires only for non-looped clips, you can mix looped and non-looped states freely in the same machine without them interfering with each other.

Querying state for gameplay logic

The active state name is often useful outside the animation system. For example, you might want to prevent the player from starting a new action while they are mid-jump, or play a sound effect the first time the machine enters the "run" state. 

// Prevent tongue during jump
if (IsActionPressed("Tongue") && !sm_->IsInState("jump"))
    sm_->SetTrigger("doTongue");

// Conditional sound on entering run
std::string curState = sm_->GetCurrentState();
if (curState != prevState_ && curState == "run")
    PlaySound("Sounds/BreathIn.wav");
prevState_ = curState;

Full transition type reference

Method Fires when… Typical use
AddTransition(from, to, key) key is a live trigger or a bool that is true Start walking, start jumping, start any action triggered by a key press
AddTransitionBool(from, to, key, expect) bool key equals expect Return to idle when movement stops (expect = false)
AddTransitionFloat(from, to, key, threshold, gt) float key > threshold (or < when gt=false) Walk-to-run speed threshold, health-based state changes
AddAutoTransition(from, to) Non-looped animation in from reaches end Jump/attack/gesture return to idle automatically

Condition variable reference

Method Type Persists? Notes
SetBool(key, value) / GetBool(key) bool Yes — until you change it Use for continuous states (moving, crouching, armed)
SetFloat(key, value) / GetFloat(key) float Yes — until you change it Use for continuous numeric quantities (speed, health, blend weight)
SetTrigger(key) one-shot bool No — consumed on first firing transition Use for discrete events (button pressed, hit taken, collectible gathered)

Lua equivalent

The Lua scripting API mirrors C++ one-to-one. Attach the script via entity:AddScript("Scripts/player_anim.lua") and implement Start and Update as global functions. 

-- Scripts/player_anim.lua
-- Demonstrates the Animation State Machine from Lua

local sm     -- AnimStateMachine
local cc     -- CharacterController
local player -- Entity

function Start()
    player = GetEntity()

    -- Character controller
    cc = player:AddCharacterController(0.35, 1.8)
    cc:SetGravity(25.0)

    -- Animation State Machine
    sm = player:AddAnimStateMachine()

    -- States
    sm:AddState("idle",   "Anims/Idle.ani",   true)
    sm:AddState("walk",   "Anims/Walk.ani",   true)
    sm:AddState("run",    "Anims/Run.ani",    true)
    sm:AddState("jump",   "Anims/Jump.ani",   false)
    sm:AddState("tongue", "Anims/Tongue.ani", false)

    -- Transitions
    sm:AddTransition    ("idle", "walk", "moving")
    sm:AddTransitionBool ("walk", "idle", "moving", false)
    sm:AddTransitionFloat("walk", "run",  "speed", 4.0, true)
    sm:AddTransitionFloat("run",  "walk", "speed", 4.0, false)
    sm:AddTransition    ("idle", "jump", "jump")
    sm:AddTransition    ("walk", "jump", "jump")
    sm:AddTransition    ("run",  "jump", "jump")
    sm:AddAutoTransition("jump",   "idle")
    sm:AddTransition    ("idle", "tongue", "doTongue")
    sm:AddAutoTransition("tongue", "idle")

    -- Input bindings
    BindAxis2D("Move")
        :Keys(Key.Left, Key.Right, Key.Down, Key.Up)
        :GamepadStick(GamepadAxis.LeftX, GamepadAxis.LeftY)
        :InvertY(true)

    BindAction("Jump")  :Key(Key.Space)   :GamepadButton(GamepadButton.A)
    BindAction("Tongue"):Key(Key.T)       :GamepadButton(GamepadButton.Y)
    BindAction("Sprint"):Key(Key.LShift)  :GamepadButton(GamepadButton.LB)
end

function Update(dt)
    local input   = GetAxis2D("Move")
    local topSpeed = IsActionDown("Sprint") and 7.0 or 3.0

    cc:Move(Vector3(input.x, 0, input.y) * topSpeed, dt)

    local vel   = player:GetLinearVelocity()
    local speed = Vector2(vel.x, vel.z):Length()

    sm:SetBool ("moving", speed > 0.15)
    sm:SetFloat("speed",  speed)

    if IsActionPressed("Jump")   then
        sm:SetTrigger("jump")
        cc:TryJump(10.0)
    end

    if IsActionPressed("Tongue") then
        sm:SetTrigger("doTongue")
    end

    sm:Update(dt)   -- must be last, after all conditions are set

    -- Print state name to a label or the console for debugging
    -- Log("Current state: " .. sm:GetCurrentState())
end

Lua tip: GetEntity()

Inside a Lua script attached with entity:AddScript(), the global function GetEntity() returns the Entity the script is attached to. This is the standard way to get a self-reference in simple-3d Lua scripts.

Common mistakes

Common mistakes to avoid

  • Forgetting sm->Update(dt). Without this call no transition will ever evaluate and the animation will stay frozen in the default state. Always put it at the end of your Update() method, after all condition variables have been set.
  • Using a trigger when you need a bool. Triggers are consumed after the first matching transition fires. If you use a trigger for a continuous condition like "is moving", it will flip the state once and then stop working. Use SetBool for anything that should remain true across multiple frames.
  • Adding an AddAutoTransition on a looped state. Auto-transitions only fire when a non-looped animation reaches its end. On a looped clip the animation never "ends", so the auto-transition will never fire. Always pair AddAutoTransition with loop = false.
  • Setting condition variables after sm->Update(dt). Values written after the Update call will not be evaluated until the next frame. Write all SetBool, SetFloat, and SetTrigger calls before the final Update.
  • Calling AddAnimationController() separately. AddAnimStateMachine() creates the AnimationController internally. If you call AddAnimationController() beforehand, you will end up with two controllers competing, leading to incorrect blending.
  • Transition from a state that does not exist. State names are arbitrary strings — a typo in AddTransition("idel", ...) will silently register a transition from a non-existent state. It will never fire. Double-check spellings, especially when copying from another state.
  • Expecting transitions to fire immediately in Start(). Transitions are evaluated inside Update(dt). Any condition you set in Start() will take effect on the first frame's Update call, not before.

Retrieving the state machine after creation

If you create the state machine in one place (e.g. a factory function or a Lua script) but need to drive it from another (e.g. a different C++ component), use entity->GetAnimStateMachine(). It returns nullptr if AddAnimStateMachine() was never called, so always null-check the result in code paths where the entity might not have one. 

AnimStateMachine* sm = someEntity->GetAnimStateMachine();
if (sm) {
    sm->SetTrigger("hit");
}

What you learned

  • entity->AddAnimStateMachine() creates a declarative animation controller that replaces manual play/stop calls with a rules-based graph.
  • AddState(name, clipPath, loop) registers a named state. The first state added is the default; use SetDefaultState(name) to override.
  • Three transition methods cover every common case: AddTransition (bool or trigger), AddTransitionBool (explicit bool value), and AddTransitionFloat (threshold comparison).
  • AddAutoTransition(from, to) returns from a non-looped one-shot animation automatically without any game-code condition check.
  • SetBool and SetFloat set persistent condition variables; SetTrigger sets a one-shot event that is consumed after it fires.
  • sm->Update(dt) must be called every frame, after all condition variables are written, to evaluate transitions.
  • GetCurrentState() returns the active state name; IsInState(name) is a convenience predicate — both are useful for sound cues, UI display, and gameplay gating.
  • The Lua API is identical: call the same methods on the same objects using the sm:MethodName() colon syntax.
